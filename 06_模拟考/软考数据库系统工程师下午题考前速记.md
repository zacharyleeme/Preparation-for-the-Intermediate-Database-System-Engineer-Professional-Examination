# 软考数据库系统工程师：下午题错题类型考前速记

> 资料来源：`10_Learning/软考中级_数据库系统工程师/06_模拟考` 中 2020~2023 年下午题阅卷、错题复盘、错题重写与判分文件。
>
> 用途：基于**已经做错过的下午题**做考前快速纠偏，优先解决“会再错”的题型。
>
> 记忆原则：**错题优先、模板优先、术语优先、步骤优先。**

---

## 1. 考前速记定位

这份资料不是教材提纲，而是从下午题错题里抽出来的**高频失分点 + 标准答题模板**。

适合在考前：
- 快速查漏补缺
- 回忆答题模板
- 纠正高频误判
- 临考前 1 小时扫一遍

## 3. 高频错题类型详解

### 3.1 错题类型：数据库设计题（E-R 图、实体联系、主外键）

#### 1）常见考法

- 根据业务描述补全 E-R 图
- 由实体联系图写出关系模式
- 写主键、外键、完整性约束
- 说明 1:1、1:n、m:n 联系如何落到关系模式
- 题目常混入“新增实体”“新增联系”“新增关系模式”

#### 2）我的常见错误

- 只写实体属性，没写联系和基数
- 只写关系模式字段，没写主键和外键
- 把 1:n、m:n 关系判断反
- 看到“客户手机号码”“公司名称”就直接当主键，没核验唯一性
- 关系模式只写“属性”，没说明“谁引用谁”

#### 3）正确解题步骤

1. 先找**实体**：谁是独立对象？
2. 再找**联系**：谁和谁发生什么关系？
3. 判断**基数**：1:1 / 1:n / m:n。
4. 再写**关系模式**：实体变表，联系变表或外键。
5. 最后补**主键/外键/约束**。

#### 4）考前速记口诀 / 判断规则

- **先实体，后联系，再基数，最后落表。**
- **能唯一标识一条记录的，才配叫主键。**
- **表里谁带着别人的主键，谁就是外键端。**

#### 5）标准答题模板

- `实体A(主键A, 属性...)`
- `实体B(主键B, 属性...)`
- `联系C(主键A, 主键B, 联系属性...)`
- `外键：某属性 REFERENCES 某表(主键列)`
- `1:n：在 n 端放外键`
- `m:n：单独建联系表，主键通常是双方主键组合`

#### 6）典型易错点

- 忘记包裹“关系模式 + 主键 + 外键”三件套
- m:n 联系直接硬塞到一个实体表里
- 看到“每个 X 只能属于一个 Y”，却把基数写反
- 联系表主键少了区分属性，无法唯一标识记录

---

### 3.2 错题类型：E-R 图与关系模式转换

#### 1）常见考法

- 把 ER 图转换成若干关系模式
- 为新增业务补充新的实体和联系
- 给出完整性约束关系
- 判断关系模式中哪个字段该做外键

#### 2）我的常见错误

- 只补属性，不补联系
- 关系模式中漏掉联系属性
- 1:n 联系写成 m:n
- 看到关系模式里有“客户电话”“公司名称”，却没判断它是外键还是普通属性
- 新增实体时没有同步补主键和外键

#### 3）正确解题步骤

1. 画出/列出所有实体。
2. 找出实体间联系及基数。
3. 按 1:n、m:n 规则转换。
4. 联系表中加入参与实体主键。
5. 明确哪一个属性引用哪一张表。

#### 4）考前速记口诀 / 判断规则

- **1:n 外键放 n 端。**
- **m:n 先建联系表，主键用双方主键组合。**
- **联系属性要跟着联系走，不要丢。**

#### 5）标准答题模板

- `A(主键A, ... )`
- `B(主键B, ... )`
- `AB联系表(主键A, 主键B, 联系属性...)`
- `主键：(主键A, 主键B)`
- `外键：主键A REFERENCES A(主键A)；主键B REFERENCES B(主键B)`

#### 6）典型易错点

- 新增业务时漏掉“新增关系模式”
- 外键引用错表、错列
- 主键只写单字段，实际上需要联合主键
- 联系基数没看清，就直接下结论

---

### 3.3 错题类型：关系模式规范化

#### 1）常见考法

- 判断是否满足 2NF / 3NF / BCNF
- 写出部分依赖、传递依赖
- 分解成若干关系模式
- 标出每个表的主键、外键

#### 2）我的常见错误

- 候选键没找对，后面全错
- 把“部分依赖”说成“传递依赖”
- 分解方案写对了，但原因写错了
- 只会背“3NF 比 2NF 高”，不会判断违反原因
- 分解后没补主键和外键

#### 3）正确解题步骤

1. 先找**候选键**。
2. 再区分**主属性 / 非主属性**。
3. 看是否存在**部分依赖**。
4. 再看是否存在**传递依赖**。
5. 按题目要求分解并写主键外键。

#### 4）考前速记口诀 / 判断规则

- **先找键，再谈依赖。**
- **组合键的一部分决定非主属性 = 部分依赖。**
- **非主属性决定非主属性 = 传递依赖。**
- **BCNF：每个函数依赖左边都得是码。**

#### 5）标准答题模板

- `候选键：...`
- `存在部分依赖：X → Y，X 是组合键的一部分`
- `存在传递依赖：X → Y，Y → Z`
- `满足最高范式：...`
- `分解为：A(...), B(...), C(...)`
- `主键：...；外键：...`

#### 6）典型易错点

- 把“消除部分依赖”说成“消除传递依赖”
- 组合主键漏写一半
- 把非主属性当主属性
- 认为“只要分解了就是 3NF”

---

### 3.4 错题类型：函数依赖与候选键判断

#### 1）常见考法

- 给出关系模式，写出左侧只有一个属性的函数依赖
- 求候选键
- 判断主属性、非主属性
- 判断是否满足 2NF/3NF/BCNF

#### 2）我的常见错误

- 候选键没找全
- 主属性/非主属性混淆
- 把“能推出别的属性”误当主键
- 把单向依赖和双向依赖混淆

#### 3）正确解题步骤

1. 先写出所有已知函数依赖。
2. 求属性闭包，找候选键。
3. 标主属性、非主属性。
4. 判断是否存在部分依赖/传递依赖。
5. 再下范式结论。

#### 4）考前速记口诀 / 判断规则

- **闭包先算，候选键先定。**
- **主属性属于任一候选键。**
- **非主属性才谈“对码的部分/传递依赖”。**

#### 5）标准答题模板

- `候选键：...`
- `主属性：...`
- `非主属性：...`
- `函数依赖：A→B, C→D ...`
- `因此最高范式：...`

#### 6）典型易错点

- 漏看“左侧只有一个属性”这一限制
- 把候选键直接写成某个单属性
- 只看业务直觉，不做闭包验证

---

### 3.5 错题类型：SQL 填空题

#### 1）常见考法

- CREATE TABLE 的完整性约束
- SELECT 查询、连接、排序、分组
- 子查询、EXISTS / NOT EXISTS
- UNION / UNION ALL / EXCEPT
- 插入、删除、更新语句
- 视图、聚合函数、集合运算

#### 2）我的常见错误

- `REFERENCES`、`CHECK`、`ORDER BY` 等关键字拼错
- `COUNT` 和 `SUM` 混用
- 排序题漏写 `DESC`
- JOIN 条件写反或漏写
- `NOT EXISTS` 和 `EXCEPT` 语义会，但空缺位置不对
- `WHERE` 和 `GROUP BY` 顺序混乱

#### 3）正确解题步骤

1. 先判断空缺属于哪类：表名、关键字、条件、连接、排序、聚合。
2. 看题干要的是行数、总量、平均值还是是否存在。
3. 再选对应语法模板。
4. 最后检查拼写、方向、大小写、表名列名。

#### 4）考前速记口诀 / 判断规则

- **删行用 DELETE，删表用 DROP。**
- **授权用 GRANT，回收用 REVOKE。**
- **个数用 COUNT，总和用 SUM，平均用 AVG。**
- **从大到小一定写 DESC。**
- **SQL 填空先看“空缺语法位置”，不是只看语义。**

#### 5）标准答题模板

- `SELECT 字段列表`
- `FROM 表1 JOIN 表2 ON ...`
- `WHERE 条件`
- `GROUP BY 分组字段`
- `HAVING 分组条件`
- `ORDER BY 字段 DESC`
- `EXISTS/NOT EXISTS`
- `UNION / UNION ALL / EXCEPT`

#### 6）典型易错点

- 把 `ORDER` 忘成 `ORDER BY`
- 连接条件左右表名写反
- `SUM(记录条数)` 这种错误表达
- 外键约束写成普通条件
- `NULL` 用 `=` 判断

---

### 3.6 错题类型：视图、游标、触发器、存储过程

#### 1）常见考法

- 写视图定义
- 用游标更新数据
- 编写触发器
- 编写存储过程
- 处理异常与事务提交/回滚

#### 2）我的常见错误

- `COUNT` / `SUM` / `AVG` 混用
- `NEW` / `OLD` 方向写错
- `cursor / open / fetch / close` 顺序不稳
- 异常处理时忘了 `COMMIT` / `ROLLBACK`
- 视图中分组字段与聚合字段不对应

#### 3）正确解题步骤

1. 先看题目要的是“定义”还是“操作过程”。
2. 视图题先确定：分组字段 + 聚合字段。
3. 游标题先写：declare → open → fetch → loop → close。
4. 触发器题区分 `BEFORE` / `AFTER`，`INSERT/UPDATE/DELETE`。
5. 过程题最后检查 `COMMIT / ROLLBACK`。

#### 4）考前速记口诀 / 判断规则

- **个数用 COUNT，总量用 SUM，平均用 AVG。**
- **NEW 是新值，OLD 是旧值。**
- **游标四步：声明、打开、提取、关闭。**
- **成功提交 COMMIT，异常回滚 ROLLBACK。**

#### 5）标准答题模板

- 视图：`CREATE VIEW 视图名 AS SELECT ... GROUP BY ...`
- 游标：`DECLARE cursor_name CURSOR FOR ...` / `OPEN` / `FETCH` / `CLOSE`
- 触发器：`CREATE TRIGGER ... BEFORE/AFTER INSERT/UPDATE/DELETE ON ... FOR EACH ROW`
- 存储过程：`BEGIN ... EXCEPTION ... END`

#### 6）典型易错点

- 没区分“统计条数”和“统计数量总和”
- 触发器把 `NEW` 写成普通字段名
- 游标程序只写了过程框架，漏具体字段
- 过程题异常处理缺少回滚

---

### 3.7 错题类型：事务、并发控制、锁协议

#### 1）常见考法

- 判断脏读、不可重复读、幻读、丢失修改
- 分析调度表中 T4 / T7 / T8 的读值
- 解释共享锁、排它锁
- 说明两段锁协议 2PL
- 判断是否可以避免死锁

#### 2）我的常见错误

- 把丢失修改误判成脏读
- 把“值变了”和“记录数变了”混淆
- 2PL 的“两段”表述不严谨
- 认为 2PL 可以避免死锁
- 锁协议和并发异常没对应起来

#### 3）正确解题步骤

1. 先看是**读未提交**还是**写覆盖**。
2. 再判断是脏读 / 不可重复读 / 幻读 / 丢失修改。
3. 锁题先区分 S 锁与 X 锁。
4. 2PL 题固定写“先加锁后解锁，解锁后不再加锁”。
5. 死锁题记住：2PL 不能避免死锁，只能保证可串行化。

#### 4）考前速记口诀 / 判断规则

- **读未提交 = 脏读。**
- **同一行值变 = 不可重复读。**
- **记录数变 = 幻读。**
- **写被覆盖 = 丢失修改。**
- **S 锁共读，X 锁独占。**
- **2PL：先扩展（加锁），后收缩（解锁）。**

#### 5）标准答题模板

- `脏读：读到了未提交的数据`
- `不可重复读：同一事务前后两次读结果不同`
- `幻读：前后两次查询记录数不同`
- `丢失修改：后写覆盖先写`
- `两段锁协议：先加锁后解锁，解锁后不再加锁`
- `2PL 不能避免死锁，可采用死锁检测/预防/超时机制`

#### 6）典型易错点

- 把“更新被覆盖”写成“脏读”
- 忘记 2PL 的“解锁后不能再加锁”
- 题干问“是否可避免死锁”，答成“可以”
- S 锁 / X 锁在选择题里看反

### 3.8 错题类型：数据库恢复与日志

#### 1）常见考法

- 检查点恢复
- 事务故障 / 系统故障 / 介质故障区分
- Undo / Redo 判断
- 后备副本、日志副本、恢复步骤

#### 2）我的常见错误

- 把系统故障写成事务故障
- 不会分已提交和未提交事务
- 检查点恢复步骤表述不规范
- 介质故障和系统故障流程混淆
- redo / undo 拼写或方向不稳

#### 3）正确解题步骤

1. 先判断故障类型：事务故障 / 系统故障 / 介质故障。
2. 再看检查点位置和提交状态。
3. 已提交但未稳定的事务 → Redo。
4. 未提交事务 → Undo。
5. 介质故障先恢复后备副本，再结合日志重做。

#### 4）考前速记口诀 / 判断规则

- **已提交看 Redo，未提交看 Undo。**
- **检查点恢复：扫日志，分列表，先 Undo，后 Redo。**
- **介质故障：先后备副本，再日志重做。**

#### 5）标准答题模板

- `反向扫描日志，确定 redo/undo 事务列表`
- `对未提交事务执行 undo`
- `对已提交事务执行 redo`
- `介质故障：装入最新后备副本，再重做已完成事务`

#### 6）典型易错点

- 把 crash/system failure 写成 transaction failure
- 只写“恢复”，不写“redo/undo 分类”
- 先写操作再写判断，顺序乱

---

### 3.9 错题类型：数据库安全与权限

#### 1）常见考法

- 授权与回收权限
- 外键约束 `REFERENCES`
- 完整性约束 `PRIMARY KEY / FOREIGN KEY / CHECK / NOT NULL`

#### 2）我的常见错误

- `REFERENCES` 拼写错
- 表名、列名写错
- 把外键约束写成普通条件
- `NOT NULL` 与 `CHECK` 混淆

#### 3）正确解题步骤

1. 先判断约束类型。
2. 再写标准关键字。
3. 再填表名和列名。
4. 最后检查拼写。

#### 4）考前速记口诀 / 判断规则

- **给权 GRANT，收权 REVOKE。**
- **外键约束固定写 REFERENCES。**
- **范围约束优先 CHECK。**

#### 5）标准答题模板

- `PRIMARY KEY(...)`
- `FOREIGN KEY(...) REFERENCES 表名(主键列)`
- `CHECK(条件)`
- `NOT NULL`
- `GRANT ... TO ... WITH GRANT OPTION`
- `REVOKE ... FROM ... [CASCADE|RESTRICT]`

#### 6）典型易错点

- 关键字拼写不稳
- 把 `REFERENCES` 写成 `REFERENCE` / 其他错误拼法
- `CHECK` 条件漏字段名

---

### 3.10 错题类型：分布式数据库与新型数据库

#### 1）常见考法

- CAP 理论
- 数据透明性
- NoSQL 分类

#### 2）我的常见错误

- 概念能背，但容易写反
- `CAP` 忘了“最多满足两个”
- 透明性种类混乱

#### 3）正确解题步骤

1. 先识别题眼：CAP / 透明性 / NoSQL。
2. 再背定义。
3. 再用排除法。

#### 4）考前速记口诀 / 判断规则

- **CAP：最多满足两个。**
- **常见透明性：分片、位置、模型。**
- **NoSQL 四大类：键值、列存储、文档、图。**

#### 5）标准答题模板

- `CAP = 一致性 + 可用性 + 分区容错性，最多满足两个`
- `透明性：分片透明性、位置透明性、模型透明性`
- `NoSQL：键值、列、文档、图`

#### 6）典型易错点

- 把“代码透明性”当成标准概念
- 把 NoSQL 四类写成关系型数据库等错误项

---

### 3.11 错题类型：数据仓库与 OLTP / OLAP

#### 1）常见考法

- OLTP 与 OLAP 定义判断
- 事务处理与分析处理区别

#### 2）我的常见错误

- 记反
- 题干中“事务”与“分析”没有建立关键词映射

#### 3）正确解题步骤

1. 看题干是否强调交易、更新、写入。
2. 若强调分析、统计、决策 → OLAP。
3. 若强调业务处理、在线事务 → OLTP。

#### 4）考前速记口诀 / 判断规则

- **T = Transaction → OLTP。**
- **A = Analysis → OLAP。**

#### 5）标准答题模板

- `OLTP：联机事务处理`
- `OLAP：联机分析处理`

#### 6）典型易错点

- 只记英文缩写，不记中文含义
- 看到“在线”就乱选

## 4. SQL 类题目考前模板

### 4.1 SELECT 查询模板

#### 单表查询

```sql
SELECT 列1, 列2
FROM 表名
WHERE 条件;
```

#### 多表连接

```sql
SELECT ...
FROM A, B
WHERE A.x = B.x;
```

或

```sql
SELECT ...
FROM A JOIN B ON A.x = B.x;
```

#### GROUP BY

```sql
SELECT 分组列, 聚合函数(列)
FROM 表
WHERE 行条件
GROUP BY 分组列;
```

#### HAVING

```sql
SELECT 分组列, 聚合函数(列)
FROM 表
GROUP BY 分组列
HAVING 分组后条件;
```

#### ORDER BY

```sql
ORDER BY 列 ASC;
ORDER BY 列 DESC;
```

#### 子查询

```sql
SELECT ...
FROM 表
WHERE 列 IN (SELECT ...);
```

```sql
SELECT ...
FROM 表
WHERE EXISTS (SELECT ...);
```

```sql
SELECT ...
FROM 表
WHERE NOT EXISTS (SELECT ...);
```

#### IN / NOT IN

- `IN`：属于集合
- `NOT IN`：不属于集合

#### EXISTS / NOT EXISTS

- `EXISTS`：子查询有结果就真
- `NOT EXISTS`：子查询无结果才真

#### 联考速记

- **先筛行，再分组，再筛组。**
- **子查询先判断：是查集合还是查单值。**
- **从大到小必须加 DESC。**

---

### 4.2 CREATE TABLE 完整性约束模板

```sql
CREATE TABLE 表名 (
    列1 数据类型 PRIMARY KEY,
    列2 数据类型 NOT NULL,
    列3 数据类型,
    列4 数据类型,
    FOREIGN KEY(列3) REFERENCES 另一表(主键列),
    CHECK(条件)
);
```

#### 常见写法

- 主键：`PRIMARY KEY(列)`
- 外键：`FOREIGN KEY(列) REFERENCES 表(列)`
- 非空：`NOT NULL`
- 唯一：`UNIQUE`
- 范围：`CHECK(...)`

#### 考场提醒

- `REFERENCES` 不能拼错
- `CHECK` 一定写全表达式
- 外键要引用对方的主键列

---

### 4.3 视图、索引、触发器、存储过程模板

#### 视图

```sql
CREATE VIEW 视图名 AS
SELECT ...
FROM ...
GROUP BY ...;
```

- “个数”常用 `COUNT(*)`
- “总量”常用 `SUM(字段)`
- “平均”常用 `AVG(字段)`

#### 游标

```sql
DECLARE cursor_name CURSOR FOR SELECT ...;
OPEN cursor_name;
FETCH cursor_name INTO ...;
CLOSE cursor_name;
```

#### 触发器

```sql
CREATE TRIGGER 触发器名
BEFORE|AFTER INSERT|UPDATE|DELETE ON 表名
FOR EACH ROW
BEGIN
    -- NEW / OLD
END;
```

#### 存储过程

```sql
CREATE PROCEDURE 过程名(...)
BEGIN
    -- SQL语句
EXCEPTION
    -- 异常处理
END;
```

#### 易错提醒

- `NEW` = 新值，`OLD` = 旧值
- 游标顺序：`DECLARE → OPEN → FETCH → CLOSE`
- 过程题最后检查 `COMMIT` / `ROLLBACK`

## 5. 数据库设计类题目考前模板

### 5.1 E-R 图识别方法

1. 找实体：独立对象。
2. 找属性：对象特征。
3. 找联系：实体间关系。
4. 判断基数：1:1、1:n、m:n。
5. 看是否有联系属性。

### 5.2 实体、属性、联系判断方法

- 能独立存在的是实体
- 描述实体的是属性
- 连接两个实体的是联系
- 联系上附带的信息是联系属性

### 5.3 E-R 图转关系模式规则

- 实体 → 表
- 1:n → 在 n 端加外键
- 1:1 → 一端加外键，必要时可合并
- m:n → 单独建联系表
- 联系属性 → 放进联系表

### 5.4 主键、外键判断方法

- 主键：唯一标识一条记录
- 外键：引用别的表主键
- 联系表主键：通常是双方主键组合，必要时加区分属性

### 5.5 联系类型转换规则

- **1:1**：任选一端加外键
- **1:n**：n 端加外键
- **m:n**：建新表，主键用双方主键组合

### 5.6 弱实体、联系属性、多值属性

- 弱实体：主键不完整，依赖强实体
- 联系属性：跟联系走，放联系表
- 多值属性：通常拆表存储

---

## 6. 规范化与函数依赖考前模板

### 6.1 函数依赖判断方法

1. 先写出题干给出的所有依赖。
2. 求候选键。
3. 标主属性、非主属性。
4. 判断部分依赖、传递依赖。

### 6.2 部分函数依赖

- **组合键的一部分 → 非主属性**
- 这就是部分依赖
- 常导致不满足 2NF

### 6.3 传递函数依赖

- **主键 → 非主属性 A，A → 非主属性 B**
- 这就是传递依赖
- 常导致不满足 3NF

### 6.4 候选键求解步骤

1. 先找必须出现的属性。
2. 求闭包。
3. 看能否推出全部属性。
4. 最小化属性集。

### 6.5 1NF、2NF、3NF、BCNF 判断规则

- **1NF**：属性原子性
- **2NF**：无部分依赖
- **3NF**：无传递依赖
- **BCNF**：每个函数依赖左部都是码

### 6.6 模式分解判断方法

1. 先找候选键。
2. 再看是否有部分/传递依赖。
3. 按依赖拆分。
4. 检查主键、外键。

### 6.7 保持函数依赖与无损连接判断

- 分解后要尽量保持依赖
- 无损连接看能否无信息丢失地还原原表
- 题目若只问“是否满足某范式”，先按依赖判断

### 6.8 易错提醒

- 组合键题最容易错在“只看单属性依赖”
- 规范化题先找键，再判依赖，再分解
- 分解后忘写主键外键会丢分

## 7. 事务、并发与恢复考前模板

### 7.1 事务 ACID

- A：原子性
- C：一致性 / 持续性（按题意）
- I：隔离性
- D：持久性

> 下午题里常考的是**持续性 / 持久性**，不要随手写成一致性。

### 7.2 并发异常

- **脏读**：读到未提交数据
- **不可重复读**：同一行前后值变了
- **幻读**：前后查询记录数变了
- **丢失修改**：后写覆盖前写

### 7.3 共享锁 S 锁

- 允许并发读
- 不允许写冲突
- 共享锁不能与排它写同时作用在同一数据项上

### 7.4 排它锁 X 锁

- 独占
- 读写都受限
- 用于写操作保护

### 7.5 两段锁协议 2PL

- **第一阶段：加锁（扩展阶段）**
- **第二阶段：解锁（收缩阶段）**
- **一旦开始解锁，就不能再申请新锁**

### 7.6 冲突可串行化判断

- 通过前驱图 / 优先图判断
- 无环 ⇒ 冲突可串行化
- 有环 ⇒ 不是冲突可串行化

### 7.7 优先图 / 前驱图判断

1. 把每个事务当成顶点。
2. 发生冲突时连边。
3. 判断是否有环。

### 7.8 日志恢复、检查点、UNDO / REDO

- 已提交且未稳定：Redo
- 未提交：Undo
- 检查点恢复：先扫描日志，再分类，再执行恢复
- 介质故障：先恢复后备副本，再结合日志重做

### 7.9 易错提醒

- **脏读**和**丢失修改**一定分开
- 2PL 不能避免死锁，只能保证可串行化
- 检查点恢复不是“直接回滚一切”
- crash/system failure 不要写成 transaction failure

## 9. 我的错题纠偏清单

| 错题问题 | 错因 | 正确记法 | 考前提醒 |
|---|---|---|---|
| ER 图只写属性，没写联系和基数 | 只盯字段，不盯关系 | **实体 + 联系 + 基数 + 主外键** | 设计题先画关系再落表 |
| 水电费缴费记录写成多对多 | 没看出“记录表中已有客户主键” | 一条记录只对应一个客户，通常是 **1:N** | 表里带外键，通常是 N 端 |
| 3NF 题把部分依赖说成传递依赖 | 候选键没先定 | **组合键一部分决定非主属性 = 部分依赖** | 先找键再判断依赖 |
| SQL 外键约束拼错 `REFERENCES` | 关键字记忆不牢 | 外键固定模板：`REFERENCES 表(主键)` | 拼写要逐字母背 |
| ORDER BY 漏写 DESC | 没看题干“从大到小” | **从大到小 = DESC** | 排序题最后检查方向 |
| COUNT / SUM / AVG 混用 | 没区分“个数”和“总量” | 个数=COUNT，总量=SUM，平均=AVG | 先问自己题目要的是哪一种 |
| 未售出商品写成 NOT EXISTS | 语义对了，语法位置不对 | 按标准空缺写 **EXCEPT** 或题目指定模板 | 空缺位置比语义更重要 |
| 丢失修改判成脏读 | 没区分“读未提交”与“写覆盖” | **读未提交=脏读；写覆盖=丢失修改** | 两事务同时写同一数据要警惕丢失修改 |
| 2PL 表述不严谨 | 把别的事务混进来 | **同一事务：先加锁后解锁，解锁后不再加锁** | 两段锁记成“扩展 + 收缩” |
| 检查点恢复写成事务故障 | 故障类型混淆 | crash/system failure 先按系统故障处理 | 先判故障类型，再写 Undo/Redo |

## 10. 最终考前提醒

1. **先拿稳数据库设计题**：实体、联系、基数、主键、外键，一个都别漏。
2. **SQL 填空优先保证结构完整**：关键字、连接、排序、聚合、约束都要对。
3. **规范化题必须按步骤判断**：先找候选键，再看部分依赖，再看传递依赖。
4. **事务并发题要画图或列顺序**：先判异常类型，再判锁协议和结果。
5. **不确定时优先写标准术语和通用模板**：比“看起来像对的”更安全。

---

> 临考一句话：
>
> **设计题先写主外键，SQL 题先看模板，规范化先找键，并发题先判异常，恢复题先分故障。**
