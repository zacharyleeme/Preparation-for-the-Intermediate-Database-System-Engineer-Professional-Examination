# 软考数据库系统工程师下午题：1页终极背诵版

> 适用场景：考前最后 30～60 分钟快速扫读  
> 来源：2020～2023 下午题真实错题复盘  
> 核心目标：少丢模板分、少犯重复错、把能拿的分拿稳

---

## 1. 数据库设计题：先拿稳主外键

### 必背流程

**实体 → 属性 → 联系 → 基数 → 关系模式 → 主键 → 外键**

### 秒记规则

- **1:n**：外键放 **n 端**。
- **m:n**：单独建联系表，主键通常是双方主键组合。
- 联系有属性：属性放进联系表。
- 表里带着别的表主键：它就是外键端。
- 能唯一标识一条记录，才是主键。

### 标准模板

```text
实体A(主键A, 属性...)
实体B(主键B, 属性...)
联系AB(主键A, 主键B, 联系属性...)
主键：(主键A, 主键B)
外键：主键A REFERENCES A(主键A)，主键B REFERENCES B(主键B)
```

### 必避坑

- 不要只写属性，必须写**联系和基数**。
- 不要只写字段，必须写**主键和外键**。
- 新增实体题，最后一定落成**新增关系模式**。

---

## 2. 规范化 / 函数依赖：先找键

### 必背流程

**函数依赖 → 候选键 → 主属性/非主属性 → 部分依赖 → 传递依赖 → 分解**

### 秒记规则

- **组合键的一部分 → 非主属性** = 部分依赖，不满足 2NF。
- **非主属性 → 非主属性** = 传递依赖，不满足 3NF。
- **BCNF**：每个函数依赖左部都必须是码。
- 分解后每张表都要写**主键、外键**。

### 标准模板

```text
候选键：...
存在部分依赖：X → Y，X 是组合键的一部分，Y 是非主属性。
存在传递依赖：K → X，X → Y。
因此不满足 ...NF。
分解为：A(...), B(...), C(...)
主键：...；外键：...
```

### 必避坑

- 不要把“部分依赖”写成“传递依赖”。
- 不要凭业务直觉乱组主键，先看记录粒度。
- 组合主键不要漏字段。

---

## 3. SQL 填空：先看空缺位置

### 必背顺序

```sql
SELECT ...
FROM ...
WHERE ...
GROUP BY ...
HAVING ...
ORDER BY ... DESC;
```

### 必背约束

```sql
PRIMARY KEY(...)
FOREIGN KEY(...) REFERENCES 表名(主键列)
CHECK(条件)
NOT NULL
UNIQUE
```

### 聚合函数

- **个数**：`COUNT(*)` / `COUNT(列)`
- **总量**：`SUM(列)`
- **平均**：`AVG(列)`
- **从大到小**：必须写 `DESC`

### 集合 / 子查询

```sql
WHERE 列 IN (SELECT ...)
WHERE 列 NOT IN (SELECT ...)
WHERE EXISTS (SELECT ...)
WHERE NOT EXISTS (SELECT ...)
EXCEPT
UNION ALL
```

### 必避坑

- `REFERENCES` 不能拼错。
- `CHECK(stock >= 0)` 不是 `NOT NULL`。
- 未售出/未参加：看题目空缺位置，可能是 `NOT EXISTS`、`NOT IN` 或 `EXCEPT`。
- 语义对不等于填空对，必须符合原 SQL 结构。

## 4. 视图 / 游标 / 触发器 / 存储过程

### 视图模板

```sql
CREATE VIEW 视图名 AS
SELECT 分组列, COUNT(*)/SUM(列)/AVG(列)
FROM 表
GROUP BY 分组列;
```

### 游标模板

```sql
DECLARE cursor_name CURSOR FOR SELECT ...;
OPEN cursor_name;
FETCH cursor_name INTO ...;
CLOSE cursor_name;
```

### 触发器模板

```sql
CREATE TRIGGER 触发器名
BEFORE|AFTER INSERT|UPDATE|DELETE ON 表名
FOR EACH ROW
BEGIN
  -- NEW.字段 / OLD.字段
END;
```

### 秒记规则

- `NEW` = 更新后 / 插入的新值。
- `OLD` = 更新前 / 删除前旧值。
- 游标四步：**声明 → 打开 → 提取 → 关闭**。
- 成功：`COMMIT`；异常：`ROLLBACK`。

### 必避坑

- 题目问销售数量总和，用 `SUM`，不是 `COUNT`。
- 触发器字段引用要写 `NEW.字段` 或 `OLD.字段`。
- 游标更新时，定位字段不能漏。

## 5. 事务并发：先判异常类型

### 四类异常

| 现象 | 类型 |
|---|---|
| 读到未提交数据 | 脏读 |
| 同一行两次读值不同 | 不可重复读 |
| 两次查询记录数不同 | 幻读 |
| 一个事务的修改被另一个覆盖 | 丢失修改 / 丢失更新 |

### 锁、三级封锁协议与 2PL

- **S 锁**：共享读。
- **X 锁**：排它写。
- **一级封锁协议**：修改数据前必须加 X 锁，事务结束才释放 → 解决**丢失修改**。
- **二级封锁协议**：一级 + 读数据前加 S 锁，读完即可释放 → 解决**丢失修改 + 脏读**。
- **三级封锁协议**：一级 + 读数据前加 S 锁，事务结束才释放 → 解决**丢失修改 + 脏读 + 不可重复读**。
- 速记：**一防丢，二防脏，三防不可重读**。
- **2PL 两段锁协议**：
  - 第一阶段：加锁 / 扩展阶段
  - 第二阶段：解锁 / 收缩阶段
  - 一旦解锁，不再申请新锁
- 2PL 能保证冲突可串行化，但**不能避免死锁**。
- 死锁处理：检测、预防、超时、解除。

### 必避坑

- 写覆盖 = 丢失修改，不是脏读。
- 读未提交 = 脏读。
- 2PL 约束的是**同一事务自身**，不是禁止其他事务解锁。

## 6. 恢复 / 日志 / 检查点

### 先判故障类型

- 事务故障：单个事务失败。
- 系统故障：宕机、crash、掉电但介质未坏。
- 介质故障：磁盘损坏、数据库无法启动。

### Undo / Redo

- 已提交但需恢复效果：`REDO`
- 未提交事务影响：`UNDO`
- 检查点恢复：**扫日志 → 分 redo/undo 列表 → undo 未提交 → redo 已提交**

### 介质故障模板

```text
装入最新数据库后备副本，使数据库恢复到最近一次转储时的一致状态；
再装入相应日志文件副本，重做备份后已经提交的事务。
```

### 必避坑

- crash/system failure 不要写成 transaction failure。
- 不要把日志里最后的值直接当恢复值。
- 先按提交状态分类，再算最终值。

## 7. 权限 / 完整性固定表达

```sql
GRANT 权限 ON 对象 TO 用户 WITH GRANT OPTION;
REVOKE 权限 ON 对象 FROM 用户 CASCADE;
PRIMARY KEY(...)
FOREIGN KEY(...) REFERENCES 表(...)
CHECK(字段 >= 0)
NOT NULL
```

### 秒记

- 给权：`GRANT`
- 收权：`REVOKE`
- 外键：`REFERENCES`
- 范围：`CHECK`

---

## 8. 最后 10 分钟只背这些

- 设计题：**实体、联系、基数、关系模式、主键、外键**。
- 1:n 外键放 n 端；m:n 建联系表。
- 规范化：**先找候选键，再判部分依赖、传递依赖**。
- 部分依赖：组合键一部分决定非主属性。
- 传递依赖：非主属性决定非主属性。
- BCNF：函数依赖左边必须是码。
- SQL：`SELECT FROM WHERE GROUP BY HAVING ORDER BY DESC`。
- 个数 `COUNT`，总量 `SUM`，平均 `AVG`。
- 外键约束：`REFERENCES`。
- 范围约束：`CHECK`。
- NEW 新值，OLD 旧值。
- 游标：`DECLARE → OPEN → FETCH → CLOSE`。
- 脏读=读未提交；丢失修改=写覆盖。
- 三级封锁协议：**一防丢，二防脏，三防不可重读**。
- 2PL：先加锁后解锁，不能避免死锁。
- 恢复：已提交 Redo，未提交 Undo。
- 检查点：扫日志、分列表、Undo、Redo。
- 介质故障：先后备副本，再日志重做。

---

## 9. 临场答题策略

1. **数据库设计题先拿结构分**：关系模式、主键、外键必须写。
2. **SQL 填空先保语法分**：关键字、表名、列名、排序方向别错。
3. **规范化题别跳步**：候选键没找准，后面全危险。
4. **并发题先列读写顺序**：读未提交/写覆盖一眼分清。
5. **恢复题先分类故障**：事务故障、系统故障、介质故障流程不同。

> 最后一口气背：
>
> **设计看主外键，SQL 看模板，范式先找键，并发判异常，恢复分故障。**
