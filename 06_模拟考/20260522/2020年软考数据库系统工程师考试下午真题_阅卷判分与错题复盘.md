## 四、错题复盘

### 错题 1：第1题-问题1（ER 图补全）

**错题原题**

```text
某小区快递驿站代为收发各家快递公司的包裹，需要开发信息系统。
需求包括：
1. 记录快递公司和快递员信息；一个快递公司可以有若干快递员，一个快递员只能属于一家快递公司。
2. 记录客户信息。
3. 记录包裹信息，包括包裹编号、到达时间、客户手机号码和快递员手机号码。
【问题1】根据问题描述，补充图1-1的实体联系图。（6分）
```

**错误答案 / 作答情况**

```text
补充快递公司模式的属性：公司名称，地址，电话
补充快递员模式的属性：姓名，快递员手机号，所属公司名称
补充客户模式的属性：姓名，客户手机号，客户等级
```

**正确答案 / 参考采分点**

- 需补充实体及属性：快递公司、快递员、客户、包裹等。
- 需体现联系及基数：
  - 快递公司 — 快递员：一对多。
  - 快递员 — 包裹：一对多。
  - 客户 — 包裹：一对多。
- 包裹应包含编号、到达时间等属性，并通过客户手机号码、快递员手机号码与客户、快递员建立联系。

**答案解析**

本题不是单纯补充实体属性，而是补全 ER 图。ER 图采分通常看三类信息：实体、属性、联系及联系基数。作答只写了部分快递公司、快递员、客户属性，没有明确补充包裹实体，也没有写出快递公司与快递员、客户与包裹、快递员与包裹之间的联系和 1:N 基数，因此只能部分得分。

**避免方法**

- ER 图题先按“实体 → 属性 → 联系 → 基数”四步检查。
- 题干出现“一个 A 可以有多个 B，一个 B 只能属于一个 A”，立即写 1:N。

### 错题 2：第1题-问题3（水电费缴费记录实体与联系）

**错题原题**

```text
若快递驿站还兼有代缴水电费业务，请增加新的“水电费缴费记录”实体，并给出客户和水电费缴费记录之间的“缴纳”联系。
“水电费缴费记录”实体包括编号、客户手机号码、缴费类型、金额和时间。
请给出“水电费缴费记录”的关系模式，并说明其完整性约束。（5分）
```

**错误答案 / 作答情况**

```text
“水电费缴费记录”模式包含属性：编号，客户手机号码，缴费类型，金额，时间
客户与水电费缴纳记录为多对多联系
```

**正确答案**

```text
水电费缴费记录（编号，客户手机号码，缴费类型，金额，时间）
编号为主键；客户手机号码为外键，引用客户（客户手机号码）。
客户与水电费缴费记录之间通常为一对多联系：一个客户可有多条缴费记录，一条缴费记录只对应一个客户。
```

**答案解析**

关系模式属性写对，但联系基数错误。缴费记录中已经包含“客户手机号码”，说明每条缴费记录对应一个客户；一个客户可以多次缴费，产生多条记录，因此应为 1:N，而不是 M:N。完整性约束还应说明主键和外键。

**避免方法**

- 如果某个实体表中直接含有另一个实体的主键，通常意味着 N 端持有外键。
- 先判断“单条记录能不能对应多个客户”，如果不能，多对多就不成立。

### 错题 3：第2题-问题2（3NF 分解原因表述）

**错题原题**

```text
如果要将人员分配表规范化为满足3NF，请用100字以内的文字简要说明解决方案。（3分）
```

**错误答案 / 作答情况**

```text
满足3NF 需要消除传递函数依赖，可以将当前人员分配表拆分为：
人员信息表（人员编号，人员姓名）
人员职责表（人员编号，小区编号，人员职责）
小区信息表（小区编号，物业经理姓名）
```

**正确答案**

```text
拆分为三个表：
人员表（人员编号，人员姓名）
小区表（小区编号，物业经理姓名）
责任表（人员编号，小区编号，人员职责）
```

**答案解析**

拆分结果正确，但原因表述有偏差。本题原表候选键为（人员编号，小区编号），存在：人员编号→人员姓名、小区编号→物业经理姓名，即非主属性对组合主键的部分函数依赖，因此首先是不满足 2NF。拆分的关键是消除部分依赖，而不是传递依赖。

**避免方法**

- 组合键题先查“部分依赖”。
- 单属性决定非主属性，且该单属性只是组合键的一部分，基本就是部分依赖。

### 错题 4：第3题-问题1（建表完整性约束 SQL）

**错题原题**

```text
请将创建订单表的 SQL 语句补充完整，要求定义实体完整性约束、参照完整性约束，以及其他完整性约束。
```

**错误答案 / 作答情况**

```text
(a)：prensece on CUSTOMERS(Con)
(b)：prensence on GOODS(Gno)
(c)：check Onumber between 1 and 99
```

**正确答案**

```sql
(a) REFERENCES CUSTOMERS(Cno)
(b) REFERENCES GOODS(Gno)
(c) CHECK(Onumber >= 1 AND Onumber <= 99)
-- 或 CHECK(Onumber BETWEEN 1 AND 99)
```

**答案解析**

`REFERENCES` 是参照完整性约束关键字，拼写必须正确；客户编号应为 `Cno`，不能写成 `Con`。`CHECK` 的区间语义基本正确，但标准 SQL 写法应为 `CHECK(...)`，且字段名和表达式要完整。

**避免方法**

- 外键约束固定模板：`REFERENCES 表名(主键列)`。
- SQL 填空题拼写错误通常直接扣分，关键字必须按原样背。

### 错题 5：第3题-问题2（查询金额排序）

**错题原题**

```text
查询所有订单的详细情况，输出订单号、客户姓名、商品名称、单价、数量和金额，查询结果按照金额从大到小排列。
```

**错误答案 / 作答情况**

```text
(d)：Oprice * Onumber
(e)：CUSTOMERS.Cno = ORDERS.Cno
(f)：GOODS.Gno = ORDERS.Gno
(g)：order
(h)：Oamount
```

**正确答案**

```sql
(d) Oprice * Onumber
(e) ORDERS.Cno = CUSTOMERS.Cno
(f) ORDERS.Gno = GOODS.Gno
(g) ORDER
(h) Oamount DESC 或 6 DESC
```

**答案解析**

金额表达式、连接条件、`ORDER` 关键字基本正确；但题干明确要求“按照金额从大到小排列”，必须体现降序 `DESC`。只写 `Oamount` 默认通常为升序，不满足题意。

**避免方法**

- 题干出现“从大到小 / 降序 / 由高到低”，答案必须有 `DESC`。
- 排序题最后一空优先检查 `ASC/DESC`。

### 错题 6：第3题-问题3（创建已售商品视图）

**错题原题**

```text
创建已售商品信息视图，给出已售商品的编号、名称、订单个数及平均每单商品数量。
```

**错误答案 / 作答情况**

```text
(i)：view
(j)：sum(Onum)
(k)：avg(GAnum)
(l)：group
(m)：ORDERS.Gno
```

**正确答案**

```sql
(i) VIEW
(j) COUNT(*) 或 COUNT(Ono) 或 COUNT(1)
(k) AVG(Onumber)
(l) GROUP
(m) ORDERS.Gno
```

**答案解析**

订单个数应使用 `COUNT`，不能用 `SUM(Onum)`；平均每单商品数量应对订单表中的数量字段 `Onumber` 求平均，即 `AVG(Onumber)`，不能对别名 `GAnum` 自身求平均。

**避免方法**

- “个数”= `COUNT`。
- “平均数量”= `AVG(原始数量字段)`。
- 聚合函数不要对尚未生成的别名做聚合。

### 错题 7：第3题-问题4（查询未售出商品）

**错题原题**

```text
查询未售出商品的编号和名称。此功能由 SQL 语句实现，请补全。
```

**错误答案 / 作答情况**

```text
(n)：GOODS
(o)：not exists
```

**正确答案**

```text
(n) GOODS
(o) EXCEPT
```

**答案解析**

`GOODS` 表名正确。该题标准 SQL 空缺语境使用集合差运算 `EXCEPT`，表示从商品集合中排除已售商品集合。`NOT EXISTS` 在另一个完整 SQL 模板中可以表达类似语义，但不符合本题给定空缺位置的标准答案。

**避免方法**

- SQL 填空要看“空缺所在语法位置”，不要只看语义可替代。
- 未售出 = 商品集合 − 已售商品集合，集合差常用 `EXCEPT`。

### 错题 8：第4题-问题1（游标更新程序）

**错题原题**

```text
基于游标，用 SQL 实现价格更新程序，请补全空缺处代码。
```

**错误答案 / 作答情况**

```text
(a)：cursor
(b)：open
(c)：Pprice
(d)：commit
```

**正确答案**

```text
(a) cursor
(b) open
(c) Pno，Pprice，Mno
```

### 错题 9：第4题-问题2（并发调度不一致性类型）

**错题原题**

```text
T0 时刻 A 商品库存 p_nums 为 100。
请说明 T4、T7 时刻，用户3事务读取到的 p_nums 数值分别是多少。
请说明 T8 时刻事务调度结果是否正确？若不正确请说明属于哪一种数据不一致性。
```

**错误答案 / 作答情况**

```text
T4 时刻读取到 100；T7 时刻读取到 99。
T8 时刻事务调度结果不正确，当前问题属于脏读。
```

**正确答案**

```text
T4 时刻 p_nums = 100。
T7 时刻 p_nums = 99。
T8 时刻事务调度结果不正确，属于丢失修改。
```

**答案解析**

两个读取值判断正确，调度结果不正确也判断正确；但不一致性类型错误。本题两个用户同时购买同一商品，都基于相同或过期库存值进行修改，导致其中一个事务的修改被另一个事务覆盖，属于丢失修改。脏读强调“读到了另一个事务未提交的数据”，本题核心是更新覆盖。

**避免方法**

- 读未提交 = 脏读。
- 两个事务都修改同一数据，后提交覆盖先提交 = 丢失修改。
- 同一事务两次读同一行值不同 = 不可重复读。

### 错题 10：第4题-问题3（两段锁协议与死锁处理）

**错题原题**

```text
为保证并发事务的正确性，系统要求所有事务需遵循两段锁协议。
（1）解释两段锁协议，并说明“两段”的含义。
（2）说明两段锁协议是否可以避免死锁？如不能避免，应采取什么措施解决死锁问题。
```

**错误答案 / 作答情况**

```text
（1）两段锁协议是指按顺序对事务进行加锁和解锁操作，两段的含义是分为加锁和解锁阶段，进入加锁阶段后，其他事务不能进行解锁。进入解锁阶段后事务不能进行加锁。
（2）两段锁协议只能确保事务串行化执行，不能消除死锁。如果需要消除死锁需要确保事务不能持有已加锁数据的锁。
```

**正确答案**

```text
两段锁协议：事务在对任何数据进行读写之前必须先对该数据加锁；在释放一个封锁之后，该事务不再申请和获得任何其他锁。
“两段”：第一阶段为获得封锁的扩展阶段，第二阶段为释放封锁的收缩阶段。
两段锁协议不能避免死锁；可采用死锁检测机制，发现死锁后按一定算法解除死锁。
```

**答案解析**

作答抓到了“加锁阶段/解锁阶段”和“解锁后不能再加锁”的核心，但“进入加锁阶段后，其他事务不能进行解锁”表述错误，两段锁约束的是同一个事务自身的加锁/解锁行为，不是禁止其他事务解锁。死锁处理措施也不准确，标准答案强调死锁检测和解除机制。

**避免方法**

- 两段锁协议约束的是“同一事务”内部行为。
- 背固定句：先扩展只加锁，后收缩只解锁；一旦解锁，不再加锁。
- 2PL 不能防死锁，死锁靠检测、预防或超时等机制处理。

### 错题 11：第5题-问题3（基于检查点的恢复过程）

**错题原题**

```text
请用100字以内的文字，简要描述系统出错后，基于检查点的恢复过程。（5分）
```

**错误答案 / 作答情况**

```text
因为当前系统故障为事务故障，所以系统出错后需要先重启数据库系统。然后再对日志中记录的事务进行恢复。由于当前日志中已经创建了检查点，所以只需要分析并恢复检查点前后，到事务故障前的事务。对日志中在故障前已经完成提交的事务进行rodo重做，对日志中故障前未完成提交的事务进行 Undo撤销操作。
```

**正确答案**

```text
步骤1：反向扫描日志文件，确定需要 redo 的事务和需要 undo 的事务。
步骤2：对需要 undo 的事务撤销已经执行的操作。
步骤3：对需要 redo 的事务重新执行已执行的操作。
```

**答案解析**

redo/undo 的基本方向正确，但故障类型表述错误：题干为系统出错/crash 场景，不应称为事务故障。检查点恢复过程应强调“从日志反向扫描，确定 redo/undo 列表，再执行 undo 和 redo”。作答中“恢复检查点前后”表述不够严谨，且 `redo` 拼写为 `rodo`。

**避免方法**

- 检查点恢复固定三步：扫日志 → 定 redo/undo → undo 未完成、redo 已提交。
- crash/system failure 不要写成 transaction failure。
- 专业术语 `redo / undo` 拼写要稳定。

## 五、错题原因归纳

| 失分类型 | 具体表现 | 纠偏建议 |
|---|---|---|
| ER 图表达不完整 | 只写属性，漏联系和基数 | ER 题固定检查“实体、属性、联系、基数” |
| 范式原因表述不准 | 分解结果对，但把部分依赖说成传递依赖 | 组合键先查部分依赖，非主属性链式依赖才是传递依赖 |
| SQL 关键字与字段拼写错误 | `REFERENCES` 拼错，`Cno` 写成 `Con` | SQL 填空关键字必须逐字母背准 |
| 聚合函数误用 | `COUNT` 写成 `SUM`，`AVG(Onumber)` 写成 `AVG(GAnum)` | “个数=COUNT，平均=AVG(原始字段)” |
| 并发异常类型混淆 | 丢失修改误判为脏读 | 两事务更新覆盖=丢失修改；读未提交=脏读 |
| 恢复过程表述不规范 | 系统故障写成事务故障，恢复步骤不完整 | 检查点恢复背三步：扫描日志、确定列表、undo/redo |

## 六、后续复盘重点

1. **ER 图题**：不要只写关系模式，要画出/描述联系和 1:N、M:N 基数。
2. **SQL 填空题**：关键字拼写、字段名、排序方向、聚合函数是主要扣分点。
3. **并发控制题**：重点区分丢失修改、脏读、不可重复读、幻读。
4. **两段锁协议**：记住它保证冲突可串行化，但不能避免死锁。
5. **恢复题**：检查点恢复要用标准流程表达，避免把 crash 写成事务故障。
