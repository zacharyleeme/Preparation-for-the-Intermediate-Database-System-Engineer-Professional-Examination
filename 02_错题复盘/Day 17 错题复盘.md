### 第27题（下午题）

**题型归属：** 物理设计 / 场景分析

**题目：**  
某日志表 `Log(log_id, user_id, event_type, event_time, detail)` 每天新增数据量很大，常见查询是：
- 按 `event_time` 范围查询；
- 按 `event_type` 统计；
- 偶尔按 `user_id` 查单个用户日志。

请说明：在物理设计层面，优先应关注哪些查询瓶颈与设计要点。

**你的实际作答：**  
- 优先关注 `event_time` 、`event_type` 和 `user_id` 字段
- 建议在 `event_time` 和 `event_type` 字段上创建组合索引，在 `user_id` 字段创建单列索引
- 

**参考答案：**  
这个场景下优先应关注三类瓶颈：第一，大量新增数据下索引维护成本是否过高；第二，按 `event_time` 范围查询和按 `event_type` 统计是否会发生大范围扫描；第三，按 `user_id` 查个人日志时是否需要更快定位。设计时应围绕高频查询路径做有选择的索引，而不是对所有列都平均加索引。

**本题采分点：**
1. 写出大数据量写入带来的维护压力
2. 写出范围查询 / 统计 / 单用户查询这几个核心路径
3. 能说明设计应围绕高频路径做取舍

**本次已命中的采分点：**
- 写到了 `event_time`、`event_type`、`user_id` 这些与核心查询路径相关的字段方向

**本次漏掉的采分点：**
- 没有先写“每天新增数据量很大”带来的索引维护压力这一前置瓶颈
- 没有把“范围查询、统计查询、单用户定位”明确展开成需要分别关注的三类查询路径
- 直接跳到索引方案，没有回到题目更想考的“应围绕高频路径做取舍，而不是平均给所有列加索引”这一设计原则

**失分原因类型：**
- [ ] 不会知识点
- [x] 会概念但不会下手
- [x] 会分析但写不规范
- [ ] 只写结论没写依据
- [x] 漏步骤 / 漏条件 / 漏关键词
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- 这类物理设计题，先不要急着写“建什么索引”；
- 应先判断：写入压力大不大、哪些查询最频繁、哪些路径最值得优先优化；
- 只有把“维护压力 + 高频查询路径 + 取舍原则”写出来，后续的索引建议才更像标准答案。

**标准作答路径：**
1. 先写：该日志表每天新增量大，索引设计首先要考虑写入维护成本不能过高；
2. 再写：当前主要要关注 `event_time` 范围查询、`event_type` 统计和 `user_id` 单用户定位这几类核心访问路径；
3. 最后写：应围绕高频查询路径有选择地设计索引，而不是平均给所有列都加索引。

**纠偏记忆：**
- **物理设计题先写“写入压力 + 高频路径 + 取舍原则”，再谈具体索引方案。**

**建议复习动作：**
1. 按参考答案把本题重写 2 遍；  
2. 再补 2 道“高写入表的物理设计取舍”小题；  
3. 下次看到“每天新增很多”的题干时，先写维护成本，再写查询路径。

### 第29题（下午题）

**题型归属：** 机制协同 / 综合优化表达

**题目：**  
请围绕“索引、SQL 改写、统计信息 / 优化器、物理设计如何共同提升查询性能”写一段 4～6 句标准答案，要求既写出各自作用，也写出为什么合起来有效。

**你的实际作答：**  
- 创建索引可以使SQL沿着更优的执行计划执行，缩小SQL扫描范围
- SQL改写可以让SQL更适合在当前索引环境中执行
- 优化器决定SQL使用更优的优化算子
- 

**参考答案：**  
查询性能优化通常不是单靠一个手段完成。索引负责缩短访问路径、减少扫描范围；SQL 改写负责让查询写法更容易被优化器有效利用；统计信息和优化器帮助数据库选择更合适的执行路径；物理设计则从表和索引组织层面进一步支撑高频查询。正因为这些手段作用层次不同、彼此配合，数据库才能在真实业务场景下获得更稳定的性能提升。

**本题采分点：**
1. 写出索引、SQL 改写、统计信息 / 优化器、物理设计各自作用
2. 能说明为什么它们合起来有效
3. 结尾回到“稳定性能提升”

**本次已命中的采分点：**
- 写到了索引、SQL 改写、优化器这三类手段的部分作用

**本次漏掉的采分点：**
- 没有写出统计信息与物理设计在优化中的明确作用分工
- 没有把“这些手段为什么合起来有效”这一层协同逻辑写出来
- 结尾缺少“数据库因此获得更稳定性能提升”的总结收束句

**失分原因类型：**
- [ ] 不会知识点
- [x] 会概念但不会下手
- [x] 会分析但写不规范
- [ ] 只写结论没写依据
- [x] 漏步骤 / 漏条件 / 漏关键词
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- 综合优化题不是把几个名词并列写出来就够了；
- 要写清：索引解决访问路径，SQL 改写解决写法适配，统计信息 / 优化器解决执行计划选择，物理设计解决底层组织支撑；
- 最后还要补一句：这些层次彼此配合，才能在真实业务场景下形成更稳定的性能提升。

**标准作答路径：**
1. 先写：索引负责缩短访问路径、减少扫描范围；
2. 再写：SQL 改写让查询更容易利用索引，统计信息 / 优化器帮助数据库选择更合适的执行计划；
3. 接着写：物理设计从表和索引组织层面支撑高频查询；
4. 最后写：正因为这些手段分工不同、彼此协同，系统才能获得更稳定的查询性能提升。

**纠偏记忆：**
- **性能优化综合题不能只列手段，要写“各自作用 + 协同原因 + 总结句”。**

**建议复习动作：**
1. 按“索引 → SQL 改写 → 统计信息 / 优化器 → 物理设计 → 协同结论”的顺序重写本题 2 遍；  
2. 单独口头复述一遍四类手段各自作用；  
3. 以后写综合优化题时，强制自己补上最后一句“稳定性能提升”的结论。

## 三、本次练习暴露出的关键易错点

### 1. 知识判断层面的易错点

- 当前对“哪些列更适合建索引、联合索引顺序为什么这样排、哪些 SQL 写法不利于索引利用”的基础判断已经比较稳；
- 但一进入物理设计场景题，还容易直接跳到“建什么索引”，而没有先写“写入压力、查询瓶颈、取舍原则”；
- 对综合优化题中的“索引 / SQL 改写 / 优化器 / 物理设计”已经知道有这些模块，但它们各自分工与协同关系还没有完全写成固定套路。

### 2. 下午题书写层面的易错点

- 有些题方向是对的，但还会漏“为什么更优”或“还有什么代价”这一层；
- 涉及 SQL 改写时，已经能改函数条件，但对“减少返回列、避免 `SELECT *`”的意识还不够稳定；
- 模板题已经开始形成顺序感，但综合题里仍会出现“写到中段就停、结尾总结句没补上”的情况。

### 3. 采分点意识层面的易错点

- 当前最大问题已经不是“完全不会索引优化题”，而是**会写主线，但细节采分点覆盖还不够满**；
- 像“覆盖索引、减少回表、减少返回列、维护成本、存储开销、围绕高频路径取舍”这类短语，后续要尽量主动落笔；
- 综合优化题除了写手段名词，还必须补上“各自作用 + 为什么合起来有效 + 最后一句总结结论”，否则很容易再次出现 1/3 或 2/3 式失分。

## 四、Day 17 需要继续巩固的重点

1. 物理设计题的固定顺序：**先看写入压力 → 再看高频查询路径 → 再做索引取舍**
2. SQL 改写题的固定检查项：**函数 / 前导 `%` / `SELECT *` / 隐式转换 / 返回列过多**
3. 索引设计题的标准句序：**当前瓶颈 → 核心列 → 顺序依据 → 为什么有效 → 代价权衡**
4. 少数返回列场景下，要主动联想到：**是否存在覆盖索引机会、是否可以减少回表**
5. 频繁写入表的索引设计要同时写：**维护成本 + 存储开销 + 围绕核心查询取舍**
6. 综合优化题的标准展开：**索引 → SQL 改写 → 统计信息 / 优化器 → 物理设计 → 协同结论**
7. 所有综合题最后都要补 1 句收束：**这些手段分工不同、彼此配合，才能带来更稳定的性能提升**
