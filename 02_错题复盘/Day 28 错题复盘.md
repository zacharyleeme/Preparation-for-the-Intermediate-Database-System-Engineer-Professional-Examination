### B. 下午题 / 主观题正式错题

#### 第21题（下午题）

**题型归属：** 并发异常分析 / 丢失修改  
**你的实际作答：**
```text
- 执行完并发序列后，X值为3
- 该问题属于丢失修改，因为前一次 Wo的修改结果被后一次 WI覆盖了。在并发操作中 ao 提交了查询操作后，需要对数据X进行加锁，在ao 事务完成提交前，不允许其他事务对数据X进行修改操作。
```

**参考答案：**
- 执行完该并发序列后，数据库中的 `X = 4`；
- 若按正确串行逻辑，最终结果本应为 `X = 3`；
- 这里发生的是 **丢失修改**：后写回覆盖了前一事务的修改效果。

**本题采分点：**
1. 算对最终值 `X = 4`；
2. 指出正确串行结果本应为 `X = 3`；
3. 写出“后写覆盖前写”的原因；
4. 点名异常类型为 **丢失修改**。

**本次已命中的采分点：**
- 能指出异常类型是 **丢失修改**；
- 能说明“前一次修改结果被后一次覆盖”。

**本次漏掉的采分点：**
- 把并发序列执行后的最终值算成了 `3`，标准应为 `4`；
- 没有明确写出：**若按正确串行逻辑，最终应回到 `3`**；
- 结果是把“并发执行后的错误结果”和“串行执行下的正确结果”混在了一起。

**失分原因类型：**
- [ ] 不会知识点
- [ ] 会概念但不会下手
- [x] 会分析但计算 / 判断链未锁死
- [ ] 只写结论没写依据
- [x] 漏关键中间结论
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- 这题要先按并发序列**机械演算最终值**：
  - 出库事务基于旧值 `3` 写回 `2`；
  - 入库事务仍基于旧值 `3` 算出 `4`，再写回；
  - 所以并发执行后的数据库值是 `4`；
- 之后再与正确串行结果比较：正常情况下先减 1 再加 1，或先加 1 再减 1，最终都应为 `3`；
- 正是因为“后写覆盖前写”，所以这是 **丢失修改**。

**建议复习动作：**
1. 重写第21题 2 遍，每次都按“并发最终值 → 正确串行值 → 异常名称”三步写；
2. 再口头复述 1 遍：**丢失修改题先算错值，再说本应是多少。**

#### 第24题（下午题）

**题型归属：** 两段锁协议概念辨析  
**你的实际作答：**
```text
- 因为两段锁协议可以确保数据在被查询或修改后，同一行数据无法被其他事务修改。确保事务可以有序运行。
- 因为两段锁协议只是确定事务加锁和解锁的运行顺序，并不能完全消除锁的存在。
```

**参考答案：**
> 两段锁协议要求事务先进入加锁阶段，再进入解锁阶段；一旦开始释放锁，就不能再申请新锁。这样可以保证冲突操作的先后顺序与某个串行顺序一致，因此能保证可串行化调度。  
> 但 2PL 并不限制多个事务以相反顺序持有并等待对方所需的锁，所以仍可能形成循环等待，从而发生死锁。

**本题采分点：**
1. 写出“两段锁协议 = 先加锁阶段，再解锁阶段”；
2. 写出“能保证可串行化调度”；
3. 写出“仍可能因循环等待而死锁”。

**本次已命中的采分点：**
- 已经意识到 2PL 会约束加锁 / 解锁顺序；
- 已经表达出“事务可以更有序运行”的方向。

**本次漏掉的采分点：**
- 没把“**加锁阶段 / 解锁阶段**”这个标准术语写完整；
- 没明确点出“**可串行化**”这一规范结论；
- 没写出“**循环等待**”才是 2PL 仍会死锁的核心原因。

**失分原因类型：**
- [ ] 不会知识点
- [ ] 会概念但不会下手
- [x] 会分析但写不规范
- [x] 只写了方向，没写标准术语
- [x] 漏关键关键词
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- 2PL 不是笼统地“有锁所以有序”，而是要明确写出：**事务先进入加锁阶段，开始解锁后就不能再申请新锁**；
- 因为冲突操作的先后顺序被这个规则钉住，所以它能保证**可串行化**；
- 但若事务按相反顺序请求锁，仍可能形成**循环等待**，因此 2PL **不能保证一定不会死锁**。

**建议复习动作：**
1. 把第24题压缩成 2 句标准答案并默写 2 遍；
2. 固定背一句：**2PL 保可串行化，不保无死锁。**
3. 把“循环等待”四个字单独写进自己的并发题模板里。

#### 第30题（下午题）

**题型归属：** 死锁分析与改写  
**你的实际作答：**
```text
- 因为T1先对数据项A进行加锁，T2先对数据项B进行了加锁。然后T1向数据B申请加锁时，需要等待T2事务对数据项B先解锁，事务T2对数据项A进行加锁申请，又需要等待T1事务对数据项A进行解锁。两个事务对对方需要解锁的事务进行了加锁，所以形成了死锁
- 在事务T1对数据项A加锁，事务T2对数据项B加锁之后，此时不允许事务对其他数据进行加锁，需要两个事务对当前加锁数据项解锁之后，才允许进行加锁操作。
```

**参考答案：**
- 死锁原因：`T1` 持有 `A` 等 `B`，`T2` 持有 `B` 等 `A`，形成**循环等待**；
- 更安全的改写：让两个事务按**相同顺序申请锁**（例如都先申请 `A`，再申请 `B`），或在无法一次拿到所需锁时回退重试；在满足 2PL 前提下，**统一锁申请顺序**是最常见的防死锁思路。

**本题采分点：**
1. 写出死锁的核心原因是 **循环等待**；
2. 写出更安全的改写思路是 **统一锁申请顺序**；
3. 改写思路不能破坏 2PL。

**本次已命中的采分点：**
- 已经把死锁成因说明清楚，本质上就是 **循环等待**。

**本次漏掉的采分点：**
- 没写出标准防死锁思路：**两个事务按相同顺序申请锁**；
- 你给出的改写更接近“先解锁再去拿别的锁”，这会破坏题目要求保留的 **2PL 约束**；
- 因此第 2 问没有落到考试采分点上。

**失分原因类型：**
- [ ] 不会知识点
- [ ] 会概念但不会下手
- [x] 会分析但改写策略没对准标准答案
- [ ] 只写结论没写依据
- [x] 漏关键防死锁动作
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- 这类题分两步：
  1. 先说为什么会死锁：**循环等待**；
  2. 再说怎么在不破坏 2PL 的前提下尽量避免：**统一加锁顺序**；
- 最稳的标准回答是：**让 T1 和 T2 都先申请 A，再申请 B**，这样就不容易形成相反方向的锁等待链。

**建议复习动作：**
1. 把第30题第二问改写成“统一锁申请顺序”版本并默写 2 遍；
2. 把“2PL 保可串行化，不保无死锁；防死锁常用统一加锁顺序”写进 Day 29 的答题模板。

## 三、本次练习暴露出的关键易错点

### 1. 并发异常题容易把“并发后的错误结果”和“串行下的正确结果”混在一起

第21题最典型：
- 你已经知道异常名称是 **丢失修改**；
- 但最终值计算时，把“本该是多少”写成了“实际并发后是多少”。

一句话纠偏：

> **并发异常题先算并发后的真实结果，再写正确串行值，最后点异常名。**

### 2. 2PL 题会写方向，但“可串行化 / 无死锁”边界还没写硬

第24题、第30题共同暴露：
- 你已经知道 2PL 和加锁顺序有关；
- 但还没有把以下两句完全机械化：
  - **2PL 保可串行化**；
  - **2PL 不保证无死锁**。

一句话纠偏：

> **2PL 的标准答案必须同时出现“可串行化”和“循环等待仍可能死锁”。**

### 3. 隔离级别题要写“异常名称”，不要写泛化系统现象

第25题说明：
- 你知道 `Read Committed` 这一路的方向；
- 但第 3 问应该回答“**不可重复读 / 幻读**”，而不是“阻塞 / 死锁”。

一句话纠偏：

> **隔离级别题优先写：能防什么异常、还剩什么异常。**

### 4. 死锁改写题的标准动作还没完全锁成“统一加锁顺序”

第30题说明：
- 你能把死锁成因讲清楚；
- 但还没把“怎么改得更安全”稳定写成考试爱收的那句：
  - **两个事务按相同顺序申请锁**。

一句话纠偏：

> **死锁分析题先写循环等待，改写题优先写统一锁申请顺序。**

### 5. 恢复主线已经明显稳了，可以作为 Day 29 的承接基础

- `Redo / Undo` 判断比较稳；
