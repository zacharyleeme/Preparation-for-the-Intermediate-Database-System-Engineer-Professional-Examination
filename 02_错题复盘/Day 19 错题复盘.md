### 第19题（下午题）

**题型归属：** 安全设计 / 风险识别

**题目：**  
某企业的数据库中包含员工工资、身份证号、手机号和部门销售数据。现在需要给“经营分析人员”开放查询能力，但他们只需要看按部门汇总后的销售统计结果，不应直接看到员工身份证号、手机号和工资明细。

请说明：这个场景下，至少应先识别哪些核心安全目标与主要风险。

**你的实际作答：**  
- 安全目标：只能查询 **按部门汇总后的销售统计结果**  
- 主要风险：只拥有数据查询权限，没有数据增删改权限；只能查询 **按部门汇总后的销售统计结果** ，不能查询员工身份证号、手机号和工资明细

**参考答案：**  
这个场景下，至少要先识别两类安全目标：第一，保护员工工资、身份证号、手机号等敏感数据的保密性；第二，保障数据库中业务数据不被越权访问或误改。主要风险包括：经营分析人员若直接访问基础表，可能看到不该看的敏感列；若权限设计过宽，还可能带来越权查询、误用数据甚至合规风险。整体上，应先明确“只开放部门汇总结果，而不开放敏感明细”这一保护边界。

**本题采分点：**
1. 写出保密性 / 敏感数据保护目标
2. 写出越权访问、敏感列暴露等风险
3. 能说明“只开放汇总、不开放明细”的边界

**本次已命中的采分点：**
- 写到了“只开放部门汇总结果、不开放敏感明细”这一边界方向

**本次漏掉的采分点：**
- 没有把安全目标明确写成**保密性 / 敏感数据保护**与**防越权访问 / 防误改**
- 把“只能查汇总、不能查明细”写成了目标描述，但没有先抽象出保护对象与风险类型
- “只拥有查询权限、没有增删改权限”更像控制手段或权限状态，不是本题真正要先识别的主要风险

**失分原因类型：**
- [ ] 不会知识点
- [x] 会概念但不会下手
- [x] 会分析但写不规范
- [ ] 只写结论没写依据
- [x] 漏步骤 / 漏条件 / 漏关键词
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- 这类安全设计题，第一步不是直接写“只能查什么”，而是先写**保护什么数据、怕什么风险**；
- 对本题来说，最关键的是敏感数据保密性与防止越权访问 / 误改；
- 只有先把“保护对象 + 风险”写清，后续的角色、视图、只读权限与审计才有落点。

**标准作答路径：**
1. 先写：需要保护工资、身份证号、手机号等敏感数据的保密性；
2. 再写：主要风险是越权查看基础明细、敏感列暴露、权限过宽导致误用或合规风险；
3. 最后补：只应开放部门汇总结果，不应开放敏感明细。

**纠偏记忆：**
- **安全设计题先写“保护对象 + 风险”，再写控制手段。**

**建议复习动作：**
1. 按“保密性 / 越权风险 / 保护边界”的顺序重写本题 2 遍；  
2. 再补 2 道“最小权限 + 敏感列控制”的安全设计小题；  
3. 以后看到“只能看汇总、不能看明细”的题干时，先口头说出：**保密性、越权访问、保护边界。**

### 第22题（下午题）

**题型归属：** 综合表达 / 安全设计模板复写

**题目：**  
请用“保护对象与风险 → 控制手段 → 为什么有效 → 代价与结论”的顺序，围绕上题写一段 4～6 句的标准答案。

**你的实际作答：**  
- 确定安全设计保护对象和可能存在风险  
- 明确采用最小权限，视图、审计、备份、只读权限等控制手段  
- 说明相应控制手段能实现的效果  
- 控制授权可以带来安全保障的同时，也会增加系统维护工作和维护复杂度

**参考答案：**  
当前场景中需要重点保护的是员工工资、身份证号、手机号等敏感数据，主要风险是经营分析人员越权查看基础明细。更合理的设计应基于最小权限原则，为分析人员建立只读角色，并通过汇总视图仅开放部门销售统计结果。这样既能满足分析需求，又能避免直接暴露敏感列，同时配合审计保留关键访问痕迹。需要注意的是，这类分层控制虽然能显著提升安全性，但也会增加权限管理、视图维护与审计存储方面的成本。总体上，应在满足业务分析需求的前提下，优先保障敏感数据的安全边界。

**本题采分点：**
1. 顺序完整：风险 → 手段 → 原因 → 代价 / 结论
2. 写出最小权限、角色、视图、审计中的关键层
3. 有“提升安全性但增加管理成本”的收束句

**本次已命中的采分点：**
- 基本遵守了“风险 → 手段 → 作用 → 代价”的顺序
- 写到了最小权限、视图、审计、只读权限等控制层
- 写到了“会增加维护复杂度”这一代价方向

**本次漏掉的采分点：**
- 表达过于抽象，没有把本题中的**工资、身份证号、手机号、部门汇总结果**这些具体对象落进去
- 控制手段里加入了“备份”，但这不是本题最核心的权限分层手段，反而冲淡了角色 / 视图 / 审计主线
- 结尾虽然写了代价，但还缺少“在满足业务分析需求前提下优先保障安全边界”的收束句

**失分原因类型：**
- [ ] 不会知识点
- [x] 会概念但不会下手
- [x] 会分析但写不规范
- [ ] 只写结论没写依据
- [x] 漏步骤 / 漏条件 / 漏关键词
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- 模板题不能只写空泛框架，必须把**题目场景里的具体保护对象、风险和手段**落下来；
- 本题的主线应是：敏感明细保护 → 最小权限 → 只读角色 → 汇总视图 → 审计留痕；
- 结尾要补一句：方案既满足分析需求，也守住敏感数据边界，但增加管理与维护成本。

**标准作答路径：**
1. 先写：要保护工资、身份证号、手机号等敏感数据，主要风险是越权查看基础明细；
2. 再写：基于最小权限建立只读角色，并通过汇总视图仅开放部门销售统计结果，同时保留审计；
3. 接着写：这样既满足分析需求，又避免敏感列暴露；
4. 最后写：方案会增加权限管理、视图维护与审计存储成本，但更能保障安全边界。

**纠偏记忆：**
- **综合模板题不能只写框架词，必须把题目里的具体对象、具体风险、具体手段写出来。**

**建议复习动作：**
1. 按“对象 / 风险 / 角色 / 视图 / 审计 / 代价”顺序重写本题 2 遍；  
2. 再补 2 道“权限分层 + 汇总视图”的 4～6 句模板题；  
3. 以后写这类题时，检查自己有没有写出**具体对象名词**，而不是只写抽象框架。 

### 第23题（下午题）

**题型归属：** 备份恢复目标判断 / RPO-RTO

**题目：**  
某订单系统要求：
- 最多只能丢失 **15 分钟** 数据；
- 故障后最多允许业务中断 **2 小时**；
- 订单数据需要长期保留，并要求备份文件不能明文裸放。

请说明：该场景中的 **RPO** 和 **RTO** 分别是什么，它们分别在约束什么。

**你的实际作答：**  
- RPO：最多只能丢失 **15 分钟** 数据，约束最多允许丢失的数据量  
- PTO：故障后最多允许业务中断 **2 小时** ，约束故障出现后最长允许的恢复时间

**参考答案：**  
该场景中的 RPO 是“最多只能丢失 15 分钟数据”，它约束的是数据丢失容忍窗口；RTO 是“故障后最多允许业务中断 2 小时”，它约束的是系统恢复服务所需的时间上限。也就是说，方案设计既要考虑如何缩小数据损失，也要考虑恢复步骤不能过慢。

**本题采分点：**
1. 明确写出 RPO = 15 分钟数据丢失容忍
2. 明确写出 RTO = 2 小时业务中断容忍
3. 能说明二者分别约束数据损失和恢复时间

**本次已命中的采分点：**
- 写对了 RPO = 最多只能丢失 15 分钟数据
- 写出了“故障后最多允许业务中断 2 小时”这一恢复时间要求方向

**本次漏掉的采分点：**
- 将 **RTO** 写成了 **PTO**，术语口径错误
- 对 RPO 的解释写成“约束允许丢失的数据量”，但更准确应写成**数据丢失时间窗口 / 数据丢失容忍窗口**
- 这类题不只考大概意思，还考缩写口径与考试标准表述是否稳

**失分原因类型：**
- [ ] 不会知识点
- [x] 会概念但不会下手
- [x] 会分析但写不规范
- [ ] 只写结论没写依据
- [x] 漏步骤 / 漏条件 / 漏关键词
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- **RPO** 是最多能容忍丢失多长时间窗口内的数据；
- **RTO** 是最多能容忍业务中断多长时间；
- 这两个缩写和对应口径必须写稳，不能写成 PTO 或把时间窗口写成数据量概念。

**标准作答路径：**
1. 先写：RPO = 最多只能丢失 15 分钟数据，约束数据丢失容忍窗口；
2. 再写：RTO = 故障后最多允许中断 2 小时，约束系统恢复服务的时间上限；
3. 最后补：方案设计既要缩小数据损失，也要控制恢复时间。

**纠偏记忆：**
- **RPO 看“能丢多久的数据”，RTO 看“能停多久的服务”。**

**建议复习动作：**
1. 把 RPO / RTO 的定义各默写 3 遍；  
2. 再补 3 道“给定业务条件判断 RPO / RTO”的小题；  
3. 以后写备份恢复题时，先把两个缩写单独写在草稿里确认无误。 

### 第27题（下午题）

**题型归属：** 运维分析 / 告警优先级判断

**题目：**  
某数据库平台在同一小时内出现以下现象：
- 失败登录次数突然激增；
- 慢 SQL 告警明显增多；
- 备份目录所在磁盘空间接近打满；
- 夜间备份任务刚好计划在 2 小时后执行。

请说明：面对这组现象，运维分析时优先应关注哪些风险点，并先从哪些角度下手。

**你的实际作答：**  
- 登录失败次数激增，同时慢SQL告警明显增多。需要检测当前数据库负载是否明显增加，对应数据库服务器CPU，内存等资源使用是否到达瓶颈。  
- 备份目录磁盘空间不足，并且夜间备份任务还未执行。需要有限考虑对备份目录对应磁盘进行扩容，确保后续备份任务可以正常进行。

**参考答案：**  
这组现象中，优先应关注两类高风险问题：第一，失败登录激增可能代表安全事件正在发生；第二，备份目录磁盘即将打满，可能直接影响两小时后的备份任务执行。与此同时，慢 SQL 增多也可能与资源紧张、异常访问、磁盘压力等问题有关。分析时应先保护安全与备份可用性，再结合性能告警判断这些问题是否存在关联，避免单点思维只盯某一个告警。

**本题采分点：**
1. 写出安全事件风险
2. 写出备份任务受磁盘影响的风险
3. 写出慢 SQL 可能与其他问题有关联
4. 能体现“先后优先级”

**本次已命中的采分点：**
- 写到了备份目录磁盘空间不足会影响后续备份任务执行
- 写到了慢 SQL 增多需要结合数据库负载与资源情况分析

**本次漏掉的采分点：**
- 没有把“失败登录次数激增”首先判断为**潜在安全事件**，而是直接转成负载分析
- 没有明确写出应先保护**安全性与备份可用性**，再判断性能问题与资源关联
- “扩容磁盘”属于可能的后续动作，但本题更先考的是**风险识别与优先级顺序**

**失分原因类型：**
- [ ] 不会知识点
- [x] 会概念但不会下手
- [x] 会分析但写不规范
- [ ] 只写结论没写依据
- [x] 漏步骤 / 漏条件 / 漏关键词
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- 多告警并发题，第一步不是马上给技术动作，而是先分辨**哪类风险最急**；
- 本题最急的两层是：可能存在安全异常、备份任务即将受磁盘空间影响；
- 性能问题要放在安全与备份风险之后，结合它们的联动关系再看。

**标准作答路径：**
1. 先写：失败登录激增可能代表异常访问或暴力尝试，属于安全事件风险；
2. 再写：备份目录磁盘接近打满，可能直接影响两小时后的备份任务执行；
3. 接着写：慢 SQL 增多可能与资源紧张、异常访问或磁盘压力有关，需要联动排查；
4. 最后写：应先保护安全与备份可用性，再分析性能告警与资源关联。

**纠偏记忆：**
- **多告警题先排优先级：安全事件 / 备份可用性优先，再看性能关联。**

**建议复习动作：**
1. 按“安全风险 → 备份风险 → 性能关联 → 优先级”顺序重写本题 2 遍；  
2. 再补 2 道“多告警联动分析”小题；  
3. 以后遇到失败登录激增时，第一反应先写：**可能存在安全事件。**

### 第28题（下午题）

**题型归属：** 运维联动 / 监控协同分析

**题目：**  
继续上题，请说明：为什么**安全审计、容量监控、性能监控、备份任务监控**不能彼此割裂，而要联动分析。

**你的实际作答：**  
- 安全审计用于监控数据库敏感操作和操作痕迹，容量监控用于监控服务器磁盘适用于情况，性能监控用于监控服务器性能表现，备份任务监控检测备份任务是否正常执行。

**参考答案：**
