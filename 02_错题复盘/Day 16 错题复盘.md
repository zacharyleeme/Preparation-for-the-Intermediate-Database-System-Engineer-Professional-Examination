## 三、本次练习暴露出的关键易错点

### 1. 知识判断层面的易错点

- 当前对恢复题的大方向判断已经比较稳，但“**恢复对象集合**”这一步还没有在所有模板题里完全固化；
- 对介质故障、系统故障、立即修改、延迟修改的基本区分已经建立，但个别术语表述还会出现细小偏差；
- 对日志、检查点、WAL、备份的作用已经能分开说出，但“这些机制为什么合在一起有效”还需要再写得更标准。

### 2. 下午题书写层面的易错点

- 综合题里仍偶尔会把“事务集合”这一步省掉，直接跳到提交状态和 Undo / Redo；
- 个别题的方向没错，但结尾收束句不够稳，容易少掉最后 1 个采分点；
- 模板题已经开始形成顺序意识，但还需要把“对象 → 动作 → 依据 → 结论”写得更像固定动作，而不是临场拼接。

### 3. 采分点意识层面的易错点

- 当前最大问题已经不是“不会做恢复题”，而是**会做、会写，但有时还没把采分点写满**；
- 像“检查点时仍活跃事务”“检查点后新出现事务”“数据库文件本身不可用”“只保留已提交事务结果的正确状态”这类短语，后续应尽量直接落笔；
- 机制协同题和通用模板题，除了中间过程，**最后一句总结结论**要逐渐写成习惯动作。

## 四、Day 16 需要继续巩固的重点

1. 恢复综合题的固定顺序：**故障类型 → 事务集合 → 提交状态 / 更新策略 → Undo / Redo → 恢复目标**
2. 检查点题的固定口径：**先写哪些事务要纳入恢复分析，再写从哪里开始分析日志**
3. 介质故障题的标准口径：**数据库文件本身不可用 → 先恢复备份 → 再根据日志重做已提交事务 → 未提交结果不保留**
4. 立即修改与延迟修改的标准术语，避免再把“立即修改”写成“立即提交”
5. 恢复机制协同题的标准展开：**WAL 保日志可靠 → 日志给恢复依据 → 检查点缩小范围 → 备份解决介质损坏**
6. 所有综合题的标准收束句：**数据库应恢复到只保留已提交事务结果的正确状态**
7. 模板题里“事务集合”这一步要单独写出，不要并进“提交状态判断”里
