### 第22题（下午题）

**题型归属：** SQL 分组统计

**题目：**  
已知表 `Student(Sno, Sname)`、`SC(Sno, Cno, Grade)`，请写 SQL：查询选课门数不少于 3 门的学生学号及选课门数。

**你的实际作答：**
```sql
select S.Sno,count(SC.Cno)
from Student S
join SC 
on S.Sno=Sc.Sno
group by S.Sno,SC.Cno 
having count(SC.Cno) >= 3;
```

**参考答案：**
```sql
SELECT Sno, COUNT(*) AS CourseCount
FROM SC
GROUP BY Sno
HAVING COUNT(*) >= 3;
```

**本题采分点：**
1. 使用 `GROUP BY Sno`
2. 使用 `COUNT(*)` / `COUNT(Cno)` 统计选课门数
3. 使用 `HAVING COUNT(...) >= 3`

**本次已命中的采分点：**
- 写出了计数统计思路
- 写出了 `HAVING COUNT(...) >= 3`

**本次漏掉的采分点：**
- `GROUP BY` 粒度应是 `Sno`，不应再按 `Cno` 分组

**失分原因类型：**
- [ ] 不会知识点
- [x] 会概念但不会下手
- [ ] 会分析但写不规范
- [ ] 只写结论没写依据
- [x] 漏步骤 / 漏条件 / 漏关键词
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- 题目要统计“每个学生选了几门课”，分组单位必须是“学生”，不是“学生 + 课程”。
- 一旦按 `Sno, Cno` 同时分组，每组通常只对应 1 门课，`COUNT` 就失去题目要求的统计意义。

**标准作答路径：**
1. 先确认统计对象是“学生”；
2. 再确认统计值是“选课门数”；
3. 写 `GROUP BY Sno`；
4. 用 `HAVING COUNT(...) >= 3` 保留满足条件的学生。

**纠偏记忆：**
- **问“每个学生有几门课” → 只按学生分组。**
- **问“每个学生每门课” → 才可能按学生 + 课程分组。**

**建议复习动作：**
1. 重写 3 道 `GROUP BY + HAVING` 统计题；
2. 专门对比“按一列分组”和“按两列分组”的结果差异；
3. 口头复述：题目问谁，就按谁分组。

### 第23题（下午题）

**题型归属：** SQL 反查

**题目：**  
已知表 `Student(Sno, Sname)`、`SC(Sno, Cno, Grade)`，请写 SQL：查询没有任何选课记录的学生学号和姓名。可使用 `NOT EXISTS` 或 `LEFT JOIN ... IS NULL`。

**你的实际作答：**
```sql
select S.Sno, S.Sname
from Student S
left join SC 
on S.Sno = SC.Sno
where SC.Cno is null;
```

**参考答案：**
```sql
SELECT S.Sno, S.Sname
FROM Student S
WHERE NOT EXISTS (
  SELECT 1
  FROM SC
  WHERE SC.Sno = S.Sno
);
```

或：

```sql
SELECT S.Sno, S.Sname
FROM Student S
LEFT JOIN SC ON SC.Sno = S.Sno
WHERE SC.Sno IS NULL;
```

**本题采分点：**
1. 查询主体是学生表
2. 正确表达“不存在选课记录”
3. 连接 / 子查询条件写法稳健、判空位置合理

**本次已命中的采分点：**
- 查询主体是学生表
- 使用了 `LEFT JOIN ... IS NULL` 的反查思路

**本次漏掉的采分点：**
- 更稳健的空值判断应落在连接键 `SC.Sno IS NULL`，而不是 `SC.Cno IS NULL`

**失分原因类型：**
- [ ] 不会知识点
- [ ] 会概念但不会下手
- [x] 会分析但写不规范
- [ ] 只写结论没写依据
- [x] 漏步骤 / 漏条件 / 漏关键词
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- 你已经抓住了“左连接后判空”的核心思路；
- 但核查按更稳健口径，应优先判断连接键是否为空，即 `SC.Sno IS NULL`；
- 这样能更直接表达“该学生在选课表中没有匹配记录”。

**标准作答路径：**
1. 先明确题型是“反查”；
2. 再选 `NOT EXISTS` 或 `LEFT JOIN ... IS NULL`；
3. 若用左连接判空，优先判断连接键是否为空；
4. 输出学生学号与姓名。

**纠偏记忆：**
- **左连接反查时，优先判“连接键空”，不要随手判别的列空。**

**建议复习动作：**
1. 把“有 / 没有 / 不存在”三类 SQL 各写 2 道；
2. 统一练 `NOT EXISTS` 与 `LEFT JOIN ... IS NULL` 双写法；
3. 以后看到反查题，先问自己：我判空的是不是连接键？

### 第27题（下午题）

**题型归属：** 抢红包 2PL

**题目：**  
红包记录 `X` 初始为空。第一个响应者应最终抢到红包。请使用 `XLock`、`R`、`W`、`Commit`、`Unlock` 写出一种满足 2PL 的处理思路，并说明后续响应者如何处理。

**你的实际作答：**
- `XLocki(X) → Ri(X) → 如果 X 为空，则 W(1,X) → commit → ULocki(X)`
- `XLocki(X) → Ri(X) → 如果 X 非空，则 rollback → ULocki(X)`

**参考答案：**
```text
XLock1(X)
a1 = R1(X)
若 X 为空，则 W1(b1, X)
Commit1
Unlock1(X)

XLock2(X)
a2 = R2(X)
发现 X 已非空，不再写入
Commit2 / Rollback2
Unlock2(X)
```

**本题采分点：**
1. 用排他锁保护“读 → 判断 → 写”
2. 第一个响应者先完成写入
3. 后续响应者必须等待前一个事务释放锁后再读取
4. 整体流程满足 2PL

**本次已命中的采分点：**
- 使用排他锁保护关键区
- 写出了“为空则写入，不为空则回滚 / 不写”的核心分支
- 整体大框架符合 2PL 思路

**本次漏掉的采分点：**
- 没有把“后续响应者等待前一事务释放锁后再进入”明确写出来

**失分原因类型：**
- [ ] 不会知识点
- [x] 会概念但不会下手
- [ ] 会分析但写不规范
- [x] 只写结论没写依据
- [x] 漏步骤 / 漏条件 / 漏关键词
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- 抢红包题真正的保护点不只是“加锁”，而是要把**后续事务等待前一个事务释放锁**这个并发约束写出来；
- 如果不把“等待后再读”明确写出，答案就少了最关键的并发控制含义。

**标准作答路径：**
1. 第一个响应者先 `XLock(X)`；
2. `R(X)` 后判断是否为空；
3. 若为空则 `W(X)` 并 `Commit`；
4. `Unlock(X)`；
5. 后续响应者等待锁释放后再 `XLock(X)`、再 `R(X)`；
6. 发现已非空，则不写入，提交或回滚后释放锁。

**纠偏记忆：**
- **抢红包题不是只写“谁写了”，而是必须写“后者先等，再读，再放弃 / 回滚”。**

**建议复习动作：**
1. 重写 2 道“抢红包 / 首次写入者获胜”题；
2. 每次都强制写出“等待”这一步；
3. 把 2PL 流程压缩成一句口令：**先锁住，后判断；先提交，后解锁；后者等待，再进入。**

## 三、本次练习暴露出的关键易错点

### 1. 知识判断层面的易错点

- `GROUP BY` 的分组粒度仍然容易跟着表结构走，而不是跟着题目统计对象走。
- 反查 SQL 已经知道要用 `LEFT JOIN ... IS NULL` 或 `NOT EXISTS`，但对“判哪个字段为空最稳健”还不够敏感。
- 2PL 的核心概念已掌握，但一到写步骤题，容易把“等待关系”省略掉。

### 2. 下午题书写层面的易错点

- 会写核心思路，但不总是把**步骤链条**写完整。
- SQL 题容易出现“方向对、细节丢”的情况：结构对了，分组列或判空列写偏。
- 并发题容易把“锁 → 读写 → 提交 → 解锁”的框架写出来，但没有把“后续事务何时进入”交代清楚。

### 3. 采分点意识层面的易错点

- 你现在最大的风险不是“不会”，而是**已经知道套路，但还没把评分点写满**。
- SQL 题的评分点常落在：分组对象、筛选位置、判空位置。
- 并发题的评分点常落在：等待、锁顺序、提交后解锁。
