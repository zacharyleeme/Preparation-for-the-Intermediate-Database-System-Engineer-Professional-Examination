## 三、本次练习暴露出的关键易错点

### 1. 知识判断层面的易错点

- 已经能稳定识别“**多次发生 + 需要保留历史 + 需要统计**”通常意味着要独立建模，但一到具体字段落位时，仍可能把 **唯一标识** 写偏；
- 对“**支付记录**”“**订单明细**”“**收货地址**”这些记录型 / 明细型关系模式，已经能抓住大方向，但还没有完全压实到“字段精确、主外键稳、边界收准”的层级；
- 对扩展需求题，已知道不能只加标志字段，但还需要继续强化：**新需求引入的是一条新的业务链条，而不是一个孤立字段**。

### 2. 下午题书写层面的易错点

- 当前最主要的问题，不再是“完全不会”，而是 **方向对了，但还差半步写满**；
- 模板题的句序比 Day 21 明显更稳，但仍会出现：
  - 关系模式名写出来了，**唯一标识没完全收准**；
  - 知道要拆“订单头 / 明细”，但 **没有把核心字段和关联方向写到最稳**；
  - 结尾会写“长期查询统计和统一管理”，但对 **原设计为什么不够、扩展需求引入了什么新链条** 还不够展开；
- 一旦题目涉及“支付记录 / 收货地址 / 订单明细 / 扩展需求统一管理”这类结构化表达场景，当前答案还会在 **字段取舍与关系边界** 上出现轻微漂移。

### 3. 采分点意识层面的易错点

- 只要题干出现“**详细金额、支付时间、支付方式、付款方、统计情况**”，就要主动想到：
  - 不只是金额字段；
  - 很可能需要独立支付记录模式；
  - 唯一标识应优先落到 **支付记录本身**；
- 只要题干出现“**一份订单可包含多个商品**”，就要优先联想到：
  - 订单整体信息与商品项信息分层；
  - 数量、单价等属于 **明细层**；
  - 不能只按订单编号唯一把所有内容压回单表；
- 只要题干出现“**统一管理**”“**扩展需求**”，就要补写：
  - 哪些对象 / 联系会受影响；
  - 为什么只加一个标志字段不够；
  - 新结构如何支持后续查询、统计与统一管理。

## 四、Day 22 需要继续巩固的重点

1. 再练“**支付记录的唯一标识优先落到支付编号 / 支付记录编号，而不是直接拿订单编号顶上去**”这条规则；
2. 再练“**订单头 / 订单明细分离**”的标准表达，特别是把 **订单编号、商品编号、数量、单价** 这类核心字段写稳；
3. 再练“**收货地址模式必须能看出它归属于哪个顾客**”，避免只写地址文本本身；
4. 再练“**原设计不足 → 新模式设计 → 统计 / 查询 / 统一管理价值**”这一组标准收束句；
5. 对支付记录类题目，继续固定写法：**支付编号主键 + 快递编号外键 + 金额 / 时间 / 方式 / 付款方**；
6. 对订单与商品类题目，继续固定写法：**订单头信息与订单明细信息分离，数量等业务发生信息放在明细 / 联系模式中**；
7. 对扩展需求题，继续固定写法：**先写受影响对象 / 联系，再写为什么不能只加标志字段，最后写统一管理链条**；
8. 虽然本轮没有正式错题，但后续仍要把“**采分点命中率提升**”作为比“是否判错”更重要的硬指标。
