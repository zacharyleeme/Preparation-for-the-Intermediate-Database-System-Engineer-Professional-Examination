### B. 下午题 / 主观题错题

### 第22题（下午题）

**题型归属：** SQL 反查

**题目：**  
已知表 `Student(Sno, Sname)`、`SC(Sno, Cno, Grade)`，请分别用 `NOT EXISTS` 和 `LEFT JOIN ... IS NULL` 写出 SQL：查询没有任何选课记录的学生学号和姓名。

**你的实际作答：**  
```sql
-- NOT EXISTS 写法
select S.Sno,S.Sname
from Student
where not exists(select 1 from SC where S.Sno = SC.Sno);

-- LEFT JOIN 写法
select S.Sno,count(SC.Cno)
from Student S
left join SC 
on S.Sno = SC.Sno
where SC.sno is null;
```

**参考答案：**  
```sql
-- NOT EXISTS 写法
SELECT S.Sno, S.Sname
FROM Student S
WHERE NOT EXISTS (
  SELECT 1
  FROM SC
  WHERE SC.Sno = S.Sno
);

-- LEFT JOIN 写法
SELECT S.Sno, S.Sname
FROM Student S
LEFT JOIN SC ON SC.Sno = S.Sno
WHERE SC.Sno IS NULL;
```

**本题采分点：**
1. 查询主体是 `Student`
2. 正确表达“不存在选课记录”
3. `LEFT JOIN` 写法优先判断连接键 `SC.Sno IS NULL`

**本次已命中的采分点：**
- 使用了 `Student` 作为查询主体
- 识别出了 `NOT EXISTS` 与 `LEFT JOIN ... IS NULL` 两种反查思路
- 左连接写法中使用了连接键 `SC.Sno IS NULL` 的判空方向

**本次漏掉的采分点：**
- `NOT EXISTS` 写法中 `Student` 缺少别名 `S`，实际语句不完整
- `LEFT JOIN` 写法输出字段错误，题目要求“学号和姓名”，你写成了 `学号 + count(SC.Cno)`
- 左连接写法没有把 `S.Sname` 一起输出

**失分原因类型：**
- [ ] 不会知识点
- [ ] 会概念但不会下手
- [x] 会分析但写不规范
- [ ] 只写结论没写依据
- [x] 漏步骤 / 漏条件 / 漏关键词
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- 你已经抓住了“反查”的主思路；
- 当前问题不是不会，而是**落笔不稳**：别名、输出列、结果粒度没完全对齐题目要求。

**标准作答路径：**
1. 先确定查询主体为 `Student`；
2. 若用 `NOT EXISTS`，给主体表加别名并写关联条件；
3. 若用 `LEFT JOIN`，输出 `Sno, Sname`，并判断 `SC.Sno IS NULL`；
4. 最后检查输出字段是否完全匹配题意。

**纠偏记忆：**
- **反查题：主体表先保留，连接键判空，输出列按题目逐项核对。**

**建议复习动作：**
1. 重写 2 道 `NOT EXISTS` 反查题；
2. 重写 2 道 `LEFT JOIN ... IS NULL` 反查题；
3. 每次写完强制检查：**主体表别名、输出字段、判空字段** 三项。

### 第25题（下午题）

**题型归属：** 并发控制 / 冲突可串行化

**题目：**  
给定调度：`R1(A), W2(A), W1(B), R2(B)`，请回答：  
1. 哪些操作之间存在冲突？  
2. 冲突图中有哪些边？  
3. 该调度是否冲突可串行化？为什么？

**你的实际作答：**  
- `R1(A)` 与 `W2(A)` 存在冲突，方向为 `T1→T2`；`W1(B)` 与 `R2(B)` 存在冲突，方向为 `T2→T1`  
- 冲突存在边 `T1→T2，T2→T1`  
- 因为调度存在环，所以不是冲突可串行化

**参考答案：**  
1. `R1(A)` 与 `W2(A)` 冲突，方向为 `T1 → T2`；`W1(B)` 与 `R2(B)` 冲突，方向也为 `T1 → T2`。  
2. 冲突图中只有边 `T1 → T2`。  
3. 冲突图无环，因此该调度冲突可串行化，等价于串行顺序 `T1 → T2`。

**本题采分点：**
1. 按“不同事务、同一数据项、至少一个写”找冲突
2. 正确判断边方向
3. 用“有环 / 无环”给出结论

**本次已命中的采分点：**
- 找出了两对真正的冲突操作

**本次漏掉的采分点：**
- `W1(B)` 在 `R2(B)` 之前发生，因此方向应为 `T1 → T2`，不是 `T2 → T1`
- 冲突图只有一条方向边，不构成环
- 最终结论应为“冲突可串行化”，不是“不可串行化”

**失分原因类型：**
- [ ] 不会知识点
- [x] 会概念但不会下手
- [ ] 会分析但写不规范
- [ ] 只写结论没写依据
- [x] 漏步骤 / 漏条件 / 漏关键词
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- 冲突图边方向由**操作先后顺序**决定，不是由读 / 写类型随意判断；
- 这道题的两组冲突都指向同一个方向，所以图无环。

**标准作答路径：**
1. 先找冲突对；
2. 再看谁先发生，确定边方向；
3. 画出冲突图；
4. 最后用“有环 / 无环”判断是否冲突可串行化。

**纠偏记忆：**
- **先找冲突，再看先后，最后看有无环。**

**建议复习动作：**
1. 再做 3 道冲突图方向判断题；
2. 每题都把“前者 → 后者”明确写出来；
3. 强制口头复述：**边方向由先后顺序定。**

### 第26题（下午题）

**题型归属：** 2PL / 死锁规避

**题目：**  
事务 `T1` 与 `T2` 都需要访问数据项 `A`、`B`。请说明为什么两个事务都按 `A → B` 的顺序申请锁有助于降低死锁风险，并写出一个满足 2PL 的简短执行序列。

**你的实际作答：**  
- 因为按顺序对 A，B 申请锁，可以使得加锁和解锁分别在不同阶段进行，事务一次只对一个苏韩剧下  
- `XLock1(A) → XLock2(A) → R1(A) → R2(A) → commit → UnLock1(A) → UnLock2(A)`  
- `等待之前锁释放 → XLock1(B) → XLock2(B) → R1(B) → R2(B) → commit → UnLock1(B) → UnLock2(B)`

**参考答案：**  
- 两个事务都按 `A → B` 顺序申请锁，可以避免 `T1` 持有 `A` 等 `B`、`T2` 持有 `B` 等 `A` 的循环等待，从而降低死锁风险。  
- 示例序列：  
```text
T1: Lock(A) → Lock(B) → R/W(A) → R/W(B) → Commit → Unlock(B) → Unlock(A)
T2: 等待 T1 释放冲突锁后 → Lock(A) → Lock(B) → R/W(A) → R/W(B) → Commit → Unlock(B) → Unlock(A)
```

**本题采分点：**
1. 说明统一加锁顺序破坏 / 减少循环等待
2. 序列满足先加锁后解锁
3. 一旦解锁，不再申请新锁

**本次已命中的采分点：**
- 抓住了“按 A → B 顺序申请锁”这一核心方向

**本次漏掉的采分点：**
- 没有准确说出“降低死锁风险”的关键原因是**减少 / 破坏循环等待**
- 你给出的执行序列中，多事务对同一锁的申请与解锁顺序混乱
- 序列实际不满足标准 2PL，出现了不清晰甚至错误的加锁 / 解锁安排

**失分原因类型：**
- [ ] 不会知识点
- [x] 会概念但不会下手
- [ ] 会分析但写不规范
- [x] 只写结论没写依据
- [x] 漏步骤 / 漏条件 / 漏关键词
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- 统一加锁顺序的本质，不是“看起来整齐”，而是避免事务互相交叉等待形成环；
- 2PL 序列必须清楚体现：**先把需要的锁拿够，再进行读写，提交后释放锁，后一事务等待后再进入**。

**标准作答路径：**
1. 先说明统一顺序可减少循环等待；
2. 写 `T1`：先 `Lock(A) → Lock(B)`，再读写，提交后解锁；
3. 写 `T2`：等待 `T1` 释放冲突锁后再按同顺序进入；
4. 保证一旦开始解锁，不再申请新锁。

**纠偏记忆：**
- **2PL 序列最怕“先解锁再继续加锁”。**

**建议复习动作：**
1. 重写 2 道“统一加锁顺序 + 2PL”题；
2. 每次都先写一句：**统一顺序是为了防循环等待**；
3. 只写“先锁完 → 再读写 → 提交 → 解锁 → 后者等待进入”的干净序列。

### 第27题（下午题）

**题型归属：** 日志恢复 / Redo 与 Undo

**题目：**  
某系统故障时事务状态如下：  
- `T1`：检查点后开始，写过 `A`，故障前已提交  
- `T2`：检查点后开始，写过 `B`，故障前未提交  
- `T3`：检查点前已提交，检查点后无活动  
- `T4`：检查点后开始，未写数据，故障前未提交  
请判断各事务应 `Redo`、`Undo` 还是通常不处理，并说明理由。

**你的实际作答：**  
- `T1` 应 redo，因为检查点后写过 A，并且故障前事务已提交  
- `T2` 应 undo，因为检查点后写过 B，并且故障前事务未提交  
- `T3` 通常不处理，因为检查点后，未写数据  
- `T4` 通常不处理，因为检查点前已提交，并且检查点后无活动

**参考答案：**  
- `T1`：Redo。已提交且写过 `A`。  
- `T2`：Undo。未提交但写过 `B`。  
- `T3`：通常不处理。检查点前已提交，检查点后无活动。  
- `T4`：通常不处理数据项更新。检查点后开始，未写数据，未提交。

**本题采分点：**
1. 已提交写事务 Redo
2. 未提交写事务 Undo
3. 检查点前稳定完成且无后续活动通常不处理
4. 未写数据事务通常不处理数据项更新

**本次已命中的采分点：**
- `T1` 的恢复动作与理由正确
- `T2` 的恢复动作与理由正确

**本次漏掉的采分点：**
- `T3` 与 `T4` 的“通常不处理”结论虽对，但理由写反了
- 没有把“检查点前稳定完成”和“未写数据”这两个不处理场景准确对应到具体事务

**失分原因类型：**
- [ ] 不会知识点
- [x] 会概念但不会下手
- [x] 会分析但写不规范
- [ ] 只写结论没写依据
- [x] 漏步骤 / 漏条件 / 漏关键词
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- 恢复题不是只背 `Redo / Undo` 两个词，还要把**事务状态—写数据与否—提交状态—检查点位置**逐项对上；
- `T3` 与 `T4` 这类“都不处理”的题，最容易在理由配对上出错。

**标准作答路径：**
1. 先列出事务是否在检查点后活跃；
2. 再看是否写过数据；
3. 再看故障前是否提交；
4. 最后对应到 `Redo / Undo / 通常不处理`。

**纠偏记忆：**
- **恢复题先分状态，再下结论；别把理由写串位。**

**建议复习动作：**
1. 再做 2 道“事务状态表 → 恢复动作”题；
2. 每次都先画三列：**是否写过 / 是否提交 / 检查点后是否活跃**；
3. 把“检查点前稳定完成”和“未写数据”分别各举 1 个例子口头复述。

## 三、本次练习暴露出的关键易错点

### 1. 知识判断层面的易错点

- 计算机组成基础概念混淆仍明显：**寄存器 vs Cache、地址总线 vs 数据总线、海明码 vs 奇偶校验**。
- 数据结构基础结论不稳：**二叉排序树中序遍历有序、堆排序最坏复杂度 `O(nlogn)`** 没压牢。
- 操作系统基础题存在场景错配：**时间片轮转适用分时系统、分页地址由页号和页内地址组成** 还会混。

### 2. 下午题书写层面的易错点

- SQL 题已经能抓到大方向，但**字段完整性、别名、结果粒度** 还不够稳。
- 并发题常见问题不是“不会”，而是**边方向、等待关系、标准步骤链** 写不完整。
- 恢复题能给出大结论，但**理由与事务状态的对应关系** 容易串位。

### 3. 采分点意识层面的易错点

- 目前最大的风险是：**知道模板，但没有把评分点写满**。
- SQL 反查题除了“会用 `NOT EXISTS` / `LEFT JOIN`”，还要保证 **别名、输出列、判空列** 都对齐。
- 2PL 题除了“统一加锁顺序”，还必须明确写出 **后者等待、先加锁后解锁、解锁后不再加锁**。
- 恢复题除了 `Redo / Undo` 结论，还要把 **检查点、是否写过数据、是否提交** 一一对应。
