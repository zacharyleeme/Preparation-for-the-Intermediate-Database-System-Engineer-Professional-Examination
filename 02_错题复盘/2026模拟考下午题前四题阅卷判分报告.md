## 二、第1题判分：5 / 15

### 题目结构

- 问题1：4 分，创建 `购票` 表，补主键、检查约束、两个外键。
- 问题2：6 分，插入购票信息 + 触发器更新 VIP 折扣。
- 问题3：5 分，分组统计查询 + 一次中转航班查询。

### 问题1：2 / 4

| 空 | 标准答案要点 | 实际作答 | 判分 | 说明 |
|---|---|---|---:|---|
| a | `PRIMARY KEY` 或 `NOT NULL UNIQUE` | `primary key` | 1 | 正确 |
| b | `CHECK(购票金额 > 0)` | `(8)` | 0 | 未表达检查约束 |
| c | `FOREIGN KEY (身份证号) REFERENCES 旅客(身份证号)` | `formary key on 旅客(身份证号)` | 0.5 | 有外键意图，但 SQL 关键字和参照写法错误 |
| d | `FOREIGN KEY (航班编号) REFERENCES 航班(航班编号)` | `formary key on 航班(航班编号)` | 0.5 | 有外键意图，但 SQL 关键字和参照写法错误 |

**小计：2 / 4。**

### 问题2：1 / 6

| 空 | 标准答案要点 | 实际作答 | 判分 | 说明 |
|---|---|---|---:|---|
| e | `票价 * 折扣 * VIP折扣` | `'航班.票价 × 折扣.折扣 × 旅客.VIP折扣' as 购票金额` | 0.5 | 公式方向对，但写成字符串/表达不规范 |
| f | `航班.航班编号 = 折扣.航班编号` | `购票.航班编号` | 0 | 连接条件错误 |
| g | `INSERT` | `update` | 0 | 触发时机错误，应为插入购票后触发 |
| h | `购票` | `旅客` | 0 | 触发表错误 |
| i | `VIP折扣 = vip_value(nrow.身份证号)` | `旅客.VIP折扣 = float vip_value(char 身份证号)` | 0.5 | 知道要更新 VIP 折扣并调用函数，但语法和参数错误 |
| j | `旅客.身份证号 = nrow.身份证号` | `旅客.身份证号='xxxx'` | 0 | 未关联新购票行 |

**小计：1 / 6。**

### 问题3：2 / 5

| 空 | 标准答案要点 | 实际作答 | 判分 | 说明 |
|---|---|---|---:|---|
| k | `旅客.身份证号 = 购票.身份证号 AND 搭乘日期 BETWEEN ...` | 日期条件 + `SUM(购票金额)>10000` | 0.5 | 日期筛选有方向，但缺连接条件，且聚合条件误放 WHERE |
| l | `旅客.身份证号, 姓名 HAVING SUM(购票金额)>=10000` | `旅客.身份证号` | 0.25 | 分组字段不完整，缺 HAVING |
| m | `SUM(购票金额) DESC` | `SUM(购票金额) desc` | 1 | 正确 |
| n | `航班1.航班编号, 航班1.目的地, 航班2.航班编号` | `航班编号_广州,航班编号_中转,航班编号_北京` | 0.25 | 有输出航班信息意图，但字段与别名不合格 |
| o | 中转条件：广州出发、北京到达、前段目的地=后段起飞地 | 空 | 0 | 未作答 |

**小计：2 / 5。**

### 第1题结论

**得分：5 / 15。**  
主要失分点是 SQL 语法与采分点不成形：外键要写 `FOREIGN KEY ... REFERENCES ...`，触发器要落在 `购票` 的 `INSERT` 后，分组统计的聚合筛选要用 `HAVING`，中转查询必须写出两个航班别名间的连接条件。

## 四、第3题判分：7 / 15

### 题目结构

- 问题1：3 分，库存更新失败时的事务处理。
- 问题2：6 分，并发执行导致的问题与 2PL 加锁序列。
- 问题3：6 分，E-SQL 补空。

### 问题1：0.5 / 3

标准答案：将“写订单记录”和“修改库存表”作为一个完整事务；若修改库存表失败，应 `ROLLBACK`，撤销已写入的订单记录，保持数据库一致性。

实际作答：取消库存表修改、查订单、反馈库存不足、采购补库存。

判分：**0.5 / 3**。  
说明：有“处理失败”的意识，但没有抓住事务原子性与回滚订单记录这个核心采分点。

### 问题2：0.5 / 6

| 项 | 标准答案要点 | 实际作答 | 判分 | 说明 |
|---|---|---|---:|---|
| 问题类型 | 丢失修改，客户1库存更新被客户2覆盖 | 判断为死锁 | 0 / 3 | 问题类型错误 |
| 2PL 序列 | 对库存 A 在读写前加锁，客户1释放后客户2再加锁；满足 2PL 且持锁时间短 | 只写了“等待事务1释放锁”框架，指令基本空缺 | 0.5 / 3 | 有等待意识，但未形成有效序列 |

**小计：0.5 / 6。**

### 问题3：6 / 6

### 第3题结论

## 五、第4题判分：12 / 15

### 题目结构

- 问题1：5 分，介质故障后的完整恢复步骤。
- 问题2：4 分，redo / undo 事务列表。
- 问题3：6 分，恢复后 X、Y、Z 的值。

### 问题1：4 / 5

标准答案要点：
1. 重新安装原系统 DBMS；
2. 用完整备份 `Backup_original` 恢复到完整备份状态；
3. 依次应用增量备份文件；
4. 利用系统日志 `syslog` 恢复到故障前状态。

实际作答：指出介质故障，先用 `Backup_original` 全量恢复，再用增量备份恢复，再分析 `syslog` 对已提交事务 redo、未提交事务 undo。

判分：**4 / 5**。  
说明：总体方向正确，缺“重新安装 DBMS / 原系统环境”这一标准步骤；日志处理表述略泛，但可得大部分分。

### 问题2：4 / 4

标准答案：
- redo：`T1、T3`
- undo：`T2`

实际作答完全一致。

**小计：4 / 4。**

### 问题3：4 / 6

标准答案：
- `X = 1`
- `Y = 8`
- `Z = 5`

实际作答：
- `X = 1`
- `Y = 50`
- `Z = 5`

判分：
- X：2 / 2
- Y：0 / 2
- Z：2 / 2

**小计：4 / 6。**

Y 值错因：
- `T2` 未提交，其对 Y 的修改要 undo；
- 但 `T3` 已提交，并且把 Y 改为 8，所以恢复后 Y 应保留 T3 的提交结果 `8`，不是回到初始值 `50`。

### 第4题结论

**得分：12 / 15。**  
这是前四题中表现最好的一题。恢复流程、redo / undo 队列基本清楚；唯一明显失分点是恢复数据值时，没有区分“撤销 T2”之后仍要保留已提交事务 T3 对同一数据项的影响。

## 六、阅卷人角度的总评

### 1. 当前优势

- 第4题恢复主线掌握较好，redo / undo 队列判断正确。
- 第3题 E-SQL 补空能抓住 `return`、`商品ID=:MID`、`COMMIT` 三个关键点。
- 第2题候选人关系的部分依赖与 1NF / 2NF 判断方向正确。

### 2. 主要失分模式

1. **SQL 语法采分点不稳定**
   - 外键约束写法不完整；
   - `CHECK` 约束未写出；
   - 连接条件缺失；
   - 聚合条件误放在 `WHERE`；
   - 触发器事件和触发表判断错误。

2. **规范化题只写结论，不写完整分解**
   - 面试官关系未给出分解后的关系模式；
   - 招聘安排 4NF 完全缺答；
   - 主键没有列出。

3. **并发控制题误判问题类型**
   - 题目给出的交错执行是“丢失修改”，不是死锁；
   - 2PL 序列必须写出加锁、读、写、解锁，不能只写“等待”。

4. **恢复题数据值推导仍需按日志顺序细算**
   - undo 未提交事务后，仍要保留已提交事务的修改。

### 3. 后续纠偏建议

1. 重写第1题 SQL 空：主键、外键、`CHECK`、触发器、`GROUP BY/HAVING`、自连接中转查询。
2. 重写第2题规范化：候选人、面试官、招聘安排三组关系都按“候选键—依赖—范式—分解—主键”模板写。
3. 重写第3题并发：把“丢失修改”和“死锁”分开；强制写一遍满足 2PL 的库存更新序列。
4. 重写第4题日志值推导：按日志顺序列出 T1/T2/T3 对 X/Y/Z 的影响，再根据 redo/undo 得最终值。

---

## 七、一句话判分结论

前四题合计 **31 / 60**：第4题恢复类表现较稳，第2题候选人范式判断有基础，但第1题 SQL 综合与第3题事务并发是主要扣分区；下一轮应优先补 **SQL 采分点、规范化分解、丢失修改/2PL、日志值推导** 四类题。
