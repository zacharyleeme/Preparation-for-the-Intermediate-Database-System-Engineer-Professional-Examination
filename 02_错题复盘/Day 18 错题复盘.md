### B. 下午题 / 主观题正式错题

### 第24题（下午题）

**题型归属：** SQL 改写

**题目：**  
继续上题，请把这条 SQL 改写成更利于索引利用的形式，并简要说明理由。

**你的实际作答：**  
- select dept_id,hire_date from Employee where dept_id =? and hire_date between '2024-01-01' AND '2024-12-31' ORDER BY hire_date DESC;  
- select * 会返回所有表字段，增加回表开销，相比之下只返回需要查询的列，更有利于优化查询  
- 条件 YEAR(hire_date) 无法走常规索引，不利于优化器扫描,改写为 between and 语法更适合索引扫描

**参考答案：**  
可改写为：

```sql
SELECT emp_id, emp_name, hire_date
FROM Employee
WHERE hire_date BETWEEN '2025-01-01' AND '2025-12-31'
  AND dept_id = ?
ORDER BY hire_date DESC;
```

这样做把函数条件改成了日期范围，更利于数据库利用 `hire_date` 相关索引；同时避免 `SELECT *`，也减少了不必要的返回列成本。

**本题采分点：**
1. 写出日期范围改写思路
2. 减少或避免 `SELECT *`
3. 能说明为什么更利于索引利用

**本次已命中的采分点：**
- 写出了把 `YEAR(hire_date)` 改成 `BETWEEN ... AND ...` 的范围条件思路
- 写出了避免 `SELECT *`、减少返回列与回表成本的方向
- 写出了“更利于索引扫描 / 索引利用”的原因

**本次漏掉的采分点：**
- 题干原本要求筛选 `2025` 年数据，你在改写 SQL 时写成了 `2024-01-01 ~ 2024-12-31`，与原题条件不一致
- 返回列虽然改成了精简列，但仍未完全对齐参考答案里的业务列表达
- 这类题不只是“方向对”，还要确保改写后的条件与原题语义严格一致

**失分原因类型：**
- [ ] 不会知识点
- [ ] 会概念但不会下手
- [x] 会分析但写不规范
- [ ] 只写结论没写依据
- [x] 漏步骤 / 漏条件 / 漏关键词
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- SQL 改写题首先要保证**语义不变**，不能为了改写索引友好性，把原题的年份条件改错；
- 在保证语义一致的前提下，再把函数条件改成范围条件，并顺手减少不必要的返回列；
- 所以这题真正的关键不是“会不会写 `BETWEEN`”，而是“改写后是否仍然是同一条查询的等价表达”。

**标准作答路径：**
1. 先把 `YEAR(hire_date) = 2025` 改写为 `hire_date BETWEEN '2025-01-01' AND '2025-12-31'`；
2. 再把 `SELECT *` 改成业务实际需要的列；
3. 最后补一句：这样既更利于索引利用，也减少了不必要的返回列成本。

**纠偏记忆：**
- **SQL 改写题先守住原语义，再谈索引友好性；年份、范围边界这类事实条件不能写错。**

**建议复习动作：**
1. 把本题按 `2025` 年口径重写 2 遍；  
2. 再补 2 道“函数条件改范围条件”的 SQL 改写题；  
3. 以后写改写题时，先口头确认一次：**改写前后查的是不是同一批数据。**

### 第28题（下午题）

**题型归属：** 索引收益与代价比较

**题目：**  
继续上题，请用 3～5 句说明：为什么日志表的索引设计通常要比普通查询表更克制，并补出“存储开销”这一层。

**你的实际作答：**  
- 日志表的增删改操作一般高于普通查询表  
- 频繁的增删改操作会增加表索引维护代价  
- 所以日志表的索引创建应该仅覆盖核心查询场景，而不是在所有查询列上都创建索引

**参考答案：**  
日志表每天新增量大，写入压力通常比普通查询表更高，因此索引数量不能过多。索引虽然能提升查询性能，但也会增加插入维护成本与存储开销。对于日志表，更稳妥的思路是优先保障最核心的查询路径，而不是把所有可能查询列都建立索引。

**本题采分点：**
1. 写出写入频繁
2. 写出维护成本与存储开销
3. 写出围绕核心路径取舍

**本次已命中的采分点：**
- 写出了日志表增删改更频繁、写入压力更大的特点
- 写出了频繁写入会带来索引维护代价上升
- 写出了索引应优先覆盖核心查询场景，而不是所有列平均建索引

**本次漏掉的采分点：**
- 题目明确要求补出“存储开销”这一层，但你的答案只写到了维护成本，没有把额外索引占用存储空间这一代价补出来
- 这一题考的就是“收益 + 代价”双层表达，漏掉存储开销后，答案会显得还不够完整

**失分原因类型：**
- [ ] 不会知识点
- [ ] 会概念但不会下手
- [x] 会分析但写不规范
- [ ] 只写结论没写依据
- [x] 漏步骤 / 漏条件 / 漏关键词
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- 高写入表的“克制”不是抽象态度，而是要明确写出：**索引会增加维护成本，也会增加存储开销**；
- 只有把“写入压力 + 维护代价 + 存储代价 + 核心路径取舍”都写出来，答案才完整；
- 这类题以后只要看到“为什么更克制”“补出代价这一层”，就要主动把存储空间一起落笔。

**标准作答路径：**
1. 先写：日志表写入频繁，索引数量不能过多；
2. 再写：索引虽然能提升查询性能，但会增加维护成本和存储开销；
3. 最后写：因此应围绕最核心的查询路径做取舍，而不是给所有列平均建索引。

**纠偏记忆：**
- **高写入表的索引代价至少写两层：维护成本 + 存储开销。**

**建议复习动作：**
1. 按“写入压力 → 维护成本 + 存储开销 → 核心路径取舍”的顺序重写本题 2 遍；  
2. 再补 2 道“高写入表 / 日志表 / 流水表”物理设计取舍题；  
3. 以后写这类题时，强制自己检查一遍：**代价除了维护成本，还写没写存储开销。**

### 第30题（下午题）

**题型归属：** 通用模板复写 / 性能优化题标准句序

**题目：**  
请用“先找瓶颈 → 再定索引 / SQL / 物理设计手段 → 再写为什么有效 → 最后写代价与结论”的顺序，写一段 4～6 句的通用性能优化题标准答案。

**你的实际作答：**  
- 明确当前SQL中的核心筛选条件  
- 确认需要创建索引的条件以及索引中条件的顺序  
- 创建索引可以减少扫描范围，减少回表开销  
- 索引同时也会增加表维护代价，需要综合考虑索引维护成本

**参考答案：**  
回答性能优化题时，首先应识别查询瓶颈位于筛选、连接、排序、分组还是返回列过多。其次根据瓶颈选择合适的索引、SQL 改写或物理设计方案。然后说明这些手段如何减少扫描范围、回表代价或排序开销。最后补充代价与结论：该方案能提升核心查询性能，但会带来一定的索引维护和存储成本，需要结合业务频率综合权衡。

**本题采分点：**
1. 顺序完整：瓶颈 → 手段 → 原因 → 代价 / 结论
2. 写出至少两类优化手段
3. 结尾有权衡句

**本次已命中的采分点：**
- 基本具备了“先看条件，再谈索引效果，最后补代价”的模板意识
- 写到了索引可以减少扫描范围、减少回表开销

**本次漏掉的采分点：**
- 开头只写了“核心筛选条件”，还没有把瓶颈标准入口扩展到筛选 / 连接 / 排序 / 分组 / 返回列这些更完整的判断框架
- 手段层基本只写了索引，没有把 SQL 改写、物理设计等可选手段并列展开
- 结尾虽然写了维护成本，但还缺少“该方案能提升核心查询性能，需要结合业务频率综合权衡”的收束句

**失分原因类型：**
- [ ] 不会知识点
- [x] 会概念但不会下手
- [x] 会分析但写不规范
- [ ] 只写结论没写依据
- [x] 漏步骤 / 漏条件 / 漏关键词
- [ ] 时间不足 / 表达仓促

**正确理解应为：**
- 通用模板题的关键，不是写出某一种具体手段，而是把**判断入口、手段类型、作用原因、代价结论**写成稳定句序；
- 这类题如果只写索引，就会显得答案层次偏窄，不够“通用”；
- 真正稳妥的模板，要能兼容索引、SQL 改写、物理设计等多种手段，并在最后补上权衡句。

**标准作答路径：**
1. 先写：当前要先识别瓶颈位于筛选、连接、排序、分组还是返回列过多；
2. 再写：根据瓶颈选择索引、SQL 改写或物理设计等手段；
3. 接着写：这些手段如何减少扫描范围、回表代价或排序开销；
4. 最后写：方案能提升核心查询性能，但会带来维护 / 存储代价，需要结合业务频率综合权衡。

**纠偏记忆：**
- **通用性能优化模板题至少要写两类手段，并用一句“收益 + 代价 + 权衡”把答案收住。**

**建议复习动作：**
1. 按参考答案的四步顺序重写本题 3 遍；  
2. 单独口头复述一遍“筛选 / 连接 / 排序 / 分组 / 返回列”这 5 个瓶颈入口；  
3. 以后写模板题时，强制检查：**有没有写到索引之外的第二类手段、最后有没有收束句。**

## 三、本次练习暴露出的关键易错点

### 1. 知识判断层面的易错点

- 当前对慢 SQL 分析、联合索引方向、覆盖索引意识、隐式转换风险等基础判断已经比较稳；
- 但一进入“SQL 改写”“高写入表代价比较”“通用模板复写”这类题，仍会出现**方向对、细节漏**的问题；
- 尤其是需要把多个层次同时写全时，容易漏掉年份边界、存储开销、第二类优化手段或结尾收束句。

### 2. 下午题书写层面的易错点

- 已经不太会完全写偏，但还会出现“只写到一半”的情况：
  - 改写题能想到 `BETWEEN`，但条件年份写错；
  - 代价题能想到维护成本，但忘了存储开销；
  - 模板题能想到索引，但没把 SQL 改写 / 物理设计并列写出来；
- 说明当前最大问题不是“不会”，而是**写答案时没有把固定检查清单跑完整**。

### 3. 采分点意识层面的易错点

- 现在最值得继续强化的，不是再背更多概念，而是让这些短语变成自动落笔：
  - **语义一致 / 条件边界不变**
  - **维护成本 + 存储开销**
  - **索引 + SQL 改写 + 物理设计**
  - **收益 + 代价 + 综合权衡**
- 只要题干显式要求某一层，就必须在答案里显式写出来，不能靠“我心里知道”代替真正落笔。
