## 三、本次练习暴露出的关键易错点

### 1. 知识判断层面的易错点

- 已经能大体分清 **实体、联系、联系属性、独立记录型关系模式** 这几层，但一到开放题里，仍容易把“知道方向”写成“结构还不够严谨”；
- 对 **多次发生的信息**（如维修记录、支付记录、订单明细）已经有“不能只加字段”的意识，但还要继续压实：到底应该新增什么关系模式、主键怎么定、和谁关联；
- 对 **组合键**、**主键 / 外键位置** 的概念基础是有的，但在长答案中不一定每次都会显式写出来。

### 2. 下午题书写层面的易错点

- 现在最明显的问题，不是完全不会，而是 **答案会停在“方向正确”层**，没有继续写到“结构完整、字段准确、关系清楚、结尾收束到位”；
- 模板题里已经开始会按顺序落笔，但还会出现：
  - 只写了大方向，没把关键字段写全；
  - 只写了新增模式，没把主外键关系说清；
  - 结尾只写“便于管理”，没有写到“查询 / 统计 / 统一管理 / 扩展性”等更标准的收束句；
- 一旦题目涉及 **订单头 / 明细、支付记录、维修记录** 这类“结构化表达”场景，当前答案还不够稳定。

### 3. 采分点意识层面的易错点

- 只要题干出现“**多次维修**”“**详细金额和支付时间**”“**统一管理**”“**后续统计**”这类词，就要主动联想到：
  - 不能只加一个字段；
  - 很可能要新建独立关系模式；
  - 要把主业务对象与记录对象的关联显式写出来；
- 只要题干问“**为什么不能只这样设计**”，就必须写出 **原设计不足 + 新设计价值** 两层，不能只写其中一层；
- 只要题干问“**标准答案 / 模板题**”，就要检查自己是否写全：
  - 核心对象
  - 联系类型
  - 主键 / 外键 / 组合键
  - 联系属性 / 独立记录
  - 统一管理 / 查询统计 / 扩展价值的收束句

## 四、Day 21 需要继续巩固的重点

1. 再练“**多次发生信息 ≠ 直接加字段，优先警惕独立关系模式**”这条判断线；
2. 再练“**订单头 / 订单明细**”“**快递 / 支付记录**”“**故障 / 维修记录**”这三类典型结构；
3. 写数据库设计开放题时，强制显式写出：**主键、外键、组合键、联系类型**；
4. 继续压实“**先说明原设计为什么不够，再说明新模式如何支持后续管理 / 统计 / 查询**”的标准句序；
5. 对维修记录类题目，继续固定写法：**同一故障可多次维修 → 需保留维修历史 → 新建维修记录模式**；
6. 对支付记录类题目，继续固定写法：**金额 + 时间 + 方式 + 付款方 + 与主业务对象关联 + 统计价值**；
7. 对订单与商品类题目，继续固定写法：**订单头信息与明细信息分离，数量等业务发生信息放在明细 / 联系模式中**；
8. 收束句不要只停在“便于管理”，尽量补成：**更能真实反映业务语义，并支持后续查询、统计、扩展与统一管理。**
