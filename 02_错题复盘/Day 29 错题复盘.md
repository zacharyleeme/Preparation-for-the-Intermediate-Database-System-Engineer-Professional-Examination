### A. 客观题错题

#### 第8题

**题型归属：** 冲突操作基础题

**题目：**  
下列操作对中，最可能构成冲突操作的是（B ）  
A. 同一事务中对不同数据项的两次读  
B. 不同事务对不同数据项的两次写  
C. 不同事务对同一数据项的一读一写  
D. 不同事务对同一数据项的两次读

**你的实际作答：** B

**标准答案：** C

**本题采分点：**
1. 冲突操作必须发生在不同事务之间；
2. 冲突操作必须访问同一数据项；
3. 至少有一个操作是写操作；
4. 不同事务对不同数据项的写操作不构成冲突。

**本次已命中的采分点：**
- 识别到“不同事务”和“写操作”相关。

**本次漏掉的采分点：**
- 漏掉“同一数据项”这一必要条件。

**失分原因类型：**
- [x] 漏条件 / 漏关键词
- [x] 会概念但判断时少套一层条件

**正确理解应为：**
- 冲突操作三条件：**不同事务、同一数据项、至少一个写**。
- B 项虽然是不同事务、两次写，但对象是不同数据项，因此不冲突。

**标准作答路径：**
1. 先检查是否不同事务；
2. 再检查是否同一数据项；
3. 最后检查是否至少一个写；
4. 三个条件同时满足才是冲突操作。

**纠偏记忆：**
- “不同事务 + 同一数据项 + 至少一个写”三件套缺一不可。

**建议复习动作：**
1. 把第8题加入 Day 30 开头 3 分钟口头复述；
2. 遇到冲突题先在题边写：`不同事务 / 同一数据项 / 至少一写`；
3. 与第24题冲突图一起二刷。

### B. 下午题 / 主观题错题

#### 第20题（下午题）

**题型归属：** 2PL 概念辨析

**题目：**  
请用 80～120 字回答：
1. 两段锁协议为什么能保证可串行化调度？
2. 为什么它不能保证一定不会发生死锁？

要求答案中必须出现：**加锁阶段、解锁阶段、可串行化、循环等待**。

**你的实际作答：**  
- 因为两段锁协议规定事务同时只能处于一个阶段，当事务先加锁阶段，一旦释放锁就进入解锁阶段。在进入解锁阶段后事务不能再进行加锁。所以可以确保事务串行化调度
- 两段锁协议只能确保串行话，如果事务互相持有对方需要的锁，仍可能死锁。

**参考答案：**  
两段锁协议要求事务先进入加锁阶段，再进入解锁阶段；一旦开始释放锁，就不能再申请新锁。这样可以保证冲突操作的先后顺序与某个串行顺序一致，因此保证可串行化。  
但 2PL 不能保证无死锁，因为多个事务仍可能分别持有对方需要的锁，形成循环等待。

**本题采分点：**
1. 写出加锁阶段 / 解锁阶段；
2. 写出可串行化；
3. 写出循环等待；
4. 写出不保证无死锁。

**本次已命中的采分点：**
- 写到了加锁阶段、解锁阶段；
- 写到了可串行化 / 串行化调度；
- 写到了互相持有对方需要的锁、仍可能死锁；
- 基本命中“不保无死锁”。

**本次漏掉的采分点：**
- 没有明确写出关键词 **循环等待**；
- “确保事务串行化调度”表述不够精确，应写“保证冲突可串行化”，不是把并发执行变成物理串行执行；
- 存在笔误：“串行话”。

**采分点核查：** 3 / 4

**失分原因类型：**
- [x] 会分析但写不规范
- [x] 漏关键词

**正确理解应为：**
- 2PL 保证的是调度结果等价于某个串行顺序，即可串行化；
- 2PL 仍可能因事务互相等待对方已持有的锁而形成循环等待，从而死锁。

**标准作答路径：**
1. 先写 2PL 两阶段：加锁阶段、解锁阶段；
2. 再写开始解锁后不能再申请新锁；
3. 再写因此可保证可串行化；
4. 最后写仍可能出现循环等待，所以不保证无死锁。

**纠偏记忆：**
- 2PL 四词固定写全：**加锁阶段、解锁阶段、可串行化、循环等待**。

**建议复习动作：**
1. 把第20题标准答案手写 / 口述 2 遍；
2. Day 30 遇到 2PL 概念题，必须先写关键词再扩句；
3. 避免把“可串行化”写成“完全串行执行”。

#### 第23题（下午题）

**题型归属：** 抢红包并发 / 2PL 改写

**题目：**  
某红包记录的数据项 `X` 初始为空。`Ri(X)` 表示第 `i` 个抢红包人读取 `X`，`Wi(bi, X)` 表示第 `i` 个抢红包人把自己的编号写入 `X`。若多个抢红包人几乎同时读取到 `X` 为空，就可能都认为自己可以抢到红包。

现在要求：第一个响应的人最终抢到红包。请使用 `SLocki(X)`、`XLocki(X)`、`ULocki(X)` 给出一种满足 2PL 且不产生死锁的执行思路。

**你的实际作答：**  
```text
XLocki(X) -> Wi(bi, X) -> Commit -> ULocki(X) -> Ri(X) -> SLocki(X)
```

**参考答案：**  
一种思路是让第一个响应者先获得对 `X` 的排他锁，再完成读取、判断、写入、提交和解锁，其他抢红包人等待其释放锁后再读取。示例：

```text
XLock1(X)
a1 = R1(X)
若 X 为空，则 W1(b1, X)
Commit1
ULock1(X)

XLock2(X)
a2 = R2(X)
发现 X 已非空，不再写入
Commit2 / Rollback2
ULock2(X)
```

**本题采分点：**
1. 第一个响应者先拿到锁；
2. 使用排他锁保护“读—判断—写”整体过程；
3. 后续事务等待前一事务释放锁后再进入；
4. 满足 2PL，提交后再解锁；
5. 不形成循环等待。

**本次已命中的采分点：**
- 写到了 `XLocki(X)` 和 `ULocki(X)`；
- 有“提交后解锁”的痕迹。

**本次漏掉的采分点：**
- 写成了先写 `Wi` 再读 `Ri`，顺序错误；
- 没有体现“先读 X 是否为空，再判断是否写入”；
- 没有体现其他抢红包人等待；
- `ULock` 后又 `Ri`、`SLock`，破坏了 2PL 的阶段约束；
- 没有说明不产生死锁的原因。

**采分点核查：** 1 / 5

**失分原因类型：**
- [x] 会概念但不会下手
- [x] 步骤顺序混乱
- [x] 2PL 执行序列不规范

**正确理解应为：**
- 抢红包题不是“先写再读”，而是必须保护 **读 X 是否为空 → 判断 → 写入编号** 这一整段临界区；
- 后续事务必须等第一个事务提交并释放锁后才能再读；
- 一旦解锁，就不能再申请新锁，否则不满足 2PL。

**标准作答路径：**
1. `XLock1(X)`；
2. `R1(X)`，判断 X 是否为空；
3. 若为空，则 `W1(b1, X)`；
4. `Commit1`；
5. `ULock1(X)`；
6. 后续事务等待释放后再申请 `XLock(X)`，读取后发现已非空，不再写入。

**纠偏记忆：**
- 抢红包 2PL 固定模板：**排他锁 → 读 → 判断 → 写 → 提交 → 解锁 → 后者等待再读**。

**建议复习动作：**
1. Day 30 前必须重写第23题标准序列；
2. 以后凡是 2PL 执行序列题，禁止出现“解锁后再加锁 / 再读”的写法；
3. 把“读—判断—写”视为一个必须加锁保护的整体。

#### 第24题（下午题）

**题型归属：** 冲突可串行化判断

**题目：**  
给定调度：

```text
R1(A), W2(A), R2(B), W1(B)
```

请回答：
1. 哪些操作之间存在冲突？
2. 冲突图中有哪些边？
3. 该调度是否冲突可串行化？为什么？

**你的实际作答：**  
- 根据调度情况可得冲突图，T1 —> T2 和 T2 —> T1
- 冲突图中存在边 T2
- 该调度为不冲突可串行化

**参考答案：**
1. `R1(A)` 与 `W2(A)` 冲突，方向为 `T1 → T2`；`R2(B)` 与 `W1(B)` 冲突，方向为 `T2 → T1`；
2. 冲突图中有 `T1 → T2` 和 `T2 → T1`；
3. 图中有环，因此该调度不是冲突可串行化调度。

**本题采分点：**
1. 找出两个冲突数据项；
2. 画出双向边；
3. 判断有环，不可冲突可串行化。

**本次已命中的采分点：**
- 第一行写到了 `T1 → T2` 和 `T2 → T1`；
- 最后判断为不可冲突可串行化，方向正确。

**本次漏掉的采分点：**
- 没有明确写出 `R1(A)` 与 `W2(A)`、`R2(B)` 与 `W1(B)` 两组冲突操作；
- 第二行“冲突图中存在边 T2”表述错误 / 不完整；
- 没有明确写“图中有环”。

**采分点核查：** 2 / 3

**失分原因类型：**
- [x] 只写结论没写依据
- [x] 表述不完整
- [x] 漏步骤 / 漏关键词

**正确理解应为：**
- 冲突图题必须先列冲突操作，再列边，最后写有环 / 无环；不能只跳到结论。

**标准作答路径：**
1. 对 `A`：`R1(A)` 在前，`W2(A)` 在后，产生边 `T1 → T2`；
2. 对 `B`：`R2(B)` 在前，`W1(B)` 在后，产生边 `T2 → T1`；
3. 冲突图有环；

- 冲突图答案必须三段式：**冲突操作 → 冲突边 → 有环结论**。

#### 第25题（下午题）

**题型归属：** 两段锁协议执行序列

**题目：**  
事务 `T1` 需要读取账户 `A` 和账户 `B` 的余额并汇总；事务 `T2` 需要从账户 `A` 向账户 `B` 转账。为了减少死锁风险，要求两个事务都按账户号顺序先申请 `A` 的锁，再申请 `B` 的锁。

请给出一种满足 2PL 的可能执行序列，要求体现：加锁、等待 / 串行进入、读写、提交、解锁。

**你的实际作答：**  
```text
XLock1(A) —> XLock2(A) —> W2(A) —> Commit —> R1(A) —> ULock1(A) —> ULock2(A)
等待账户A锁释放 —> XLock1(B) —> XLock2(B) —> W2(B) —> Commit —> R1(B) —> ULock1(B) —> ULock2(B)
```

**参考答案：**  
一种满足 2PL 且降低死锁风险的写法：

```text
T1: SLock(A) → SLock(B) → R(A) → R(B) → 计算汇总 → Commit → Unlock(B) → Unlock(A)
T2: 等待 T1 释放 A、B 后
T2: XLock(A) → XLock(B) → R(A) → W(A) → R(B) → W(B) → Commit → Unlock(B) → Unlock(A)
```

也可让 `T2` 先执行，但两个事务都应按相同顺序申请 `A`、`B` 的锁。

**本题采分点：**
1. 按相同顺序申请锁；
2. 体现等待 / 串行进入；
3. 提交后解锁；
4. 满足 2PL。

**本次已命中的采分点：**
- 有加锁、读写、提交、解锁等词；
- 有“等待账户 A 锁释放”的意识。

**本次漏掉的采分点：**
- `T1` 持有 `A` 锁时，`T2` 不能直接 `XLock2(A)`；
- `T1` 在申请 `B` 前已经释放 `A`，不满足严格的 2PL 写法；
- 两个事务的锁、读写和提交顺序混在一起，不能形成清晰的“一个事务持锁完成后，另一个事务等待进入”；
- 读汇总事务 `T1` 通常应使用共享锁 `SLock`；转账事务 `T2` 修改数据应使用排他锁 `XLock`；
- 没有体现 `T1` 同时锁住 `A`、`B` 后再读取汇总，或 `T2` 同时锁住 `A`、`B` 后再转账。

**采分点核查：** 1 / 4

**失分原因类型：**
- [x] 会概念但不会写执行序列
- [x] 步骤顺序混乱
- [x] 2PL 加锁 / 解锁阶段边界不稳

**正确理解应为：**
- 要满足 2PL，一个事务应先拿到需要的锁，再执行读写；一旦开始解锁，就不能再申请新锁；
- 为避免死锁，两个事务都按 `A → B` 申请锁；
- 冲突时后来的事务等待，而不是同时拿到同一数据项的排他锁。

**标准作答路径：**
1. 选择一个事务先进入；
2. 按 `A → B` 申请锁；
3. 完成全部读写 / 计算；
4. 提交；
5. 解锁；
6. 后一事务等待释放后，再按同样顺序申请锁并执行。

**纠偏记忆：**
- 2PL 执行序列不要交叉乱写，优先写成：**T1 完整执行 → T2 等待后完整执行**。

**建议复习动作：**
1. Day 30 前重写第25题标准答案；
2. 用同一模板同时复写第23题、第25题；
3. 看到“满足 2PL”时先写：`Lock(A) → Lock(B) → R/W → Commit → Unlock(B) → Unlock(A)`。

#### 第27题（下午题）

**题型归属：** 日志恢复 / Redo 与 Undo 判断

**题目：**  
某系统在检查点后有如下事务状态：

- `T1`：检查点后开始，写过 `X`，故障前已提交；
- `T2`：检查点前开始，检查点后继续写过 `Y`，故障前未提交；
- `T3`：检查点后开始，未写数据，故障前未提交；
- `T4`：检查点前已提交，且不在检查点后的活跃事务列表中。

系统故障后，请判断 `T1`、`T2`、`T3`、`T4` 分别应进入 `Redo`、`Undo`，还是通常不处理，并说明理由。

**你的实际作答：**  
- T1应进入 redo ，因为T1事务在检查点后开始，且故障前已经提交
- T2应进入 undo ，因为T2事务在检查点后继续写过Y，且故障前未提交
- T3应进入 undo ，因为T3事务在检查点后开始，虽然未写入数据，但是故障前未提交
- T4通常不处理，因为T4事务在检查点前已经提交，且不在检查点后活跃的事务列表内

**参考答案：**
- `T1`：`Redo`。检查点后开始，写过数据，故障前已提交；
- `T2`：`Undo`。检查点前开始但检查点后继续写过数据，故障前未提交；
- `T3`：通常不处理或仅作为活跃未提交但无写事务处理。因为未写数据，无需撤销数据项更新；
- `T4`：通常不处理。检查点前已提交，且不在检查点后活跃事务列表中。

**本题采分点：**
1. 已提交写事务 Redo；
2. 未提交写事务 Undo；
3. 无写事务不作为重点撤销对象；
4. 检查点前稳定完成事务通常不处理。

**本次已命中的采分点：**
- `T1` Redo 正确；
- `T2` Undo 正确；
- `T4` 通常不处理正确。

**本次漏掉的采分点：**
- `T3` 未写数据，通常不需要撤销数据项更新；不能简单因为“未提交”就列入需要 Undo 的更新事务。

**采分点核查：** 3 / 4

**失分原因类型：**
- [x] 判断条件少看“是否写过数据”
- [x] Redo / Undo 名单机械化

**正确理解应为：**
- Undo 重点针对“未提交且已经写过数据”的事务；
- 未提交但没有写数据的事务，通常没有需要撤销的数据项更新。

**标准作答路径：**
1. 先判断是否故障前提交；
2. 再判断是否写过数据；
3. 已提交且写过 → Redo；
4. 未提交且写过 → Undo；
5. 未写数据 → 通常不处理数据项更新；
6. 检查点前已稳定完成且无后续活动 → 通常不处理。

**纠偏记忆：**
- 恢复名单别只看“提交 / 未提交”，还要看 **是否写过数据**。

**建议复习动作：**

2. 每次列 Redo / Undo 表时新增一列：`是否写过数据`；
