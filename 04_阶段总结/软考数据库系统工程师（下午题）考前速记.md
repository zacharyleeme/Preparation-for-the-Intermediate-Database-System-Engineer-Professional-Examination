# 软考数据库系统工程师（下午题）考前速记

> [!summary] 下午题核心特点
> 下午题真正考的不是“死记硬背”，而是：
> - 会不会分析关系模式
> - 会不会找函数依赖
> - 会不会拆表
> - 会不会写 SQL
> - 会不会分析事务调度
> - 会不会数据库设计

下午题的分数，80% 来自：

```text
数据库设计 + 规范化 + SQL + 并发控制
```

## 四、ER 建模速记（超高频）

### 1. 联系类型

| 类型 | 特点 |
|---|---|
| 1:1 | 一对一 |
| 1:N | 一对多 |
| M:N | 多对多 |

### 2. 转关系模式规则（必背）

| ER 元素 | 转换 |
|---|---|
| 实体 | 表 |
| 属性 | 字段 |
| 联系 | 外键/新表 |

### 3. 多对多（必建新表）

经典套路：

```text
学生 —— 选课 —— 课程
```

转为：

```text
选课(学号,课程号,...)
```

### 4. 下午题标准答题模板

联系转换：

```text
联系“采购”为多对多联系，
因此转换为独立关系模式：
采购(车型编号，供应商名称，零件编码，...)
```

## 五、函数依赖（下午最核心）

### 1. 判断口诀

> [!tip]
> 谁决定谁：左边唯一确定右边。

### 2. 常见函数依赖

```text
学号决定姓名：学号 → 姓名
课程号决定课程名：课程号 → 课程名
```

### 3. 下午题高频套路

1. 找候选键。
2. 找部分依赖、传递依赖。
3. 判断范式。

## 六、规范化（下午第一大核心）

### 1. 范式口诀（必背）

| 范式   | 消除         |
| ---- | ---------- |
| 1NF  | 字段不可分      |
| 2NF  | 消部分依赖      |
| 3NF  | 消传递依赖      |
| BCNF | 决定因素必须是候选键 |
| 4NF  | 消多值依赖      |

## 七、规范化标准答题模板（重点）

### 题型：判断是否满足 3NF

标准模板：

```text
关系模式不满足3NF。
因为存在传递依赖：
A→B
B→C
因此：
非主属性C传递依赖于主键A。
```

## 八、BCNF 高频套路（超高频）

近年真题：

- 2017 下午
- 2023 下午

判断关键：看决定因素是不是候选键。

若不是：不满足 BCNF。

## 九、4NF 高频（近年增加）

2018 下午考过。

4NF 核心：消除多值依赖。

```text
A →→ B
```

高频场景：一个病例对应多药品，药品之间独立，容易形成多值依赖。

## 十、模式分解（必会）

> [!tip] 口诀
> 一个依赖拆一张表。

高频套路：

```text
原表：
R(A,B,C)
A→B
B→C

分解：
R1(A,B)
R2(B,C)
```

## 十二、CREATE TABLE 高频模板（必背）

```sql
CREATE TABLE SC(
    Sno CHAR(10),
    Cno CHAR(10),
    Grade INT,
    PRIMARY KEY(Sno,Cno),
    FOREIGN KEY(Sno)
        REFERENCES Student(Sno)
);
```

## 十三、GROUP BY 高频陷阱

原则（超高频）：`SELECT` 中只能出现 `GROUP BY` 字段和聚合函数。

错误写法：

```sql
SELECT dept,name,COUNT(*)
FROM student
GROUP BY dept;
```

其中 `name` 不合法。

## 十四、EXISTS 高频模板

```sql
SELECT *
FROM Student S
WHERE EXISTS(
    SELECT *
    FROM SC
    WHERE SC.Sno=S.Sno
);
```

## 十五、JOIN 高频模板

```sql
SELECT *
FROM Student S
JOIN SC
ON S.Sno=SC.Sno;
```

## 十六、事务并发控制（下午超高频）

近年例题：

- 2017 下午红包系统
- 2018 下午超市销售
- 2019 下午银行转账
- 2021 下午日志恢复

## 十七、事务问题（必背）

| 问题 | 含义 |
|---|---|
| 丢失修改 | 写覆盖 |
| 脏读 | 读未提交 |
| 不可重复读 | 两次读不同 |
| 幻读 | 行数变化 |

## 十八、锁机制（必考）

| 锁 | 含义 |
|---|---|
| S 锁（共享锁） | 只能读 |
| X 锁（排它锁） | 读写都阻塞 |

## 十九、三级锁协议（高频）

| 协议 | 解决问题 |
|---|---|
| 一级 | 丢失修改 |
| 二级 | 脏读 |
| 三级 | 不可重复读 |

> [!tip] 口诀
> - 一级防丢失
> - 二级防脏读
> - 三级防重复
> - 幻读串行补

## 二十、日志恢复（必背）

### Undo / Redo

| 情况 | 操作 |
|---|---|
| 未提交事务 | Undo |
| 已提交事务 | Redo |

### 检查点作用

缩短恢复时间。

## 二十一、下午题恢复分析模板

```text
事务Ti已提交，
因此需要Redo。

事务Tj未提交，
因此需要Undo。
```

## 二十二、死锁（高频）

| 死锁条件 | 含义 |
|---|---|
| 互斥 | 资源独占 |
| 请求保持 | 占有再申请 |
| 不可剥夺 | 不能强制释放 |
| 循环等待 | 环形等待 |

## 二十三、数据库设计答题套路（非常重要）

1. 找实体。一般名词：用户、商品、医院、学生。
2. 找联系。动词：购买、选课、接种、配送。
3. 判断联系类型。
4. 补主键外键。

## 二十四、下午题“看到就写”

1. **多对多**：建立独立关系模式。
2. **外键**：作为外键参照 XX 表主键。
3. **不满足 3NF**：存在传递依赖。
4. **不满足 2NF**：存在部分依赖。

## 二十五、下午题最容易丢分点

| 易错点 | 正确做法 |
|---|---|
| 不写主键 | 必写 |
| 不写外键 | 必写 |
| 不说明理由 | 必写“因为...” |
| SQL 少 GROUP BY | 检查 |
| BCNF 不会判断 | 看决定因素 |
| 多对多不拆表 | 必拆 |
| Undo/Redo 写反 | 已提交 Redo |

## 二十六、考前必须背熟的模板

### 1. 范式模板

```text
存在部分依赖，
因此不满足2NF。
```

### 2. BCNF 模板

```text
决定因素不是候选键，
因此不满足BCNF。
```

### 3. 并发控制模板

```text
事务加锁后，
其他事务无法访问数据项。
```

### 4. 恢复模板

```text
已提交事务Redo，
未提交事务Undo。
```

## 二十七、下午题最后 30 分钟速记（最重要）

1. **范式口诀**：1 原子；2 部分；3 传递；BC 决定；4 多值。
2. **锁协议口诀**：一级丢失；二级脏读；三级重复；串行幻读。
3. **恢复口诀**：提交 Redo；未交 Undo。
4. **ER 口诀**：实体变表；联系变外键；多对多单独建表。

## 二十八、下午题真正的得分关键

> [!summary]
> 下午题不是拼“会不会”，而是拼会不会写标准答案。

重点做到：

- 一定写“原因”。
- 一定写“依赖关系”。
- 一定写“主键外键”。
- 一定写“属于/不属于某范式”。
- 一定按步骤写。

即使不会完整做，也能拿步骤分。
