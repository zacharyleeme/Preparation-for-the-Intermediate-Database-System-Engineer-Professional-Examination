### 2.2 为什么 Day 6 进入子查询与嵌套查询

Day 3 学的是单表基础查询，Day 4 学的是聚合与分组，Day 5 学的是多表连接。  
但真实考试里的 SQL 题，常常不会只让你“直接查”，而会要求你：

- 先找出某门课对应的学号，再查这些学生姓名
- 先找出平均成绩，再查高于平均成绩的记录
- 先判断某学生是否存在选课记录，再决定是否返回该学生
- 先筛出满足条件的集合，再让外层查询继续处理

这类题有一个共同点：**外层查询依赖内层查询的结果。**  
这就是子查询与嵌套查询的典型场景。

原因主要有三点：

1. **子查询是 SQL 上午选择题和下午分析题的高频考点**
2. **很多题本质上在考“先查什么，再用查出的结果做什么”**
3. **子查询与连接查询经常可以互相转化，是综合 SQL 能力的重要分水岭**

简单说：

- Day 3：学会从一张表里查数据
- Day 4：学会在单表或分组上做统计
- Day 5：学会把多张表关联起来
- Day 6：学会让**一个查询的结果，成为另一个查询的条件或数据来源**

### 2.3 今日核心知识框架

子查询（Subquery）的核心思路可以先用一句人话理解：

> **先算出一个中间结果，再拿这个中间结果继续查。**

最常见的基本形式：

```sql
SELECT 列名
FROM 表名
WHERE 字段 运算符 (
    SELECT 列名
    FROM 表名
    WHERE 条件
);
```

例如，查询“选修了课程号为 `C1` 的学生姓名”：

```sql
SELECT 姓名
FROM Student
WHERE 学号 IN (
    SELECT 学号
    FROM SC
    WHERE 课程号 = 'C1'
);
```

可以拆开理解成两步：

1. 内层先查出：哪些学号选了 `C1`
2. 外层再去 Student 表中查这些学号对应的姓名

今天要先建立 4 个基础判断：

1. **内层查询返回的是一个值，还是一组值？**
2. **外层查询是拿它来比较、判断存在、还是直接当数据源？**
3. **内层查询是否依赖外层当前这一行？**
4. **这道题更适合写成连接，还是更适合写成子查询？**

最常见的几类写法：

#### 1）单行子查询
内层只返回一个值，常配合 `=`、`>`、`<` 使用。

```sql
SELECT 学号, 成绩
FROM SC
WHERE 成绩 > (
    SELECT AVG(成绩)
    FROM SC
);
```

#### 2）多行子查询
内层返回多个值，常配合 `IN` 使用。

```sql
SELECT 姓名
FROM Student
WHERE 学号 IN (
    SELECT 学号
    FROM SC
    WHERE 成绩 >= 90
);
```

#### 3）存在性子查询
重点不在具体返回什么值，而在于“有没有满足条件的记录”，常用 `EXISTS`。

```sql
SELECT 姓名
FROM Student s
WHERE EXISTS (
    SELECT *
    FROM SC
    WHERE SC.学号 = s.学号
);
```

#### 4）相关子查询
内层查询会引用外层当前行的字段，说明内层不是独立执行一次，而是会随外层记录变化而变化。

### 2.4 必备结论

- 子查询是**嵌套在另一个查询内部的查询**
- 子查询通常先由内层产出结果，再由外层继续判断或输出
- 单行子查询通常与 `=`、`>`、`<` 等比较运算符搭配
- 多行子查询通常与 `IN` 搭配
- 若子查询可能返回多行，通常不能直接与 `=` 搭配
- `EXISTS` 判断的是**是否存在满足条件的记录**
- `NOT EXISTS` 判断的是**是否不存在满足条件的记录**
- 相关子查询会引用外层字段，内层结果随外层当前行变化
- 有些题既能写成连接，也能写成子查询
- 判断写法时，先看题目是在考“直接关联”还是“先查结果再判断”

### 2.5 高频考点拆解

#### 2.5.1 什么是子查询

子查询就是把一个查询写到另一个查询里面。  
例如：

```sql
SELECT 姓名
FROM Student
WHERE 学号 IN (
    SELECT 学号
    FROM SC
    WHERE 成绩 >= 90
);
```

本质是：

- 内层先找出成绩大于等于 90 的学号集合
- 外层再去学生表里查这些学号对应的姓名

高频误区：

- 不要把子查询理解成“两个独立 SQL”
- 它们是一个完整查询里的内外层关系
- 外层要不要能正确使用内层结果，是本题判断重点

#### 2.5.2 单行子查询与多行子查询

如果内层只返回一个值，比如平均成绩、最高成绩、某个编号，就属于单行子查询。

例如：

```sql
SELECT *
FROM SC
WHERE 成绩 > (
    SELECT AVG(成绩)
    FROM SC
);
```

这里 `AVG(成绩)` 只会返回一个平均值，所以外层可以直接用 `>` 比较。

如果内层可能返回多行，比如多个学号、多个课程号，就更适合用 `IN`。

例如：

```sql
SELECT 姓名
FROM Student
WHERE 学号 IN (
    SELECT 学号
    FROM SC
    WHERE 课程号 = 'C1'
);
```

高频误区：

- 内层返回多行时还误用 `=`
- 不先判断内层是一值还是一组值
- 看见子查询就机械套语法，不先看结果类型

#### 2.5.3 `IN` 与 `EXISTS` 的基本区别

`IN` 更像是在问：

> 这个值在不在内层查询返回的集合里？

`EXISTS` 更像是在问：

> 对于当前外层这行，内层有没有至少一条满足条件的记录？

例如：

```sql
SELECT 姓名
FROM Student s
WHERE EXISTS (
    SELECT *
    FROM SC
    WHERE SC.学号 = s.学号
);
```

含义是：
- 对 Student 表中的每个学生
- 去 SC 表里看是否存在该学生的选课记录
- 若存在，则返回该学生

高频误区：

- 把 `EXISTS` 当成比较具体值
- 忽略 `EXISTS` 只关心“是否存在”，不关心返回什么列
- 没看出内层是否引用了外层字段

#### 2.5.4 相关子查询与非相关子查询

若内层查询可以单独执行，不依赖外层字段，就是**非相关子查询**。  
例如：

```sql
SELECT 姓名
FROM Student
WHERE 学号 IN (
    SELECT 学号
    FROM SC
    WHERE 成绩 >= 90
);
```

内层本身就能查出结果，不需要外层参与。

若内层要引用外层当前行的字段，就是**相关子查询**。  
例如：

```sql
SELECT 姓名
FROM Student s
WHERE EXISTS (
    SELECT *
    FROM SC
    WHERE SC.学号 = s.学号
);
```

这里 `s.学号` 来自外层当前行，因此属于相关子查询。

高频误区：

- 没注意内层是否出现外层别名
- 把所有子查询都当成“先执行一次就完”的独立查询

#### 2.5.5 子查询与连接查询的关系

有些题既可以写成连接，也可以写成子查询。  
例如“查询选修了课程 C1 的学生姓名”，既可以用连接，也可以用子查询。

连接写法偏向：
- 直接把表关联起来
- 一次性取出相关字段

子查询写法偏向：
- 先找学号集合
- 再根据学号集合去外层查姓名

考试常考的不是“只能用哪一种”，而是：
- 你能不能看懂这两类写法表达的是同一类查询逻辑
- 你能不能判断某个选项为什么错在结果类型或语义不符

#### 2.5.6 常见陷阱：多行结果却用单值比较

例如：

```sql
SELECT 姓名
FROM Student
WHERE 学号 = (
    SELECT 学号
    FROM SC
    WHERE 课程号 = 'C1'
);
```

如果选修 `C1` 的学生不止一个，内层就会返回多个学号。  
这时 `=` 就不合适，通常应改为 `IN`。

因此做子查询题时，优先问自己：

1. 内层返回一行还是多行？
2. 外层运算符是否匹配？
3. 题目要求的是比较一个值，还是匹配一组值？

### 2.6 与前序知识的映射或衔接

Day 6 的子查询，并不是突然出现的新知识，而是在继续扩大前几天的 SQL 结构理解：

- Day 3：`SELECT / FROM / WHERE` 是最基础的查询骨架
- Day 4：聚合函数让你能先做统计结果
- Day 5：连接查询让你能把多张表直接关联
- Day 6：子查询让你能把“一个查询的结果”继续拿去做下一步判断

可以这样直观理解：

- **连接查询**：多个表横向拼起来查
- **子查询**：先查出中间结果，再纵向嵌套着查

所以 Day 5 和 Day 6 不是替代关系，而是互补关系。

### 2.7 易混点速记卡片

- **单行子查询**
  - 内层返回一个值
  - 常配 `=`、`>`、`<`

- **多行子查询**
  - 内层返回多个值
  - 常配 `IN`

- **`IN`**
  - 关注“值是否属于一个集合”

- **`EXISTS`**
  - 关注“是否存在满足条件的记录”

- **非相关子查询**
  - 内层可单独执行

- **相关子查询**
  - 内层引用外层当前行字段

- **连接 vs 子查询**
  - 连接：直接关联表
  - 子查询：先得中间结果再继续查

## 4. 答案解析（默认隐藏）

<details>
<summary>点击展开答案与解析</summary>

### 第1题
**答案：B**  
**解析：** 嵌套在另一个查询内部的查询称为子查询，也常叫嵌套查询。

### 第2题
**答案：B**  
**解析：** 子查询的核心思路就是先产生中间结果，再由外层查询继续判断、筛选或输出。

### 第3题
**答案：A**  
**解析：** 若内层查询只返回一个值，通常称为单行子查询。

### 第4题
**答案：B**  
**解析：** 当内层可能返回多个值时，外层更常与 `IN` 搭配；`=` 通常适合单值比较。

### 第5题
**答案：A**  
**解析：** `EXISTS` 只关心是否存在至少一条满足条件的记录，不强调具体返回什么值。

### 第6题
**答案：B**  
**解析：** `NOT EXISTS` 用于判断不存在满足条件的记录，常见于“没有选课”“没有下单”这类题。

### 第7题
**答案：C**  
**解析：** `AVG(成绩)` 是聚合函数，会返回一个平均值，因此这里是单行子查询。

### 第8题
**答案：B**  
**解析：** “先查出选修 C1 的学号，再查这些学生姓名”正是典型的子查询思路。

### 第9题
**答案：A**  
**解析：** 这类题应先查出平均成绩，再用外层判断哪些记录高于该平均值。

### 第10题
**答案：B**  
**解析：** 内层若引用外层当前行字段，属于相关子查询。

### 第11题
**答案：B**  
**解析：** 选项 B 中内层使用了外层别名 `s.学号`，因此是相关子查询。

### 第12题
**答案：A**  
**解析：** `IN` 用于判断外层字段值是否属于内层查询返回的集合。

### 第13题
**答案：A**  
**解析：** 内层若返回多行，外层却用 `=`，就会出现结果类型与比较运算符不匹配的问题。

### 第14题
**答案：A**  
**解析：** `IN` 更像集合成员判断，`EXISTS` 更像存在性判断，这是两者的核心区别。

### 第15题
**答案：B**  
**解析：** 该语句先查出成绩大于等于 90 的学号，再到 Student 表中取出对应学生姓名。

### 第16题
**答案：A**  
**解析：** “没有选课记录”最典型的思路是 `NOT EXISTS`，判断在 SC 中不存在对应记录。

### 第17题
**答案：B**  
**解析：** 非相关子查询可以独立执行，不依赖外层字段。

### 第18题
**答案：A**

### 第19题
**答案：B**  
**解析：** 很多查询既可以用连接表达，也可以用子查询表达，只是思路组织不同。

### 第20题
**答案：B**  
**解析：** “先找集合，再去外层继续查”更贴近子查询的表达方式。

### 第21题
**答案：A**

### 第22题
**答案：A**  
**解析：** 该写法直接判断：对于外层当前学生，SC 中是否存在对应选课记录。

### 第23题
**答案：A**  
**解析：** 内层先求平均成绩，外层再查高于平均成绩的记录。

### 第24题
**答案：A**

### 第25题

### 第26题
