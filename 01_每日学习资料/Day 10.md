### 2.3 今日核心知识框架

先用一句人话理解今天的内容：

> **函数依赖是在说“知道哪些属性，就能唯一确定哪些属性”；范式是在说“这个表结构是否已经足够规范、是否还藏着冗余和异常”。**

#### 1）什么是函数依赖

在关系模式中，如果属性组 X 的值一旦确定，属性组 Y 的值也就随之唯一确定，则称：

> **Y 函数依赖于 X**，记作 **X → Y**

例如：

- 学号 → 学生姓名
- 课程号 → 课程名
- （学号，课程号）→ 成绩

这些式子表达的不是“计算公式”，而是“**确定关系**”：

- 已知某个学号，就能唯一确定学生姓名
- 已知某门课程号，就能唯一确定课程名
- 已知某个学生和某门课程的组合，就能唯一确定该次成绩

#### 2）决定因素、超键、候选键

若 X → Y，则 X 称为 Y 的**决定因素**。

继续往考试常见概念推进：

- **超键**：能唯一标识元组的属性组
- **候选键**：最小超键，即再去掉任何一个属性就不能唯一标识元组
- **主属性**：属于某个候选键的属性
- **非主属性**：不属于任何候选键的属性

例如在关系模式：

- 选课（学号，课程号，成绩）

通常（学号，课程号）可以唯一标识一条选课记录，因此：

- （学号，课程号）是候选键
- 学号、课程号属于主属性
- 成绩通常属于非主属性

#### 3）完全函数依赖、部分函数依赖、传递函数依赖

这是 Day 10 的核心区分。

##### （1）完全函数依赖

若 Y 依赖于 X，且 X 的任何真子集都不能决定 Y，则称 **Y 完全函数依赖于 X**。

例如：

- （学号，课程号）→ 成绩

成绩通常既不能仅由学号决定，也不能仅由课程号决定，因此是对组合键的完全函数依赖。

##### （2）部分函数依赖

若 Y 依赖于组合属性组 X，但只依赖于 X 的一部分，则称 **部分函数依赖**。

例如关系：

- 选课（学号，课程号，学生姓名，课程名，成绩）

若有：

- 学号 → 学生姓名
- 课程号 → 课程名
- （学号，课程号）→ 成绩

这里“学生姓名”只依赖学号，“课程名”只依赖课程号，而不是依赖整个组合键（学号，课程号），因此存在部分函数依赖。

##### （3）传递函数依赖

若 X → Y，Y → Z，并且 Y 不是候选键，则可理解为 Z **传递依赖**于 X。

例如：

- 学号 → 系号
- 系号 → 系名

则有：

- 学号 → 系名（通过系号传递得到）

这类依赖常常意味着表中存在冗余与更新异常。

#### 4）什么是范式

范式可以理解为：

> **评价关系模式规范程度的一组标准。**

数据库系统工程师上午题里，最常考的是：

- 1NF
- 2NF
- 3NF
- BCNF

你不需要把范式背成空定义，而要抓住它们分别在“消除什么问题”。

#### 5）第一范式（1NF）

1NF 的最基础要求是：

> **属性值具有原子性，不可再分。**

例如一个“联系电话”字段里直接塞多个电话号码，就不符合关系模型提倡的原子化表达。

所以 1NF 更像是“表能不能作为规范关系模式讨论”的入门门槛。

#### 6）第二范式（2NF）

2NF 建立在 1NF 基础上，核心要求是：

> **非主属性必须完全函数依赖于每一个候选键，不允许对候选键的真子集发生部分函数依赖。**

考试里通常可直接记：

- **有组合键 + 非主属性只依赖键的一部分 → 违反 2NF**

#### 7）第三范式（3NF）

3NF 建立在 2NF 基础上，核心要求是：

> **非主属性不应传递依赖于候选键。**

考试里常见口径：

- **若存在“键 → 非主属性 A → 非主属性 B”这种链条，则往往不满足 3NF**

#### 8）BCNF（巴斯-科德范式）

BCNF 比 3NF 更严格，常见判断口径是：

> **每一个决定因素都应当包含候选键，或者更直接地说：每一个决定因素本身都应是候选键。**

在上午题里，BCNF 常考成“它比 3NF 更严格”。

#### 9）为什么要做范式设计

范式不是为了背定义，而是为了解决实际问题：

- 减少冗余
- 避免插入异常
- 避免删除异常
- 避免更新异常
- 为后续模式分解提供依据

例如一个表里同时重复保存大量“学生名、课程名、系名”，一旦课程名或系名变化，就容易出现多处修改不一致的问题。

#### 10）Day 10 的真正主线

今天真正要建立的思维链是：

> **先有关系模式 → 再看属性之间谁决定谁 → 再判断是否存在部分依赖 / 传递依赖 → 再判断该模式属于哪一范式**

这条链会直接决定你后面学候选键、模式分解和无损连接时是否稳。

### 2.4 必备结论

- 函数依赖表示“已知 X，能唯一确定 Y”，记作 **X → Y**
- 决定因素是依赖左边的属性组
- 候选键是**最小超键**
- 主属性属于某个候选键，非主属性不属于任何候选键
- 完全函数依赖强调“依赖整个键，不能只依赖其中一部分”
- 部分函数依赖通常是 **2NF** 要消除的重点问题
- 传递函数依赖通常是 **3NF** 要消除的重点问题
- 1NF 强调属性值原子性
- 2NF = 1NF + 消除对候选键真子集的部分函数依赖
- 3NF = 2NF + 消除非主属性对候选键的传递函数依赖
- BCNF 比 3NF 更严格，要求每个决定因素都应是候选键
- 范式设计的核心目标是**降低冗余，减少异常，提高结构规范性**

### 2.5 高频考点拆解

#### 2.5.1 函数依赖考的是“确定关系”，不是“公式计算”

看到 X → Y 时，不要把它理解成数学函数或程序代码。  
它表达的是：**同一个 X 值，不允许对应多个不同的 Y 值。**

例如：

- 学号 → 姓名：合理
- 姓名 → 学号：通常不一定成立

高频误区：
- 把“可能相关”误当成“能唯一确定”
- 把“常见情况”误当成“严格依赖关系”

#### 2.5.2 候选键的关键不是“大”，而是“最小”

很多题目会故意拿一个能唯一标识元组的属性组来迷惑你。  
如果它还能再删掉某个属性仍保持唯一标识，那它只是超键，不是候选键。

高频误区：
- 只看“能唯一标识”，忘记“最小性”
- 把主键和候选键概念混成一类记忆

#### 2.5.3 2NF 重点抓“组合键 + 部分依赖”

只要题目里出现组合键，就要立刻警觉：

- 有没有某个非主属性只依赖组合键的一部分？

例如：

- （学号，课程号）是键
- 学号 → 学生姓名

这就说明学生姓名并不依赖整个组合键，而是只依赖其中一部分，因此不满足 2NF。

高频误区：
- 记住了“2NF 消除部分依赖”，但不会落到具体表结构中识别
- 看到组合键却没有主动检查键的真子集

#### 2.5.4 3NF 重点抓“键 → 非主属性 → 非主属性”

3NF 的题常用“院系号、院系名”“部门号、部门名”这类结构来考。  
只要出现：

- 学号 → 系号
- 系号 → 系名

就要立刻想到：

- 系名是通过系号传递依赖于学号的
- 这会带来冗余和更新异常

高频误区：
- 把“传递依赖”和“多步推理”混淆
- 没先判断中间属性是不是非主属性

#### 2.5.5 BCNF 比 3NF 更严格

上午题经常不要求你做特别复杂的 BCNF 推导，更多是考基本判断：

- 3NF 已经比较规范
- BCNF 还要进一步约束决定因素

高频误区：
- 以为 3NF 和 BCNF 没区别
- 见到“决定因素不是候选键”却没联想到 BCNF

#### 2.5.6 范式判断要有固定顺序

建议以后做题按这条顺序：

1. 先找候选键
2. 再区分主属性 / 非主属性
3. 再判断是否有部分函数依赖
4. 再判断是否有传递函数依赖
5. 最后判断是 1NF / 2NF / 3NF / BCNF 中的哪一级

这样比背定义更稳。

### 2.7 易混点速记卡片

- **函数依赖**
  - 不是计算公式，是确定关系

- **超键 vs 候选键**
  - 超键能唯一标识；候选键还要求最小

- **主属性 vs 非主属性**
  - 在候选键里的，是主属性；不在任何候选键里的，是非主属性

- **完全函数依赖**
  - 依赖整个键，不依赖键的一部分

- **部分函数依赖**
  - 非主属性只依赖组合键的一部分

- **传递函数依赖**
  - 键先决定 A，A 再决定 B

- **2NF**
  - 消除部分函数依赖

- **3NF**
  - 消除非主属性的传递函数依赖

- **BCNF**
  - 每个决定因素都应是候选键

## 4. 答案解析（默认隐藏）

<details>
<summary>点击展开答案与解析</summary>

### 第1题
**答案：B**  
**解析：** 若 X 的值一旦确定，Y 的值也能唯一确定，则称 Y 函数依赖于 X。

### 第2题
**答案：C**  
**解析：** `X → Y` 表示同一个 X 值不能对应多个不同的 Y 值，即 X 唯一确定 Y。

### 第3题
**答案：A**  
**解析：** 在依赖 `X → Y` 中，X 是决定 Y 的属性组，因此称为决定因素。

### 第4题
**答案：B**  
**解析：** 候选键是最小超键，既能唯一标识元组，又满足最小性。

### 第5题
**答案：C**  
**解析：** 属于某个候选键的属性称为主属性；不属于任何候选键的称为非主属性。

### 第6题
**答案：B**  
**解析：** 在“选课（学号，课程号，成绩）”中，若候选键是 `(学号, 课程号)`，则“成绩”通常不属于候选键，因此是非主属性。

### 第7题
**答案：A**  
**解析：** “成绩”必须依赖整个 `(学号, 课程号)` 组合，既不能只由学号决定，也不能只由课程号决定，因此是完全函数依赖。

### 第8题
**答案：B**  
**解析：** “学生姓名”只依赖学号，而候选键为 `(学号, 课程号)`，所以属于对组合键真子集的部分函数依赖。

### 第9题
**答案：C**  
**解析：** 由“学号 → 系号，系号 → 系名”可知“系名”通过“系号”传递依赖于“学号”。

### 第10题
**答案：B**  
**解析：** 1NF 最基础的要求是属性值原子化，不可再分。

### 第11题
**答案：B**  
**解析：** 2NF 主要解决的是非主属性对候选键真子集的部分函数依赖问题。

### 第12题
**答案：A**  
**解析：** 3NF 建立在 2NF 基础上，重点消除非主属性对候选键的传递函数依赖。

### 第13题
**答案：B**  
**解析：** BCNF 比 3NF 更严格，它对决定因素提出了更强约束。

### 第14题
**答案：D**  
**解析：** 若每个决定因素本身都是候选键，则该关系模式满足 BCNF。

### 第15题
**答案：B**  
**解析：** 规范化设计的核心目标是降低冗余，减少插入、删除、更新异常，而不是物理存储层面的直接优化。

### 第16题
**答案：A**  
**解析：** 该关系模式虽满足原子性，但“学生姓名”只依赖学号，“课程名”只依赖课程号，均属于对组合键的部分函数依赖，因此不满足 2NF，最高只到 1NF。

### 第17题
**答案：B**  
**解析：** 候选键是单属性“学号”，不存在部分函数依赖问题，因此可达 2NF；但又有“学号 → 系号 → 系名”的传递依赖，所以不满足 3NF，最高只到 2NF。

### 第18题
**答案：C**  
**解析：** 单属性候选键没有“真子集”可言，因此不会发生对候选键真子集的部分函数依赖；但仍可能存在传递依赖。

### 第19题
**答案：B**  
**解析：** 非主属性若只依赖组合键的一部分，就是典型的部分函数依赖，直接提示不满足 2NF。

### 第20题
**答案：B**  
**解析：** 非主属性对候选键发生传递函数依赖，是不满足 3NF 的典型信号。

### 第21题
**答案：B**  
**解析：** 候选键一定能唯一标识元组，所以一定是超键；但超键未必满足最小性，因此未必是候选键。

### 第22题
**答案：B**  
**解析：** Day 10 在 Day 9 的关系模式基础上，继续分析属性间依赖关系并判断范式级别，这是规范化设计的入口。

### 第23题
**答案：B**  
**解析：** 若已经满足 3NF，但仍存在某个决定因素不是候选键，则通常不满足 BCNF，因为 BCNF 要求每个决定因素都应是候选键。

### 第24题
**答案：B**  
**解析：** 范式判断的稳妥顺序是：先找候选键，再区分主属性与非主属性，再判断部分依赖、传递依赖等依赖类型。

### 第25题
**答案：A**  
**解析：** 同一信息被反复保存，改一处后其他地方可能不一致，这是典型的更新异常。

### 第26题
**答案：B**  
**解析：** “年龄”若可由“出生日期”推导得到，首先应判断其是否为派生属性，而不是先把它往外键上联想。

### 第27题
**答案：A**  
**解析：** 在 `R(学号, 姓名)` 中，`学号` 为决定因素且本身就是候选键，因此该关系模式可满足 BCNF。

### 第28题
**答案：B**  
**解析：** 候选键为 A，不存在部分函数依赖问题，因此至少满足 2NF；但存在 `A → B → C` 的传递依赖，所以不满足 3NF，最高只到 2NF。

### 第29题
**答案：A**  
**解析：** 候选键为 `(订单号, 商品号)`，而“订单日期”只依赖订单号，属于对组合键真子集的部分函数依赖，因此不满足 2NF，最高只到 1NF。

### 第30题
**答案：B**  
**解析：** Day 10 的主线就是：先有关系模式，再分析属性之间的决定关系，最后判断范式级别，为后续模式分解做准备。
