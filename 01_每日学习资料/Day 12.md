### 2.3 今日核心知识框架

今天要固定下来的不是零散定义，而是一套**分解题标准分析顺序**：

> **先看原模式为什么要分解 → 再看分解后是否无损 → 再看是否保持函数依赖 → 最后评价这个分解方案是否值得采用。**

#### 1）为什么要做模式分解

模式分解的本质，是把一个存在冗余、更新异常、插入异常、删除异常或范式不足的问题关系模式，拆成若干个更合理的子关系模式。

分解的常见目标包括：

- 消除部分函数依赖或传递函数依赖
- 降低数据冗余
- 减少更新异常
- 提高结构清晰度
- 让关系模式达到更高范式

所以，分解不是“拆得越碎越好”，而是要在**规范性、可恢复性、依赖保持性**之间取得平衡。

#### 2）什么叫无损连接

无损连接（Lossless Join）关注的是：

> **把原关系模式分解后，再通过连接运算能不能准确还原回原来的关系，而不产生伪元组、不丢原信息。**

如果分解后重新连接会出现多余组合、伪元组或信息扭曲，那就是有损连接。

考试里最常见的是**二元分解**判定：

若把关系模式 `R` 分解为 `R1` 和 `R2`，则在函数依赖集合 `F` 下，该分解无损连接的常用判定口径是：

> **若 `R1 ∩ R2 → R1` 或 `R1 ∩ R2 → R2` 成立，则该分解为无损连接。**

更好记的人话版本是：

> **公共属性必须至少能决定其中一个子模式的全部属性。**

#### 3）什么叫保持函数依赖

保持函数依赖（Dependency Preservation）关注的是：

> **原关系模式中的函数依赖，在分解之后，是否仍能通过各子关系模式中的函数依赖共同推出。**

直观理解：

- 若原来有依赖 `X → Y`
- 分解后至少应该能在某个子模式中直接保留，或者由各子模式投影后的依赖联合推出

如果分解之后，原有某些关键依赖无法在子模式里检查或推出，就说明**不保持函数依赖**。

#### 4）为什么“无损”与“保持依赖”不是一回事

这是 Day 12 最爱考的坑点之一。

- **无损连接**关心的是：能不能把数据表连接回去而不失真
- **保持函数依赖**关心的是：原有约束能不能继续在分解后有效表达和检查

两者相互相关，但不是同一件事。

可能出现的情况：

- 分解**无损**，但**不保持依赖**
- 分解**保持依赖**，但不一定天然就是最优范式设计

所以考试里不能因为“它无损”就顺手写成“它一定保持依赖”，也不能反过来混为一谈。

#### 5）二元分解的无损连接怎么做题

对于 `R → {R1, R2}` 的二元分解，最稳的做题顺序是：

1. 先找出公共属性 `R1 ∩ R2`
2. 再看公共属性的闭包，能否推出 `R1` 或 `R2` 的全部属性
3. 若能推出其中一个子模式全部属性，则该分解无损
4. 若公共属性无法决定任一子模式，则通常不能判为无损

常见速记：

- **公共属性是某个子模式的键 → 无损**
- **没有公共属性 → 通常有损**

#### 6）保持函数依赖怎么做题

保持函数依赖题，最稳的做法不是凭感觉，而是按顺序看：

1. 原模式的关键函数依赖有哪些
2. 每条依赖在分解后的哪个子模式中还能直接表达
3. 若不能直接表达，是否能由各子模式投影后的依赖联合推出
4. 若有关键依赖彻底散掉、无法恢复，则不保持函数依赖

考试里的简化判断口径常常是：

- **若一条原始依赖的决定属性和被决定属性被拆散，且没有任何子模式完整保留这条依赖，则要警惕“不保持函数依赖”**

#### 7）分解与范式的关系

Day 11 的范式判断，是 Day 12 分解的前提。

因为你只有先知道：

- 候选键是谁
- 非主属性是谁
- 是被部分依赖卡住，还是被传递依赖卡住

你才能合理决定：

- 应该怎么拆
- 先把哪组属性拆出来
- 拆完之后范式是否提升

所以 Day 12 不是新开一条线，而是把 Day 11 的判断结果**转化成结构设计动作**。

#### 8）考试里最实用的总流程

遇到模式分解题，优先固定成下面这条链路：

> **先找候选键与问题依赖 → 再说明为什么要分解 → 再写分解结果 → 再判无损连接 → 再判是否保持函数依赖 → 最后评价分解后的范式或设计效果。**

只要顺序稳定，下午题就不容易“想到一点写一点”。

### 2.4 必备结论

- 模式分解的目标之一是消除冗余和更新异常
- 无损连接关注“分解后再连接是否能准确还原原关系”
- 保持函数依赖关注“原有依赖约束能否在分解后继续表达和检查”
- 无损连接与保持函数依赖不是同一概念
- 二元分解中，若公共属性能决定某个子模式全部属性，则该分解无损
- 公共属性是某个子模式的键，通常可直接判无损
- 没有公共属性的分解，通常不能判为无损
- 若原依赖在分解后没有任何子模式完整保留，且无法由投影依赖联合推出，则不保持函数依赖
- 3NF 分解通常优先兼顾保持函数依赖
- BCNF 分解更严格，但有时可能不保持函数依赖

### 2.5 高频考点拆解

#### 2.5.1 “为什么分解”不能只写“为了更规范”

**核心概念：** 分解是为了解决冗余、异常和低范式问题。  
**常见考法：** 问“为什么要把该关系模式分解”“该模式存在哪类问题”。  
**高频误区：** 只写“为了提高范式”，不写具体问题。  
**快速判断提示：** 先看是否存在部分依赖 / 传递依赖，再把它对应到冗余与异常。  
**例子：** `(订单号, 商品号) → 数量`，`商品号 → 商品名`，说明商品名只依赖商品号，原模式有冗余，适合分解。

#### 2.5.2 无损连接题先盯“公共属性”，不要直接盯全表

**核心概念：** 二元分解无损连接看公共属性能否决定某个子模式。  
**常见考法：** 给出 `R1`、`R2`，让你判断是否无损。  
**高频误区：** 一上来对整张表乱看，不先抽出公共属性。  
**快速判断提示：** 先写 `R1 ∩ R2 = ...`，再判断它是不是 `R1` 或 `R2` 的键。  
**例子：** `R1(A,B)` 与 `R2(A,C)`，若 `A → B`，则公共属性 A 能决定 `R1`，该分解无损。

#### 2.5.3 保持函数依赖题先列原依赖，再看分解后谁还留着

**核心概念：** 保持依赖不是“看起来差不多”就算保留。  
**常见考法：** 给出分解方案，问是否保持函数依赖。  
**高频误区：** 只看一个子模式，忘了检查全部原依赖。  
**快速判断提示：** 先把原依赖一条条列出，再逐条对照子模式。  
**例子：** 原依赖有 `B → C`，但分解后没有任何子模式同时包含 B 和 C，就要警惕不保持依赖。

#### 2.5.4 无损 ≠ 保持依赖

**核心概念：** 这两个结论必须分开写。  
**常见考法：** 同一道题同时问“是否无损”“是否保持依赖”。  
**高频误区：** 写出一个结论后，顺手把另一个也写成同样结果。  
**快速判断提示：** 一个看连接是否失真，一个看依赖是否还在。  
**例子：** 某分解公共属性满足无损条件，但某条关键依赖被拆散，仍可能不保持依赖。

#### 2.5.5 BCNF 分解更严格，但不一定最方便实现依赖检查

**核心概念：** 更高范式不等于在考试里所有指标都自动最好。  
**常见考法：** 问 3NF 分解和 BCNF 分解在依赖保持上的区别。  
**高频误区：** 认为“范式越高就越完美”。  
**快速判断提示：** 记住 3NF 分解通常更容易兼顾依赖保持，BCNF 更强调消除异常。  
**例子：** 某 BCNF 分解虽然结构更纯净，但可能需要跨表联查才能验证原依赖。

### 2.7 标准答题模板 / 采分点提示

#### 模板1：无损连接判定题
1. 先写分解结果：`R → R1, R2`。
2. 写出公共属性：`R1 ∩ R2 = ...`。
3. 根据函数依赖求公共属性闭包。
4. 判断公共属性是否能决定 `R1` 或 `R2` 的全部属性。
5. 若能，则写“因此该分解为无损连接”；若不能，则写“因此不能判为无损连接 / 该分解为有损连接”。

#### 模板2：保持函数依赖判定题
1. 先列出原关系模式中的关键函数依赖。
2. 再看每条依赖是否在某个子关系模式中被完整保留。
3. 若不能直接保留，再判断能否由各子模式投影依赖联合推出。
4. 若所有关键依赖均能保留或推出，则写“保持函数依赖”；否则写“不保持函数依赖”。

#### 模板3：分解方案综合评价题
1. 先写原模式存在的问题（部分依赖 / 传递依赖 / 冗余 / 异常）。
2. 再给出分解结果。
3. 再判无损连接。
4. 再判是否保持函数依赖。
5. 最后写该分解后达到的范式或设计效果。

### 第24题
**参考答案：**  
该分解是无损连接。因为两个子模式的公共属性是 `商品号`，而在 `R2(商品号, 商品名, 单价)` 中，`商品号` 可以决定 `商品名` 与 `单价`，因此公共属性 `商品号` 是 `R2` 的键，可据此判定该分解无损连接。

**采分点：**
- 给出“无损连接”的结论
- 写出公共属性 `商品号`
- 说明 `商品号` 是 `R2` 的键或可决定 `R2` 其他属性

**作答提醒：**
- 用“公共属性能决定一边”来写
- 别只说“因为有共同属性所以无损”

### 第25题
**参考答案：**  
该分解保持原函数依赖。因为：
- `(订单号, 商品号) → 数量` 保留在 `R1` 中；
- `商品号 → 商品名` 与 `商品号 → 单价` 保留在 `R2` 中。  
原有主要函数依赖都能在子模式中直接表达。

**采分点：**
- 给出“保持函数依赖”的结论
- 逐条对应三条依赖与子模式
- 结论完整

**作答提醒：**
- 原依赖一条条列，不要漏
- 每条依赖对应一个子模式

### 第26题
**参考答案：**  
原关系模式至少存在以下问题：
1. **数据冗余**：同一商品在多个订单中出现时，商品名和单价会被重复存储；
2. **更新异常**：若商品单价变化，需要修改多条记录，容易不一致；
3. **插入 / 删除异常**：若某商品尚无订单，难以单独记录商品信息；若删掉最后一条订单记录，可能连商品信息一并丢失。  
分解后，`R1` 负责订单明细，`R2` 负责商品基本信息，从而减少冗余并降低异常。

**采分点：**
- 至少写出两类异常或不合理点
- 能说明分解后订单信息和商品信息分离
- 能把“解决什么问题”对应到具体子模式

**作答提醒：**
- 至少写“冗余 + 更新异常”两类
- 若能补充插入 / 删除异常更完整

### 第27题
**参考答案：**  
候选键为 **A**。因为：
- `A → B`
- `B → C`
- `C → D`  
由此可推出 `A⁺ = {A, B, C, D}`，所以 A 能决定全部属性；同时 A 为单属性，满足最小性，因此 A 是候选键。

**采分点：**
- 写出候选键 A
- 写出或说明闭包 `A⁺` 包含全部属性
- 点明最小性

**作答提醒：**
- 最稳写法是直接写 `A⁺`
- 别漏“单属性所以满足最小性”

### 第28题
**参考答案：**  
该分解保持原函数依赖。因为：
- `A → B` 保留在 `R1(A, B)` 中；
- `B → C` 保留在 `R2(B, C)` 中；
- `C → D` 保留在 `R3(C, D)` 中。  
原模式中的关键依赖均可在分解后的子模式中直接检查。

**采分点：**
- 给出“保持函数依赖”的结论
- 逐条对应三条原依赖
- 用语清楚

**作答提醒：**
- 把三条依赖逐条写出来最稳
- 不要只写“都保留了”

### 第29题
**参考答案：**  
该分解可以视为合理的无损分解。分析思路可写为：
1. `R1` 与 `R2` 的公共属性为 `B`，且 `B → C`，因此能保证这一步连接不失真；
2. `R2` 与 `R3` 的公共属性为 `C`，且 `C → D`，因此能保证后续连接关系合理；
3. 该分解顺着原依赖链逐段展开，通常可视为无损且结构清晰的分解方案。

**采分点：**
- 给出“可视为合理无损分解”的结论
- 写出公共属性 B、C
- 能结合依赖链说明思路

**作答提醒：**
- 这是综合题，写思路比只写结论更重要
- 至少写出一层公共属性与依赖关系

### 第30题
**参考答案：**  
原关系模式的主要函数依赖为 `A → B`、`B → C`、`C → D`。将其分解为 `R1(A, B)`、`R2(B, C)`、`R3(C, D)` 后，这三条依赖分别保留在对应子关系模式中，因此该分解保持函数依赖。进一步看，分解沿原依赖链逐步展开，公共属性可以支撑连接过程，因此可视为合理的无损分解。综上，该分解既较好保留了原依赖，也使结构更清晰。

**采分点：**
- 顺序完整：原依赖 → 分解结果 → 保持依赖 → 无损判断 → 结论
- 句子结构清楚
- 至少包含两个判断结论和对应理由

**作答提醒：**
- 按模板写，不要东一句西一句
- 先列依赖，再写结论更稳
