### 2.3 今日核心知识框架

今天要固定下来的，不是一堆零散术语，而是一条完整的数据库设计作答链：

#### 1）需求抽取路径

> **先找对象，再找动作，再找约束，再找重复出现的信息**

- 名词性、可独立存在、需要长期保存的信息，优先考虑实体；
- 多个对象之间发生的业务行为，优先考虑联系；
- 若某个信息描述的是“某次联系发生时才有的值”，优先考虑联系属性；
- 若同一对象能出现多次，如“多个联系电话、多个收货地址、多个维修记录”，通常不能简单塞进单个字段。

#### 2）联系类型判断路径

> **先问一边能对应几个，再问另一边能对应几个**

- 一方对多方：1:N
- 双方都可能多个：M:N
- 双方都最多一个：1:1
- 不能凭“业务重要”判联系类型，只能按题干约束判。

#### 3）ER 转关系模式路径

> **实体先成表，联系再看类型处理，最后补主键、外键、联系属性**

- 强实体：直接转成关系模式；
- 多值属性：单独拆成关系模式；
- 1:N 联系：通常把 1 端主键放到 N 端作外键；
- M:N 联系：单独建关系模式，并带两端主键与联系属性；
- 弱实体 / 依赖存在的对象：主键中常带上其依赖实体主键。

#### 4）关系模式补全路径

> **先看题干约束，再看已有模式，再补缺口，不要凭想象加列**

补空时优先检查：
- 主键是否完整；
- 外键是否对应到正确实体；
- 多值属性是否已拆表；
- 联系属性是否落到了联系表 / 联系关系模式中；
- 扩展需求出现后，是否只改 ER 图却没同步改关系模式。

#### 5）扩展需求改图改模路径

> **先判断是“增加新实体 / 新联系 / 新属性 / 新关系模式”，再落到具体修改**

例如：
- “增加送货上门服务” → 不是只补一句说明，而要判断新增的是联系、属性还是新关系模式；
- “记录支付金额和支付时间” → 如果同一业务对象可能有多次支付记录，往往要单独建表；
- “记录故障维修历史” → 如果一个故障可能维修多次，维修记录往往不能合并成单个字段。

### 2.4 必备结论

- 数据库设计题先抽实体与业务对象，再判联系，不要一上来就补关系模式。
- 多值属性通常不能直接塞进单表字段，往往需要拆成独立关系模式。
- 联系属性应放在联系所在的关系模式中，而不是随意挂到某个实体表中。
- 1:N 联系通常把 1 端主键放入 N 端作外键；M:N 联系通常单独成表。
- 组合键场景下，不能漏写全部组成字段。
- 若题干要求记录“详细金额和时间”等多次发生信息，通常要考虑独立关系模式。
- 扩展需求题必须同时考虑 **ER 图补充** 与 **关系模式修改**，不能只做一半。
- 关系模式补全题不只看“像不像”，而要回到题干约束逐项对应。
- 数据库设计下午题比的不是术语堆砌，而是“对象—关系—约束—模式”的转换纪律。
- 写答案时，**主键、外键、联系类型、联系属性** 这四类词要尽量显式写出。

### 2.5 高频考点拆解

#### 2.5.1 多值属性什么时候要拆表

**核心概念：** 一个实体的某个属性如果可重复出现多个值，往往不能简单存在同一字段里。  
**常见考法：** “一个租客有多个联系电话”“一个顾客有多组收货地址”。  
**高频误区：** 直接把多个电话 / 多个地址写成一个字段。  
**快速判断提示：** 看到“多个”“多组”“可重复登记”，优先考虑单独关系模式。  
**小例子：** 租客（租客编号，姓名，……）+ 联系电话（电话号码，租客编号）。

#### 2.5.2 联系属性别挂错位置

**核心概念：** 只在“某次业务发生时”才有的值，往往是联系属性。  
**常见考法：** 签订日期、押金、月租金、投资额、投资时间。  
**高频误区：** 把签约日期直接写进租客表，或把投资金额直接挂在公司实体上。  
**快速判断提示：** 先问：这个值属于某个对象本身，还是属于“对象之间这次关系”？  
**小例子：** 合同（合同编号，租客编号，楼编号，公寓号，经理编号，签订日期，起始日期，租期，月租金，押金）。

#### 2.5.3 1:N 与 M:N 不能凭感觉判

**核心概念：** 联系类型必须按题干“最多 / 最少 / 是否可多个”来判断。  
**常见考法：** 一个经理可管理多幢楼，一幢楼必须由一个经理管理；一个租客可租多套公寓，一份合同可能有多个租客。  
**高频误区：** 只因为某一方重要，就误判成 1:1。  
**快速判断提示：** 分别问左右两边：一方最多对应几个另一方？  
**小例子：** 经理—公寓楼是 1:N，而合同—租客在合租场景下可能不是简单 1:1。

#### 2.5.4 扩展需求往往不是“加一列”这么简单

**核心概念：** 一旦题干出现“历史记录”“多次发生”“详细金额和时间”，往往意味着需要新关系模式。  
**常见考法：** 维修记录、支付记录、快递费支付、送货上门扩展。  
**高频误区：** 直接在原表加一个“维修内容”或“支付金额”字段。  
**快速判断提示：** 先问：这条信息会不会同一主业务对象出现多次？  
**小例子：** 一个故障可能对应多次维修，维修记录应独立表达。

#### 2.5.5 关系模式补空先查“键”再查“值”

**核心概念：** 模式补空题首先要保证主键、外键、联系方向不出错。  
**常见考法：** 补楼编号、月租金、订单时间、外键编号等。  
**高频误区：** 先想业务描述，忘了检查主外键链条。  
**快速判断提示：** 先画出引用方向：这个字段来自哪个实体 / 哪个联系？  
**小例子：** 公寓（楼编号，公寓号，……）的主键往往是组合键，不是单独的公寓号。

### 2.6 下午题视角下的题型映射

| 下午题类型 | 常见问法 | 作答关键动作 | 常见丢分点 |
|---|---|---|---|
| ER 图补全题 | “补充图中的联系对象和联系类型” | 先找两端实体，再按题干约束判 1:1 / 1:N / M:N | 凭感觉判联系类型、漏写联系对象 |
| 关系模式补全题 | “补充逻辑结构设计中的空缺” | 先核对主键外键，再补联系属性或多值属性拆分结果 | 漏组合键、把联系属性挂错表 |
| 扩展需求改图改模题 | “根据新增需求补充 ER 图并修改关系模式” | 先判断新增实体 / 联系 / 属性，再落到新关系模式 | 只改图不改表，或只改表不改图 |
| 多值属性处理题 | “多个联系电话 / 多组地址如何设计” | 单独建关系模式，并保留对主实体的引用 | 直接塞进单字段 |
| 设计说明题 | “请用 4～6 句说明设计思路” | 按需求 → 实体 / 联系 → 模式 → 约束 → 结论写 | 只写结论，不写转换过程 |

### 2.7 标准答题模板 / 采分点提示

#### 模板1：ER 图补全题
1. 先根据题干找出联系所关联的实体。  
2. 再分别判断每一端最多能对应几个另一端对象。  
3. 写出联系类型：1:1、1:N 或 M:N。  
4. 若题干给出了时间、金额、数量等跟业务发生有关的信息，再判断是否属于联系属性。  
5. 最后用一句话收束：该联系反映了什么业务约束。

#### 模板2：关系模式补全题
1. 先看已有关系模式中缺的是主键、外键还是联系属性。  
2. 再回到题干找对应业务约束。  
3. 若是实体唯一标识，优先考虑主键；若是引用其他实体，优先考虑外键。  
4. 若是签订时间、金额、数量、状态等随业务发生变化的信息，优先考虑联系属性或独立关系模式。  
5. 最后检查是否漏了组合键、多值属性拆表与参照方向。

#### 模板3：扩展需求改图改模题
1. 先说明扩展需求引入了什么新业务对象或新业务行为。  
2. 判断应增加的是新实体、新联系，还是在已有联系上增加属性。  
3. 将变化后的 ER 图核心关系用文字说清。  
4. 再写出新增或修改后的关系模式。  
5. 最后补一句：这样可以支持后续统一管理 / 统计 / 查询需求。

### 2.9 易混点速记卡片

- **实体**：能独立长期保存的信息对象  
  **联系**：对象之间的业务关系

- **实体属性**：对象自身一直拥有的信息  
  **联系属性**：业务发生时才出现的信息

- **多值属性**：一个对象可有多个值  
  **普通属性**：一个对象通常只有一个值

- **1:N**：一边一个，另一边多个  
  **M:N**：两边都可能多个

- **主键**：唯一标识本表记录  
  **外键**：引用别的表主键

- **扩展需求**：先问会不会多次发生  
  **会多次发生**：优先警惕要不要独立建表

### A 轨｜上午基础巩固题（18 题）

#### 第1题

1. **题型：单选**  **来源：2016～2023 真题主线整理（ER 模型）**

   A. 生成最终 SQL 执行计划  
   B. 描述实体、联系及业务语义

#### 第2题

2. **题型：单选**  **来源：2017 下午真题主线改编（公寓楼管理）**  
   若“每个经理可以管理多幢公寓楼，每幢公寓楼必须由一个经理管理”，则经理与公寓楼之间的联系类型应为（B ）

   C. M:N

**答案：B**  
**解析：** 题干明确写出：一个经理可管理多幢楼，一幢楼必须由一个经理管理，因此是典型 1:N 联系。

### 第3题
**答案：C**  
**解析：** “多个联系电话”是多值属性，高频稳妥做法是拆成独立关系模式，并关联租客编号。

### 第4题
**答案：C**  
**解析：** 购买数量不是顾客静态属性，也不是商品静态属性，而是订单与商品这次业务发生时的数量信息，更适合放在联系属性或明细关系模式中。

### 第5题
**答案：A**  
**解析：** 若每个快递员只负责一个工作站点，则快递员模式中保留站点编号作为外键是合理表达方式之一。

### 第6题
**答案：C**  
**解析：** 题干明确“同一楼内公寓号唯一，不同楼之间可重复”，所以必须结合楼编号与公寓号共同唯一标识。

### 第7题
**答案：B**  
**解析：** 支付记录涉及金额、时间、支付方式等多次发生或需独立统计的信息，更稳妥的是单独建立支付关系模式。

### 第8题
**答案：C**  
**解析：** 签订日期、租期、押金、月租金等都依附于“这次签约 / 这份合同”，属于联系或合同模式中的信息。

### 第9题
**答案：C**  
**解析：** M:N 联系最稳妥的转换方式，就是单独建立关系模式，并带上双方主键与联系属性。

### 第10题
**答案：B**  
**解析：** 在 1:N 联系中，将 1 端主键放入 N 端作外键，核心是在 N 端表达“我归属于哪个 1 端对象”。

### 第11题
**答案：A**  
**解析：** 配送时间、签收时间、签收快照属于配送这次业务发生时才产生的信息，更偏联系发生时的属性。

### 第12题
**答案：B**  
**解析：** 若责任人由快递员兼任，责任人编号可以引用快递员的标识，而不是重新创造一个与快递员完全无关的新实体。

### 第13题
**答案：B**  
**解析：** 扩展需求题的第一步，是先判断新需求引入的是实体、联系、属性还是新的关系模式，然后再落到具体修改。

### 第14题
**答案：B**  
**解析：** 一项故障可对应多次维修，因此维修记录应作为可重复发生的记录型关系模式来处理，而不是塞进单个字段。

### 第15题
**答案：B**  
**解析：** 补关系模式空缺时，先查主键 / 外键与引用方向是否正确，再看联系属性与业务约束，是更稳妥的顺序。

### 第16题
**答案：C**  
**解析：** 主键用于唯一标识本表记录，外键用于建立和其他表之间的引用关系。

### 第17题
**答案：C**  
**解析：** 数据库设计题最常见丢分点之一，就是只写结论，不说明实体、联系、联系类型与模式转换过程。

### 第18题
**答案：A**  
**解析：** 面对数据库设计题，按“需求 → 实体 / 联系 → 模式 → 键约束”的顺序输出，更容易稳住采分点。

### 第19题
**参考答案：**  
这个场景下，至少应识别的核心实体包括：公寓楼、公寓、租客、员工（经理 / 维修工）、合同。  
“经理—公寓楼”应是 **1:N** 联系，因为一个经理可管理多幢楼，而每幢楼必须由一个经理管理。  
“租客—合同”不能简单看成 1:1，因为题干明确一份合同可对应一个或多个租客，且一个租客也可租多套公寓，实际应结合具体设计进一步表达合租与多次签约关系。

**采分点：**
- 能列出核心实体
- 能判断经理—公寓楼为 1:N
- 能指出租客—合同不是简单 1:1，需要结合题干约束处理

**作答提醒：**
- 先列实体，再判联系
- 不要漏“合同”这类承载联系属性的重要对象

### 第20题
**参考答案：**  
（a）更合理应为 **楼编号**；（b）更合理可填 **月租金**。  
公寓关系模式不能只把“公寓号”单独当主键，因为题干已说明不同公寓楼内公寓号可相同，因此只有“楼编号 + 公寓号”组合起来才能唯一标识一套公寓。

**采分点：**
- 写出（a）= 楼编号
- 写出（b）= 月租金
- 能说明公寓号必须与楼编号组合唯一

**作答提醒：**
- 先检查题干唯一性约束
- 组合键不能漏字段

### 第21题
**参考答案：**  
这里不能只在“故障”里放一个维修内容字段，因为一项故障可能对应多次维修，若只放一个字段，会丢失维修历史，也无法记录每次维修的日期与内容。  
更合理的做法，是至少增加与故障、维修工关联的“维修记录”关系模式，例如：维修记录（维修编号或故障编号 + 维修日期，故障编号，维修工编号，维修日期，维修内容）。

**采分点：**
- 写出“可多次维修”这一原因
- 写出需要保留维修历史
- 给出新增关系模式的核心字段方向

**作答提醒：**
- 先写为什么原设计不够
- 再写新增关系模式

### 第22题
**参考答案：**  
该场景的核心对象包括公寓楼、公寓、租客、员工和合同，其中经理与公寓楼之间是一对多联系。  
在关系模式设计时，公寓楼应保留经理编号作为外键，公寓则应使用“楼编号 + 公寓号”这样的组合键进行唯一标识。  
合同模式中需要承载签订日期、起始日期、租期、月租金、押金等与签约行为相关的信息。  
如果后续出现故障维修等扩展需求，还应补充新的业务对象或记录型关系模式，而不是简单增加一个字段。  
整体上，这样的设计更能真实反映业务对象、业务关系与后续管理需求。

**采分点：**
- 顺序完整：对象 → 联系 → 模式 → 约束 / 结论
- 写出组合键、外键、合同属性等关键点
- 有完整收束句

### 第23题
**参考答案：**  
这里更适合增加“快递费支付”关系模式，而不是只在“快递”表里加金额字段，原因在于支付记录不仅有金额，还涉及支付时间、支付方式、付款方等信息。  
此外，题干强调要统计支付情况，说明支付信息具有独立记录与分析价值。  
如果后续存在多次支付、补差或支付方式统计需求，单一金额字段也难以扩展。

**采分点：**
- 写出支付信息不只有金额
- 写出支付记录具有独立统计价值
- 能说明只加字段扩展性差

**作答提醒：**
- 先写“为什么不够”
- 再写“为什么独立记录更稳妥”

### 第24题
**参考答案：**  
一个合理的关系模式设计思路例如：快递费支付（支付编号，快递编号，支付金额，支付时间，支付方式，付款方）。  
其中支付编号可作为主键，快递编号可作为外键关联到快递；付款方可用于区分发件人或收件人；支付方式用于区分预支付与到付。  
若题目需要更细，还可补充支付状态等字段，但核心主线是“唯一标识 + 业务关联 + 金额时间方式”。

**采分点：**
- 唯一标识
- 与快递关联
- 金额、时间、方式、付款方

**作答提醒：**
- 主键外键要显式写
- 不要只写字段名，不说用途

### 第25题
**参考答案：**  
这类题最容易出现的 3 类错误包括：  
1. 只在原表中增加金额字段，忽略支付时间、支付方式与付款方等独立信息；  
2. 没有显式写出支付记录与主业务对象之间的关联主外键；  
3. 没有意识到“详细记录 / 统计分析 / 多次发生”通常意味着需要独立关系模式。

**采分点：**
- 只加字段不建表
- 漏主外键关联
- 忽视独立记录与统计价值

### 第26题
**参考答案：**  
该场景要求系统不仅知道快递费金额，还要记录支付时间、支付方式和付款方，并支持后续统计分析。  
因此，若只在快递表中增加一个金额字段，就无法完整表达支付记录，也不利于后续扩展。  
更合理的做法，是新增“快递费支付”关系模式，用于独立保存支付编号、快递编号、支付金额、支付时间、支付方式和付款方。  
这样既能保持主业务对象清晰，也能支持后续支付查询、对账与统计分析。  
总体上，这种设计比“只加一个字段”更符合数据库设计的结构化表达要求。

**采分点：**
- 顺序完整：业务要求 → 原设计不足 → 新关系模式 → 统计价值 / 结论
- 写出新增模式核心字段
- 有完整收束句

### 第27题
**参考答案：**  
“订单”与“订单中的商品明细”不宜简单压成一张只按订单编号唯一的表，因为一份订单可包含多个商品，这本身就是一对多或明细型结构。  
若只按订单编号唯一，往往无法自然表达同一订单中的多种商品及其数量，也会给后续统计和配送带来困难。  
更稳妥的做法，是把订单头信息与订单商品明细分开表达，或通过联系模式记录订单与商品的对应关系及数量。

**采分点：**
- 一份订单可包含多个商品
- 只按订单编号唯一难以表达明细
- 需要订单头 / 明细分离或联系模式支持

### 第28题
**参考答案：**  
“一个顾客有多组收货地址”更适合设计为独立关系模式，因为地址是可重复出现的多值信息。  
如果直接把多个地址字段写在顾客表里，不仅结构僵硬，也不利于后续扩展和统一管理。  
单独建立“收货地点”关系模式，还能自然记录收货人、联系电话等与某组地址配套的信息。

**采分点：**
- 多组地址属于多值属性
- 多地址字段结构僵硬、扩展性差
- 独立关系模式更利于统一管理

### 第29题
**参考答案：**  
这个扩展需求至少会影响订单、销售、配送以及顾客 / 收货地点等核心对象或联系。  
因为实体店销售一旦也支持送货上门，就不再只是单纯线下成交记录，而要纳入后续订单与配送统一管理。  
所以不能只在原有“销售”表末尾加一个送货标志，否则无法完整表达下单、配送、签收等后续业务链条，也不利于和网购订单统一管理。

**采分点：**
- 写出受影响的核心对象 / 关系
- 写出送货上门引入了后续订单 / 配送链条
- 写出只加标志字段不够

### 第30题
**参考答案：**  
回答数据库设计题时，首先应从需求中抽取核心实体与业务行为，例如顾客、订单、商品、合同、站点或支付记录等。  
然后根据题干约束判断实体之间是一对一、一对多还是多对多联系，并识别哪些信息属于联系属性。  
接着将实体和联系转换为关系模式，显式写出主键、外键以及必要的组合键、多值属性拆表结果。  
若题干包含扩展需求，还要说明新增的是实体、联系还是独立关系模式。  
最终答案应既能反映业务语义，又能支持后续查询、统计和统一管理。

**采分点：**
- 顺序完整：实体 / 行为 → 联系类型 → 模式 / 键约束 → 扩展价值
- 写出主外键、多值属性、联系属性等关键层
- 有完整综合收束句
