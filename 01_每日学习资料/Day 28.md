### 2.3 今日核心知识框架

今天分成两条主线：

> **主线 A：并发控制**  
> **主线 B：故障恢复**

#### 1）并发控制：先看现象，再定术语

并发题先不要急着背名词，先看“发生了什么”：

| 现象                 | 对应术语  | 快速判断句                  |
| ------------------ | ----- | ---------------------- |
| 两个事务都基于旧值计算，后写覆盖前写 | 丢失修改  | “本来两次修改都该保留，结果前面的效果没了” |
| 读到了别人未提交的数据        | 脏读    | “如果对方回滚，我读到的就是脏数据”     |
| 同一事务前后两次读同一数据，值不一样 | 不可重复读 | “我自己没改，但再读时数据变了”       |
| 同一条件两次查集合，行数变了     | 幻读    | “不是某一行值变了，而是结果集合变了”    |

#### 2）两段锁协议：会写“锁申请—等待—提交—解锁”

2PL 的重点不是口号，而是写出步骤：

1. 事务在读 / 写共享数据前先申请锁；
2. 写操作通常要申请 **排他锁 Xlock**；
3. 若锁冲突，后来的事务必须等待；
4. 一个事务一旦开始释放锁，就不能再申请新锁；
5. 满足 2PL 可以保证可串行化，但**不能保证一定不会死锁**。

#### 3）恢复题：先分故障，再看日志与检查点

恢复题的判断顺序建议固定如下：

1. 先看故障类型：事务故障 / 系统故障 / 介质故障；
2. 再看故障发生时哪些事务已经提交，哪些事务未提交；
3. 再看检查点位置：重点分析检查点后仍活跃或发生更新的事务；
4. 最后列：
   - 已提交但可能未落盘 → `Redo`
   - 未提交但已写数据 → `Undo`

#### 4）事务性质与恢复动作的对应关系

| 场景 | 被破坏的性质 | 恢复动作 |
|---|---|---|
| 正在执行的事务因系统故障中断 | 原子性 | Undo |
| 已提交事务的修改尚未写盘，系统就崩溃 | 持久性 | Redo |
| 单个事务自身逻辑错误导致失败 | 原子性 / 一致性 | 回滚该事务 |
| 磁盘介质损坏导致数据库文件丢失 | 持久保存能力 | 先恢复备份，再重做日志 |

#### 5）今天作答顺序

今天下午题建议固定按这个顺序写：

1. 先判断题型：并发异常 / 2PL / 检查点恢复 / 故障类型；
2. 再写最核心的结论：异常名、故障类型、Redo / Undo 列表；
3. 再补理由：覆盖关系、未提交、已提交、检查点后活跃等；
4. 最后检查：是否写清锁申请、等待、提交、解锁，是否把 `Redo / Undo` 颠倒。

### 2.4 必备结论

- `Redo` 针对 **已提交但可能尚未落盘** 的事务。
- `Undo` 针对 **未提交但已经写过数据** 的事务。
- 系统故障导致事务中断时，未完成事务的原子性会被破坏，需要撤销。
- 已提交事务若未持久化就崩溃，持久性会被破坏，需要重做。
- 加入检查点的主要作用是 **提高系统故障恢复效率**，不是减少并发冲突。
- 两段锁协议能够保证 **可串行化调度**，但**不能保证不发生死锁**。
- 一旦事务开始释放锁，就不能再申请新锁，这是 2PL 的核心硬规则。
- `Read Committed` 能避免脏读，但一般仍可能出现不可重复读和幻读。
- 并发异常题先看“最终值和正确串行结果是否一致”，再决定异常名。
- 恢复题先看 **检查点和提交状态**，不要一上来机械背 `Redo / Undo`。

### 2.5 高频考点拆解

#### 2.5.1 丢失修改：最常考的并发异常入口

**核心概念：** 两个事务都根据旧值计算新值，后写把前写覆盖掉。  
**常见考法：** 库存扣减、余额修改、在库数量加减。  
**高频误区：** 只会说“并发有问题”，不会明确写出“丢失修改”；或者只写最终值，不说明为什么错。  
**快速判断提示：** 看两个事务是否都读到同一个旧值，再各自写回。  
**小例子：** `T1` 和 `T2` 都读到 `X=10`，分别算出 `8` 和 `7`，后写 `7` 覆盖前写 `8`。

#### 2.5.2 两段锁协议：会定义不够，必须会改写

**核心概念：** 事务先进入加锁阶段，开始释放锁后不得再申请新锁。  
**常见考法：** 要求把给定错误调度改写成满足 2PL 的序列。  
**高频误区：** 只写“给数据加锁”，不写谁先申请、谁等待、谁提交、谁解锁。  
**快速判断提示：** 写题时至少出现这 4 个动作：`Xlock` / `等待` / `Commit` / `Unlock`。  
**小例子：** `T1: Xlock(X) → R(X) → W(X) → Commit → Unlock(X)`；`T2` 等待 `T1` 释放锁后再进入。

#### 2.5.3 2PL 能保可串行化，但不能保无死锁

**核心概念：** 2PL 约束锁的申请与释放顺序，可以保证调度与某个串行顺序等价。  
**常见考法：** 判断“哪个说法错误”或解释为什么仍可能死锁。  
**高频误区：** 把“可串行化”和“不会死锁”混成一件事。  
**快速判断提示：** 只要两个事务分别占有对方需要的锁，就可能形成循环等待。  
**小例子：** `T1` 持有 `A` 等 `B`，`T2` 持有 `B` 等 `A`。

#### 2.5.4 检查点：不是事务控制，而是恢复提速点

**核心概念：** 检查点把某时刻的恢复起点做成标记，减少系统故障恢复时需要扫描和处理的日志范围。  
**常见考法：** 选择题判断检查点作用，或在恢复题中让你先看检查点后的活跃事务。  
**高频误区：** 误以为检查点能减少并发冲突或避免死锁。  
**快速判断提示：** 看到“检查点”，第一反应应是“恢复效率”和“恢复分析起点”。

#### 2.5.5 故障类型：事务故障、系统故障、介质故障分清楚

**核心概念：** 
- **事务故障**：单个事务自身出错；
- **系统故障**：整个系统崩溃重启，内存状态丢失；
- **介质故障**：磁盘等存储介质损坏。  
**常见考法：** 给场景判断故障类型和恢复动作。  
**高频误区：** 把“程序出错”都笼统写成系统故障。  
**快速判断提示：** 看影响范围：一个事务、整个系统、还是底层存储介质。

### 2.6 下午题视角下的题型映射

| 下午题类型 | 常见问法 | 作答关键动作 | 常见丢分点 |
|---|---|---|---|
| 并发异常分析 | “最终值是多少？属于哪类不一致性？” | 先算最终值，再与正确串行结果比较，最后写异常类型 | 只写现象，不写术语；只写术语，不写原因 |

### 2.7 标准答题模板 / 采分点提示

#### 模板1：并发异常分析题

1. 先写初值和每个事务读到的旧值；
2. 再写每个事务根据旧值得出的新值；
3. 写出数据库最终值；
4. 与正确串行结果比较；
5. 最后点名异常类型，并说明“后写覆盖前写 / 读了未提交 / 前后两次读取不一致”等原因。

#### 模板2：两段锁协议改写题

1. 先确定共享数据项是谁；
2. 第一个进入的事务先申请 `Xlock`（若涉及写）；
3. 在持锁期间完成读、计算、写回、提交；
4. 提交后再 `Unlock`；
5. 第二个事务在锁冲突时等待，待前一个事务解锁后再继续；
6. 写题时尽量显式出现：`Xlock`、`等待`、`Commit`、`Unlock`。

#### 模板3：恢复题（检查点 + Redo / Undo）

1. 先找检查点位置；
2. 列出故障发生时已提交事务和未提交事务；
3. 已提交但可能未落盘的事务列入 `Redo`；
4. 未提交但已更新的事务列入 `Undo`；
5. 检查点前已稳定完成且不在活跃列表中的事务，通常不再作为重点处理对象。

#### 模板4：故障类型判断题

1. 看影响范围：单个事务 / 整个系统 / 存储介质；
2. 写故障类型；
3. 写首要恢复动作：
   - 事务故障：回滚该事务；
   - 系统故障：按日志进行 Undo / Redo；
   - 介质故障：先恢复备份，再重做日志。

**来源：真题风格整理（检查点后事务状态判断）**

某系统在检查点后有如下事务状态：

- `T1`：检查点后开始，写过数据，并在故障前提交；
- `T2`：检查点前已开始，在检查点后继续写数据，但故障前尚未提交；
- `T3`：检查点后开始，写过数据，故障前尚未提交；
- `T4`：检查点前已提交，且不在检查点后的活跃事务列表中。

#### 第30题（案例组 4 / 小问 2）

**题型归属：** 死锁分析与改写  
**来源：2019 下午真题主线改编（转账 / 汇总死锁）**

设有两个事务：

- `T1`：先对数据项 `A` 加排他锁，再申请数据项 `B` 的排他锁；
- `T2`：先对数据项 `B` 加排他锁，再申请数据项 `A` 的排他锁。

请回答：
1. 该调度为什么可能形成死锁？
2. 若仍要满足两段锁协议，给出一种更安全的改写思路，使两个事务尽量避免形成死锁。

**作答区：**
- 因为T1先对数据项A进行加锁，T2先对数据项B进行了加锁。然后T1向数据B申请加锁时，需要等待T2事务对数据项B先解锁，事务T2对数据项A进行加锁申请，又需要等待T1事务对数据项A进行解锁。两个事务对对方需要解锁的事务进行了加锁，所以形成了死锁
- 在事务T1对数据项A加锁，事务T2对数据项B加锁之后，此时不允许事务对其他数据进行加锁，需要两个事务对当前加锁数据项解锁之后，才允许进行加锁操作。

### 第9题

**解析：** 两段锁协议可以保证并发调度与某个串行调度等价，因此保证可串行化。

### 第10题
**答案：B**  
**解析：** 2PL 能保证可串行化，但不能保证一定不发生死锁；死锁仍可能因为循环等待产生。

### 第11题
**答案：B**  
**解析：** `Read Committed` 的最低保证就是不读取其他事务尚未提交的数据，即避免脏读。

### 第12题
**答案：B**  
**解析：** 服务器重启、所有运行中事务被中断，影响范围是整个系统，属于系统故障。

### 第13题
**答案：A**  
**解析：** 单个事务自身逻辑错误导致该事务被终止，而其他事务不受影响，属于事务故障。

### 第14题
**答案：C**  
**解析：** 事务故障发生时，应由 DBMS 对出错事务执行回滚，而不是依赖用户手工修补。

### 第15题
**答案：A**  
**解析：** 共享锁允许读取，但不允许直接修改被锁定数据项。

### 第16题
**答案：C**  
**解析：** 排他锁与共享锁冲突，T2 必须等待 T1 释放锁后才能继续。

### 第17题
**答案：A**  
**解析：** 恢复题首先要定位检查点、事务提交状态和故障时刻，这决定了后续 Redo / Undo 的分析范围。

### 第18题
**答案：B**  
**解析：** `T1` 已提交但可能未落盘，应 `Redo`；`T2` 未提交但已写数据，应 `Undo`。

### 第19题
**参考答案：**
```sql
SELECT D.Dno, D.Dname
FROM Dept D
LEFT JOIN Emp E ON D.Dno = E.Dno
WHERE E.EmpNo IS NULL;
```

**采分点：**
- 从 `Dept` 出发；
- 使用 `LEFT JOIN` 保留全部部门；
- 使用右表关键字段 `E.EmpNo IS NULL` 判断无匹配；
- 输出 `Dno, Dname`。

**作答提醒：** Day 27 这里的重点不是方向，而是把判空字段锁死成“右表关键字段为空”。

### 第20题
**参考答案：**
```sql
SELECT G.Gno, G.Brand
FROM GOODS G
WHERE NOT EXISTS (
    SELECT 1
    FROM SALES S
    WHERE S.Gno = G.Gno
);
```

**采分点：**
- 从 `GOODS` 出发；
- 使用 `WHERE NOT EXISTS (...)`；
- 子查询中写出 `S.Gno = G.Gno` 这一内外层关联条件。

**作答提醒：** Day 27 这里的关键不是 `NOT EXISTS` 壳子本身，而是“相关子查询”的关联条件不能漏。

### 第21题
**参考答案：**
- 执行完该并发序列后，数据库中的 `X = 4`；
- 原因是：
  - 出库事务先读到 `X = 3`，算得 `ao = 2`，写回后 `X = 2`；
  - 入库事务也基于旧值 `X = 3` 计算出 `aI = 4`，随后写回 `X = 4`；
- 若按正确串行逻辑，先出库再入库或先入库再出库，最终都应回到 `X = 3`；
- 这里发生的是 **丢失修改**：后写回覆盖了前一事务的修改效果。

**采分点：**
- 算对最终值 `X = 4`；
- 指出正确结果本应为 `3`；
- 写出“后写覆盖前写”的原因；
- 点名异常类型为丢失修改。

### 第22题
**参考答案：**
一种可接受的满足 2PL 的执行序列如下：

```text
XLockO(X)
ao = Ro(X)
ao = ao - 1
Wo(ao, X)
CommitO
UnlockO(X)

XLockI(X)
aI = RI(X)
aI = aI + 1
WI(aI, X)
CommitI
UnlockI(X)
```

若把两个事务放在同一条时间线上表达，也可以写成：

1. 出库事务 `O` 先申请 `XLockO(X)`；
2. `O` 完成读、减 1、写回、提交；
3. `O` 释放 `X`；
4. 入库事务 `I` 再申请 `XLockI(X)`；
5. `I` 完成读、加 1、写回、提交；
6. `I` 再释放 `X`。

**采分点：**
- 先加锁再读写；
- 写操作使用排他锁；
- 后一事务在锁冲突时等待 / 串行进入；
- 提交后再解锁；
- 体现“两段锁协议”而不是仅仅“加了锁”。

### 第23题
**参考答案：**
1. **不可重复读**：同一事务前后两次读取同一行数据，第二次读到的是另一个已提交事务修改后的新值。  
2. **脏读**：读取了另一个事务尚未提交的数据，而该事务后来回滚。  
3. **幻读**：两次用同一条件查询，结果集合行数发生变化，是“出现了新的符合条件的记录”。

**采分点：**
- 每个场景都写出正确术语；
- 理由要和现象一一对应；
- 区分“同一行值变了”和“结果集合变了”。

### 第24题
**参考答案：**
可接受表述示例：

> 两段锁协议要求事务先进入加锁阶段，再进入解锁阶段；一旦开始释放锁，就不能再申请新锁。这样可以保证冲突操作的先后顺序与某个串行顺序一致，因此能保证可串行化调度。  
> 但 2PL 并不限制多个事务以相反顺序持有并等待对方所需的锁，所以仍可能形成循环等待，从而发生死锁。

**采分点：**
- 写出“加锁阶段 / 解锁阶段”；
- 写出“可串行化”；
- 写出“循环等待 / 死锁仍可能发生”。

### 第25题
**参考答案：**
1. 至少应设置为 **`Read Committed`（提交读）**；
2. 它至少能避免 **脏读**；
3. 在一般定义下，仍可能出现 **不可重复读**，也可能出现 **幻读**。

**采分点：**
- 隔离级别名称正确；
- 能指出避免脏读；
- 能指出并非最高隔离，仍可能有更高层面的并发现象。

### 第26题
**参考答案：**
- 场景 1：尚未提交的事务被中断，主要破坏 **原子性**，恢复时应执行 **Undo**；
- 场景 2：已提交事务的修改尚未写盘，主要破坏 **持久性**，恢复时应执行 **Redo**。

**采分点：**
- 原子性 ↔ Undo 配对正确；
- 持久性 ↔ Redo 配对正确；
- 不把两者反写。

### 第27题
**参考答案：**
1. 在日志中加入检查点，主要是为了 **提高系统故障恢复效率**；
2. 因为检查点以前已经稳定处理完的事务通常不再是恢复重点，而检查点之后仍活跃或发生更新的事务，更可能影响 `Redo / Undo` 列表，所以恢复分析会优先关注它们。

**采分点：**
- 写出“恢复效率”；
- 能说明检查点帮助缩小恢复分析范围。

### 第28题
**参考答案：**
- `T1`：进入 **Redo**。因为检查点后开始、写过数据，并在故障前提交；
- `T2`：进入 **Undo**。因为虽然检查点前已开始，但在检查点后继续写数据，且故障前未提交；
- `T3`：进入 **Undo**。因为检查点后开始、写过数据，但故障前未提交；
- `T4`：通常**不需要处理**。因为它在检查点前已提交，且不在检查点后的活跃事务列表中，通常可视为已稳定处理。

**采分点：**
- 正确区分已提交与未提交；
- 能说明检查点后活跃事务是重点分析对象；
- `T2` 虽然在检查点前开始，但未提交且检查点后仍有更新，仍需要 Undo。

### 第29题
**参考答案：**
1. **事务故障**：首要恢复动作是 **由 DBMS 回滚该事务**；  
2. **系统故障**：首要恢复动作是 **依据日志对未提交事务 Undo、对已提交但未持久化事务 Redo**；  
3. **介质故障**：首要恢复动作是 **先恢复备份，再结合日志进行重做恢复**。

**采分点：**
- 三类故障名称判断正确；
- 恢复动作与故障类型匹配；
- 不把系统故障和介质故障混淆。

### 第30题
**参考答案：**
1. 该调度可能形成死锁，因为：
   - `T1` 已持有 `A` 的排他锁，等待 `B`；
   - `T2` 已持有 `B` 的排他锁，等待 `A`；
   - 两个事务互相等待对方释放锁，形成**循环等待**。
2. 更安全的改写思路：
   - 让两个事务按**相同顺序申请锁**，例如都先申请 `A`，再申请 `B`；
   - 或者让其中一个事务在无法一次拿到所需锁时先回退重试；
   - 在满足 2PL 的前提下，统一锁申请顺序是最常见、最稳的防死锁思路。

**采分点：**
- 写出循环等待；
- 写出“统一锁申请顺序”这一核心改写思路；
- 保留 2PL 约束，不写成随意解锁后再加锁。
