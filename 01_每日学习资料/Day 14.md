### 2.3 今日核心知识框架

今天要固定下来的，是一套 **“事务并发控制题”标准分析框架**：

> **先识别事务与操作序列 → 再找冲突操作 → 再判断是否可串行化 / 是否出现并发异常 → 最后给出封锁或隔离级别改进方案。**

#### 1）什么是事务，为什么必须把一组操作看成一个整体

事务（Transaction）是数据库中的一个逻辑工作单位，通常由多条读写操作组成。考试里最常见的场景是：

- 转账：先扣款，再加款；
- 订单：先扣库存，再生成订单；
- 成绩修改：先更新成绩，再更新统计值。

这些操作必须作为一个整体来执行，否则只做一半就会出现错误结果。

#### 2）ACID 是事务题的底座，但不能只停留在背定义

- **原子性（Atomicity）**：事务中的操作要么全做，要么全不做
- **一致性（Consistency）**：事务执行前后都应保持数据库满足约束
- **隔离性（Isolation）**：多个事务并发执行时，彼此应尽量互不干扰
- **持久性（Durability）**：事务一旦提交，其结果应被永久保存

考试真正爱问的，不只是“ACID 是什么”，而是：
- 哪个异常是隔离性问题；
- 哪个场景体现原子性失败；
- 为什么提交后仍能恢复结果是持久性要求。

#### 3）什么叫并发，为什么并发不等于错误

多个事务在时间上交替执行，叫并发。并发本身不是坏事，它能：
- 提高系统吞吐量
- 提高资源利用率
- 缩短平均响应时间

问题不在“并发”，而在“**并发是否受控**”。

#### 4）冲突操作是判断调度题的核心入口

并发控制题最稳的起手，不是先猜异常，而是先找 **冲突操作**。  
若两个操作满足以下条件，就构成冲突：

1. 来自不同事务；
2. 作用于同一数据项；
3. 至少有一个是写操作。

所以：
- `R1(A)` 与 `R2(A)` 不冲突；
- `R1(A)` 与 `W2(A)` 冲突；
- `W1(A)` 与 `R2(A)` 冲突；
- `W1(A)` 与 `W2(A)` 冲突。

#### 5）可串行化判断的关键，是把“执行顺序关系”写出来

如果一个并发调度的效果等价于某个串行调度，就称该调度可串行化。  
软考里最常见的是 **冲突可串行化** 判断，稳妥流程是：

1. 找出所有冲突操作；
2. 根据冲突先后建立事务之间的有向边；
3. 画出优先图 / 前驱图；
4. 若图无环，则调度冲突可串行化；
5. 若图有环，则不可冲突串行化。

#### 6）并发异常要会区分，不要只会背名字

高频并发异常包括：

- **丢失修改**：一个事务的更新结果被另一个事务覆盖
- **脏读**：读到了另一个事务尚未提交的数据
- **不可重复读**：同一事务中两次读取同一数据，结果不同
- **幻读**：同一事务两次按条件查询，返回的记录条数不同

考试容易混淆的地方在于：
- 不可重复读关注的是“同一行值变了”；
- 幻读关注的是“满足条件的记录集合变了”。

#### 7）锁协议是“怎么防异常”的常见答法

如果题目问“如何改进”，最常见答案路径是：
- 使用共享锁（S 锁）和排它锁（X 锁）；
- 使用两段锁协议（2PL）；
- 提高隔离级别；
- 必要时串行化执行。

#### 8）两段锁协议为什么高频

两段锁协议（2PL）常考，因为它直接服务于“如何保证可串行化”。

其核心思想是：
- **扩展阶段**：只能加锁，不能解锁；
- **收缩阶段**：只能解锁，不能加锁。

如果所有事务都遵守两段锁协议，通常可以保证调度的冲突可串行化。

#### 9）隔离级别是“控制成本与正确性”的折中工具

常见隔离级别从低到高：
- 读未提交（Read Uncommitted）
- 读已提交（Read Committed）
- 可重复读（Repeatable Read）
- 串行化（Serializable）

级别越高，能避免的异常越多，但并发性通常越差。

#### 10）下午题真正考的是“判断依据写全”

并发题很少只给“判断对错”满分，通常都要写：
- 冲突在哪里；
- 异常发生在哪一步；
- 为什么可 / 不可串行化；
- 应采用什么改进手段。

所以 Day 14 的重点，不只是“我懂了”，而是：

> **我能按固定步骤把事务并发题写成可得分的答案。**

### 2.4 必备结论

- 事务是数据库中的逻辑工作单位，通常由一组读写操作构成
- ACID 中，并发异常最直接对应的是隔离性问题
- 两个操作构成冲突，必须满足：不同事务、同一数据项、至少一个写
- `R-R` 不冲突，`R-W`、`W-R`、`W-W` 都可能冲突
- 冲突可串行化判断，核心是看优先图是否有环
- 优先图无环，则调度可冲突串行化；有环则不可冲突串行化
- 脏读读到的是未提交数据，不可重复读读到的是同一行值变化，幻读读到的是结果集变化
- 两段锁协议常用于保证调度的冲突可串行化
- 共享锁适用于读，排它锁适用于写
- 锁兼容关系要记牢：`S-S` 兼容；`S-X`、`X-S`、`X-X` 都不兼容
- 题目若问“某事务能否继续执行”，本质上是在判断“申请的新锁是否与当前已有锁兼容”
- 同一事务若先读后写同一数据项，常见做法是先加 `S` 锁，再在需要修改时升级为 `X` 锁
- 隔离级别越高，能避免的异常越多，但并发能力通常越低

### 2.5 高频考点拆解

#### 2.5.1 冲突操作识别是所有调度题的起点

**核心概念：** 冲突操作是建立事务先后约束关系的基础。  
**常见考法：** 给出一串 `R1(A)、W2(A)、R1(B)` 之类的操作，要求判断哪些冲突。  
**高频误区：** 把 `R1(A)` 与 `R2(A)` 也当成冲突。  
**快速判断提示：** 至少有一个写，才值得继续看。  
**小例子：** `R1(A)` 与 `W2(A)` 冲突，但 `R1(A)` 与 `R2(A)` 不冲突。

#### 2.5.2 可串行化判断不要直接凭感觉下结论

**核心概念：** 并发调度可否视为某个串行顺序，要看冲突先后关系是否形成环。  
**常见考法：** 让你画优先图或直接判断调度是否冲突可串行化。  
**高频误区：** 看上去“挺整齐”就直接判可串行化。  
**快速判断提示：** 先列边，再看图，不要跳步骤。  
**小例子：** 若同时出现 `T1 → T2` 和 `T2 → T1`，通常就已经危险。

#### 2.5.3 并发异常最容易混在“读到变了”和“读到多了”之间

**核心概念：** 不可重复读与幻读都表现为“前后不一致”，但层级不同。  
**常见考法：** 给业务场景，让你判断属于哪种异常。  
**高频误区：** 把“记录数量变化”也写成不可重复读。  
**快速判断提示：** 看是“同一行值变了”，还是“满足条件的结果集变了”。  
**小例子：** 同一员工工资前后变了，偏向不可重复读；同一条件查询记录数前后变了，偏向幻读。

#### 2.5.4 两段锁协议常与“为什么能保证可串行化”一起考

**核心概念：** 2PL 通过“先加锁后解锁”的阶段约束保证调度的冲突可串行化。  
**常见考法：** 问某事务执行过程是否符合 2PL，或问为什么采用 2PL 更安全。  
**高频误区：** 把“解了一个锁后又去加新锁”也当成符合 2PL。  
**快速判断提示：** 一旦开始解锁，就不能再申请新锁。  
**小例子：** 事务先加 `S(A)`、再加 `X(B)`、再解 `A`、又去加 `S(C)`，这就不符合 2PL。

#### 2.5.5 隔离级别题要会“异常 ↔ 级别”双向联想

**核心概念：** 不同隔离级别能阻止的异常不同。  
**常见考法：** 给异常，让你选更合适的隔离级别；或给隔离级别，让你判断还能出现什么异常。  
**高频误区：** 以为读已提交就能解决所有读问题。  
**快速判断提示：** 级别越高，异常越少，但成本越高。  
**小例子：** 想彻底避免幻读，通常要想到串行化级别。


#### 2.5.6 冲突可串行化判定案例（必须会完整写）

**案例 A：单向边，无环，可冲突串行化**  
调度：
- `R1(A)`
- `W1(A)`
- `R2(A)`
- `W2(A)`
- `R1(B)`
- `W1(B)`
- `R2(B)`
- `W2(B)`

**判定过程：**
1. 先看数据项 `A`：`W1(A)` 先于 `R2(A)`、`W2(A)`，形成 `T1 → T2`；  
2. 再看数据项 `B`：`W1(B)` 先于 `R2(B)`、`W2(B)`，仍形成 `T1 → T2`；  
3. 优先图中只有 `T1 → T2`，没有反向边，也没有环；  
4. 结论：该调度 **冲突可串行化**，等价串行顺序为 **`T1 → T2`**。

**本题提醒：**
- 同方向边即使出现多次，本质上仍是一条先后约束；
- 只要优先图无环，就不要被“操作交叉很多”吓到。

**案例 B：出现反向边，有环，不可冲突串行化**  
调度：
- `W1(A)`
- `R2(A)`
- `W2(B)`
- `R1(B)`

**判定过程：**
1. 在数据项 `A` 上，`W1(A)` 先于 `R2(A)`，形成 `T1 → T2`；  
2. 在数据项 `B` 上，`W2(B)` 先于 `R1(B)`，形成 `T2 → T1`；  
3. 优先图中同时存在 `T1 → T2` 和 `T2 → T1`，构成环；  
4. 结论：该调度 **不可冲突串行化**。

**本题提醒：**
- 这类题最关键的是能不能找出“反向边”；
- 一旦形成环，就不能再写等价串行顺序。

**案例 C：先剔除非冲突，再判边方向**  
调度：
- `R1(A)`
- `R2(A)`
- `W2(A)`
- `R1(B)`
- `W2(B)`

**判定过程：**
1. `R1(A)` 与 `R2(A)` 虽然访问同一数据项 `A`，但都为读操作，因此 **不冲突**；  
2. `R1(A)` 先于 `W2(A)`，形成 `T1 → T2`；  
3. `R1(B)` 先于 `W2(B)`，仍形成 `T1 → T2`；  
4. 优先图中只有 `T1 → T2`，无环，因此该调度 **冲突可串行化**，等价串行顺序为 **`T1 → T2`**。

**本题提醒：**
- 先删掉 `R-R` 这类非冲突对，能明显降低误判率；
- “同一数据项”只是第一步，还必须再检查“至少一个写”。

#### 2.5.7 共享锁与排它锁判定案例（必须会判断“能否继续执行”）

先记兼容关系：

| 当前已有锁 | 新申请 S 锁 | 新申请 X 锁 |
|---|---|---|
| S 锁 | 兼容 | 不兼容 |
| X 锁 | 不兼容 | 不兼容 |

**案例 A：读 - 读并发，可同时执行**  
场景：
- `T1` 读取 `A`
- `T2` 也读取 `A`

**判定：**
- `T1` 对 `A` 加 **共享锁 `S(A)`**
- `T2` 也申请 **共享锁 `S(A)`**
- 因为 `S-S` 兼容，所以二者 **可以同时读，不必等待**

**案例 B：先读后写，写方等待**  
场景：
- `T1` 正在读取 `A`
- `T2` 想修改 `A`

**判定：**
- `T1` 先持有 `S(A)`
- `T2` 修改 `A` 需要申请 `X(A)`
- 因为 `S-X` 不兼容，所以 `T2` **必须等待 `T1` 释放锁后才能写**

**案例 C：先写后读，读方等待**  
场景：
- `T1` 正在修改 `A`
- `T2` 想读取 `A`

**判定：**
- `T1` 持有 `X(A)`
- `T2` 读取 `A` 需要申请 `S(A)`
- 因为 `X-S` 不兼容，所以 `T2` **不能立即读，只能等待**

**案例 D：写 - 写冲突，后写等待**  
场景：
- `T1` 想修改 `A`
- `T2` 也想修改 `A`

**判定：**
- `T1` 先持有 `X(A)`
- `T2` 再申请 `X(A)`
- 因为 `X-X` 不兼容，所以 `T2` **必须等待**

**案例 E：先读后写同一数据项，常见锁升级题**  
场景：
- `T1` 先读取 `A`
- 随后 `T1` 还要修改 `A`

**判定：**
- 读取时通常先加 `S(A)`
- 需要写时，应把锁升级为 `X(A)`
- 若此时没有其他事务持有 `A` 的共享锁，升级可成功；  
- 若此时还有别的事务持有 `S(A)`，则升级可能需要等待

**这类题的稳妥答法：**
1. 先看动作是读还是写；  
2. 读通常配 `S` 锁，写通常配 `X` 锁；  
3. 再看已有锁与新锁是否兼容；  
4. 兼容则可继续执行，不兼容则阻塞 / 等待；  
5. 最后再补一句“这样做是为了避免读写冲突或写写冲突”。

### 2.9 易混点速记卡片

- **冲突操作**：同一数据项 + 不同事务 + 至少一个写  
  **非冲突操作**：读读通常不算冲突

- **脏读**：读到别人未提交  
  **不可重复读**：同一行前后读到不一样  
  **幻读**：同一条件查出的记录集合变了

- **可串行化**：等价于某个串行顺序  
  **不可串行化**：优先图形成环

- **2PL**：先加锁后解锁  
  **违反 2PL**：一边解锁，一边又申请新锁

- **共享锁 S**：读  
  **排它锁 X**：写
