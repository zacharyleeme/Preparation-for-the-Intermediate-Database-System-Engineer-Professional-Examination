### 2.3 今日核心知识框架

今天要固定下来的，不是一堆零散概念，而是 4 套标准路径：

#### 1）安全设计题标准路径

> **保护对象与风险 → 最小权限 / 角色 / 视图 / 审计 → 为什么有效 → 代价与结论**

关键提醒：
- 先写“工资、身份证号、手机号、敏感明细”等保护对象；
- 再写“越权访问、敏感列暴露、误改、合规风险”等风险；
- 然后再落“只读角色、汇总视图、细粒度授权、审计留痕”等动作。

#### 2）备份恢复题标准路径

> **业务要求 → RPO / RTO → 全量 + 高频增量 / 日志类保护 → 异地保存 + 加密 + 恢复演练 → 成本与权衡**

关键提醒：
- 先定恢复目标，不要一上来就背全量、增量、差异；
- RPO / RTO 缩写和口径必须写稳；
- 异地保存和恢复演练是区分“背知识点”和“会写方案题”的关键。

#### 3）多告警运维分析题标准路径

> **先排优先级 → 再看关联关系 → 再定应急动作 → 最后写长期加固**

关键提醒：
- 失败登录激增优先想到**潜在安全事件**；
- 备份目录空间不足优先想到**备份任务可用性风险**；
- 慢 SQL、磁盘、异常访问之间可能是联动关系，不要逐条割裂解释。

#### 4）数据库保障类综合题标准路径

> **目标与风险 → 安全 / 备份 / 运维手段 → 具体作用 → 代价、边界与综合权衡**

关键提醒：
- 不要只写“采用安全措施、备份措施、运维措施”；
- 要具体展开到：**角色 / 权限、视图、审计、备份、异地保存、恢复演练、监控告警**；
- 结尾必须有一句完整的综合收束句。

### 2.4 必备结论

- 安全设计题先识别保护对象和风险，再决定权限与控制手段。
- 最小权限原则强调“只给完成工作所必需的最小权限”。
- 角色解决“谁拥有什么权限”，视图解决“能看到什么数据”。
- 审计主要用于追踪关键访问、权限变更与异常操作，不能直接替代权限控制。
- RPO 约束可容忍的数据丢失时间窗口，RTO 约束可容忍的服务中断时间上限。
- 备份不等于高可用，前者偏数据恢复，后者偏服务连续性。
- 异地备份的核心价值之一，是降低单点环境故障导致生产数据与备份同时失效的风险。
- 恢复演练的核心价值之一，是验证备份链条、恢复步骤和恢复时间是否真实可达。
- 多告警并发时，先识别高风险事项，再判断各类告警之间的联动关系。
- 综合题不只考“列手段”，更考“为什么有效、代价是什么、结论怎么收住”。

### 2.5 高频考点拆解

#### 2.5.1 角色与视图要分工清楚

**核心概念：** 角色是权限集合，视图是受控的数据展现范围。  
**常见考法：** “为什么不能直接给基础表 SELECT？”  
**高频误区：** 把角色和视图混成一件事，只写其中一个。  
**快速判断提示：** 题目里有“只能看汇总、不能看明细、不能看敏感列”，视图几乎一定要出现。  
**小例子：** 给经营分析人员授予只读角色，并只允许查询部门汇总视图。

#### 2.5.2 RPO / RTO 不是背名词，而是反推方案约束

**核心概念：** RPO 决定数据最多能丢多久，RTO 决定业务最多能停多久。  
**常见考法：** 给 15 分钟 / 2 小时这类业务条件，要求分析方案。  
**高频误区：** 写成 PTO，或把 RPO 说成“数据量”而不是“时间窗口”。  
**快速判断提示：** 只要题目写“最多丢多久、最多停多久”，就必须把 RPO / RTO 显式写出来。  
**小例子：** 最多丢 15 分钟数据，通常不能只靠低频全量备份。

#### 2.5.3 异地备份与恢复演练是完整性答案的常见补分点

**核心概念：** 有备份文件不等于真能恢复。  
**常见考法：** “为什么有备份还不够？”  
**高频误区：** 只列全量 / 增量，不写异地、加密、演练。  
**快速判断提示：** 只要是方案题，就要检查自己有没有漏这两层。  
**小例子：** 备份都放同一块磁盘上，磁盘损坏时生产库与备份可能一起丢。

#### 2.5.4 多告警题的第一步是“先后顺序”

**核心概念：** 安全事件、备份风险、性能波动、容量压力可能互相牵动。  
**常见考法：** 给出失败登录、慢 SQL、磁盘告警、备份任务等多种现象。  
**高频误区：** 直接写扩容、加索引、查 CPU，而不先写风险优先级。  
**快速判断提示：** 先写“安全事件 / 备份可用性优先”，再写性能关联。  
**小例子：** 失败登录激增可能代表撞库或暴力尝试，而磁盘不足会直接影响即将执行的备份任务。

### 2.6 下午题视角下的题型映射

### 2.7 标准答题模板 / 采分点提示

#### 模板1：安全设计题
1. 先写要保护的对象，如工资、身份证号、手机号等敏感数据。  
2. 再写主要风险，如越权访问、敏感列暴露、误改或合规风险。  
3. 接着写控制手段：最小权限、只读角色、汇总视图、审计。  
4. 再说明这些手段如何限制可见范围并保留访问痕迹。  
5. 最后补一句：方案能提升安全性，但也会增加权限管理、视图维护和审计存储成本。

#### 模板2：备份恢复方案题
1. 先写业务允许的最大数据丢失时间与最大中断时间。  
2. 再明确 RPO 与 RTO，并说明分别约束什么。  
3. 接着给出方案：全量备份 + 高频增量 / 日志类保护 + 异地保存 + 加密保护。  
4. 再写恢复演练的意义：验证备份链条、恢复步骤与恢复时间。  
5. 最后补一句：方案能更好满足恢复目标，但会增加存储、管理与演练成本。

#### 模板3：多告警运维分析题
1. 先写高优先级风险：安全事件与备份可用性。  
2. 再写容量、慢 SQL、异常访问之间可能互相关联。  
3. 接着给出先做什么：确认安全风险、保障备份任务可执行、再看性能与资源。  
4. 最后写长期措施：审计、容量监控、性能监控、备份监控联动分析。

#### 模板4：数据库保障类综合题
1. 先识别业务目标与主要风险。  
2. 再选择至少两类以上具体手段，如角色 / 权限、视图、审计、备份、异地保存、恢复演练、监控告警。  
3. 再说明这些手段分别如何降低越权访问、缩小数据损失、提升恢复能力与增强运维可见性。  
4. 最后补一句完整收束：方案能提升数据库安全性、可恢复性与运维稳定性，但会增加管理、存储与实施复杂度，需要结合业务重要性综合权衡。

**解析：** 2019 上午真题对应考点。系统故障恢复通常需要结合日志做 UNDO 与 REDO。

### 第11题
**答案：B**  
**解析：** 2019 上午真题对应考点。审计主要用于记录和追踪，不能直接阻止一切非法修改的发生。

### 第12题
**答案：B**  
**解析：** 备份保障数据可恢复，高可用保障服务连续性，两者互补而非替代。

### 第13题

### 第14题

### 第18题
**答案：B**  
**解析：** 多告警并发时，更稳妥的起手方式是先识别安全事件与备份可用性风险，再看性能与资源关联。

### 第19题
**参考答案：**  
这个场景下，至少应先识别两类安全目标：一是保护工资、身份证号、手机号等敏感数据的**保密性**；二是防止经营分析人员**越权访问或误用不应开放的基础明细**。主要风险包括：若直接开放基础表，可能导致敏感列暴露；若权限过宽，还可能带来越权查询、误用数据和合规风险。整体保护边界应是：**只开放部门汇总结果，不开放敏感明细。**

**采分点：**
- 写出保密性 / 敏感数据保护目标
- 写出越权访问、敏感列暴露等风险
- 写出“只开放汇总、不开放明细”的边界

### 第20题
**参考答案：**  
更合理的做法，是基于最小权限原则为经营分析人员建立独立的只读角色；再通过汇总视图仅开放部门销售统计结果，不直接开放基础明细表；授权时仅授予该角色对汇总视图的查询权限，不授予对敏感明细表的直接查询或修改权限；同时对关键访问和权限变更保留审计记录。这样能够把“谁能查、能查什么、是否留痕”分层控制。

**采分点：**
- 最小权限
- 只读角色
- 汇总视图
- 审计

### 第21题
**参考答案：**  
这种做法通常不稳妥，因为它把安全控制寄托在制度约束上，而不是数据库层面的技术约束。若直接授予基础表 `SELECT` 权限，经营分析人员理论上就能接触超出业务需要的敏感列，违背最小权限原则。一旦发生误查、滥用或账户被冒用，风险会被放大。因此，更稳妥的做法是通过角色、视图和细粒度授权，在数据库层面直接收紧可见范围。

**采分点：**
- 制度约束不能替代技术约束
- 违背最小权限原则
- 基础表直接开放会扩大敏感信息暴露风险
- 回到角色 / 视图 / 细粒度授权的纠偏动作

### 第22题
**参考答案：**  
当前场景中需要重点保护的是工资、身份证号、手机号等敏感数据，主要风险是经营分析人员越权查看基础明细。更合理的设计应基于最小权限原则，为分析人员建立只读角色，并通过汇总视图仅开放部门销售统计结果，同时保留审计记录。这样既能满足业务分析需求，又能避免敏感列直接暴露并保留关键访问痕迹。需要注意的是，这类分层控制虽然能显著提升安全性，但也会增加权限管理、视图维护和审计存储成本。总体上，应在满足业务需求的前提下，优先保障敏感数据的安全边界。

**采分点：**
- 顺序完整：对象 / 风险 → 手段 → 原因 → 代价 / 结论
- 写出最小权限、角色、视图、审计
- 有综合收束句

### 第23题
**参考答案：**  
该场景中的 **RPO** 是“最多只能丢失 15 分钟数据”，它约束的是**数据丢失容忍窗口**；**RTO** 是“故障后最多允许业务中断 2 小时”，它约束的是**系统恢复服务的时间上限**。也就是说，方案设计既要考虑如何缩小数据损失，也要考虑恢复步骤不能过慢。

**采分点：**
- RPO = 15 分钟数据丢失容忍
- RTO = 2 小时业务中断容忍
- 写出二者分别约束什么

### 第24题
**参考答案：**  
更稳妥的思路是：定期做全量备份，并结合更高频的增量类或日志类保护手段，以满足较小的数据丢失窗口要求；备份文件应进行加密，并至少保留一份异地副本，以降低本地故障或泄露风险；同时要定期进行恢复演练，验证备份是否真的可恢复。整体上，这样的组合更有利于同时满足恢复目标与安全要求。

**采分点：**
- 全量备份
- 高频增量 / 日志类保护
- 异地保存
- 加密保护
- 恢复演练

### 第25题
**参考答案：**  
“已经做了备份”仍然不够，因为本地备份可能在同一场硬件故障、误删、勒索或机房事故中一起失效。异地保存的价值，在于降低单点环境同时损坏生产数据和备份的风险。恢复演练的价值，则在于验证备份链条、恢复步骤和恢复时间是否真的可执行，而不是只停留在“有备份文件”的层面。

**采分点：**
- 本地备份可能同灾失效
- 异地保存的意义
- 恢复演练用于验证可恢复性、恢复步骤与恢复时间

### 第26题
**参考答案：**  
该订单系统最多只能丢失 15 分钟数据、最多中断 2 小时，因此方案设计必须先围绕 RPO 和 RTO 展开。可采用周期性全量备份，并结合更高频的增量类或日志类保护，以缩小数据损失窗口。备份文件还应加密并保留异地副本，同时定期进行恢复演练，验证恢复链条与时效是否达标。这样做既有利于满足恢复目标，也能降低本地故障和备份泄露风险。需要注意的是，这类方案会带来额外的存储、管理与演练成本，应结合业务重要性综合权衡。

**采分点：**
- 顺序完整：业务要求 → 目标 → 方案 → 验证 / 保护 → 代价 / 结论
- 写出 RPO / RTO 反推方案
- 写出异地、加密、演练
- 有成本与权衡收束句

### 第27题
**参考答案：**  
这组现象中，优先应关注两类高风险问题：第一，失败登录激增可能代表安全事件正在发生；第二，备份目录磁盘即将打满，可能直接影响两小时后的备份任务执行。与此同时，慢 SQL 增多也可能与资源紧张、异常访问或磁盘压力有关。分析时应先保护安全与备份可用性，再结合性能告警判断这些问题是否存在关联，避免只盯某一个告警。

**采分点：**
- 安全事件风险
- 备份任务受磁盘影响的风险
- 慢 SQL 与其他问题的关联
- 先后优先级

### 第28题
**参考答案：**  
安全审计、容量监控、性能监控和备份任务监控不能割裂，是因为数据库问题往往不是单一维度发生的。失败登录激增可能代表异常访问，磁盘空间不足可能导致备份失败，也可能进一步影响数据库写入与性能；慢 SQL 增多又可能与资源压力或异常负载有关。只有把这些监控放在一起分析，才能更快识别根因、判断影响范围并制定更合理的应急与加固措施。

**采分点：**
- 多维度告警可能相互影响
- 磁盘不足可能影响备份与性能
- 联动分析更利于找根因、判断影响范围与制定措施

### 第29题
**参考答案：**  
有备份并不等于系统一定能快速继续对外提供服务。备份主要解决的是故障后如何把数据恢复回来，而主备或其他高可用措施更关注的是在故障发生后如何尽量缩短业务中断时间。若业务对停机时间要求严格，仅靠备份恢复往往不够。因此，备份和高可用通常是两条互补能力：一个保障数据可恢复，一个保障服务连续性。

**采分点：**
- 备份解决数据恢复
- 高可用解决服务连续性 / 缩短中断
- 写出二者互补而非替代

### 第30题
**参考答案：**  
回答数据库保障类综合题时，首先应识别业务目标和主要风险，例如敏感数据泄露、服务中断或恢复失败。其次根据风险选择角色与权限、视图、审计、备份、异地保存、恢复演练、监控告警等合适手段。然后说明这些手段分别如何降低越权访问、缩小数据损失、提升恢复能力和增强运维可见性。最后补充代价、边界与结论：该方案能提升数据库安全性、可恢复性与运维稳定性，但也会增加管理、存储与实施复杂度，需要结合业务重要性综合权衡。

**采分点：**
- 顺序完整：目标 / 风险 → 手段 → 原因 → 代价 / 边界 / 结论
- 至少两类以上具体手段
- 有完整综合收束句
