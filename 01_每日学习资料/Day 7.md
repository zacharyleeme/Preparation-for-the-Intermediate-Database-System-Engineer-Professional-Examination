### 2.2 为什么 Day 7 进入 SQL 综合训练与常见写法对比

Day 3 学的是单表基础查询，Day 4 学的是聚合与分组，Day 5 学的是连接查询，Day 6 学的是子查询。  
到这一步，如果还只是把每种写法分开记，做题时就很容易出现下面这些问题：

- 明明题目只考分组后筛选，却误写成 `WHERE AVG(...)`
- 明明题目更适合连接，却机械写成子查询
- 明明题目问的是“有没有记录”，却误用统计函数
- 明明题目要的是“每门课选课人数”，却没先看清输出粒度

真实考试中的 SQL 题，经常不是单独考一个点，而是会把：

- 表连接
- 条件筛选
- 分组统计
- 子查询
- 存在性判断

放在同一题里一起考。

所以 Day 7 的目标，不是再学一个新零件，而是开始练会：

> **看到题目后，先判断它到底属于哪种查询任务，再选择最合适的 SQL 组织方式。**

原因主要有三点：

1. **综合 SQL 题是上午选择题和下午分析题都很常见的高频场景**
2. **考试不只考你记不记得语法，还考你能否判断“该用哪种写法”**
3. **把连接、分组、子查询放到同一框架下理解，后续做题才会稳定**

简单说：

- Day 3：学会从一张表里查数据
- Day 4：学会做统计与分组
- Day 5：学会多表连接
- Day 6：学会用内层查询提供中间结果
- Day 7：学会把这些能力**组合起来解同一题**

### 2.3 今日核心知识框架

综合 SQL 题最容易乱的地方，不是语法本身，而是**下手顺序**。  
建议固定按下面 7 步来想：

#### 第一步：先看题目到底要什么结果
先问自己：

1. 题目最后要输出什么字段？
2. 输出结果是一条条明细，还是“每组一条”？
3. 题目关心的是“有没有”，还是“平均值 / 总数 / 最大值”？

例如：

- “查询学生姓名及成绩” → 明细查询
- “查询每门课平均成绩” → 分组统计
- “查询没有选课的学生姓名” → 存在性判断
- “查询成绩高于平均成绩的记录” → 中间结果联动 + 子查询

#### 第二步：判断需要哪些表
如果输出字段分散在不同表中，就要优先考虑连接。

例如：

- Student(学号, 姓名)
- SC(学号, 课程号, 成绩)
- Course(课程号, 课程名)

若题目要“学生姓名 + 课程名 + 成绩”，那就至少涉及：
- Student
- SC
- Course

#### 第三步：判断是先筛选、先连接，还是先分组
一般可先这样记：

- **普通条件筛选** → `WHERE`
- **连接关系** → `JOIN ... ON ...`
- **按组统计** → `GROUP BY`
- **对统计结果再筛选** → `HAVING`

例如：

```sql
SELECT sc.课程号, AVG(sc.成绩)
FROM SC sc
WHERE sc.成绩 IS NOT NULL
GROUP BY sc.课程号
HAVING AVG(sc.成绩) >= 80;
```

可按顺序理解为：

1. 先从 SC 表取数据
2. 先筛掉不需要的原始记录（`WHERE`）
3. 再按课程号分组（`GROUP BY`）
4. 再对平均成绩这个统计结果做筛选（`HAVING`）

#### 第四步：判断题目更像连接，还是更像子查询
有些题两种写法都能做，但思维入口不同。

例如“查询选修了课程 C1 的学生姓名”：

**连接写法：**
```sql
SELECT s.姓名
FROM Student s
JOIN SC sc ON s.学号 = sc.学号
WHERE sc.课程号 = 'C1';
```

**子查询写法：**
```sql
SELECT 姓名
FROM Student
WHERE 学号 IN (
    SELECT 学号
    FROM SC
    WHERE 课程号 = 'C1'
);
```

直觉区别：
- 连接：直接把表拼起来查
- 子查询：先查学号集合，再去外层查姓名

#### 第五步：判断题目是“值判断”“集合判断”还是“存在性判断”
这一步是 Day 6 错题最需要带进 Day 7 的重点。

- **值判断**：一个值和另一个值比较 → 常见 `=`、`>`、`<`
- **集合判断**：某个值是否属于一组值 → 常见 `IN`
- **存在性判断**：是否存在满足条件的记录 → 常见 `EXISTS / NOT EXISTS`

例如：

```sql
SELECT 姓名
FROM Student s
WHERE NOT EXISTS (
    SELECT *
    FROM SC sc
    WHERE sc.学号 = s.学号
);
```

这里不是在数记录，也不是在比较平均值，而是在问：

> 这个学生有没有对应的选课记录？

#### 第六步：判断题目是否需要“分组后筛选”
这是最常见综合错误点之一。

- `WHERE`：筛原始记录
- `HAVING`：筛分组结果

例如：

- “查询成绩大于 80 的记录” → `WHERE 成绩 > 80`
- “查询平均成绩大于 80 的课程” → `GROUP BY 课程号 HAVING AVG(成绩) > 80`

不要把这两类题混在一起。

#### 第七步：最后再检查输出粒度是否正确
这是综合 SQL 题最容易被忽略的一步。

例如题目是：

> 查询每门课的平均成绩

那最终结果应该是：
- 每门课一行
- 而不是每条成绩一行

所以写完 SQL 后，必须再反问自己：

1. 结果是不是“每组一条”？
2. 结果字段是否和题目要求一致？
3. 有没有多查、少查、错分组？

### 2.4 必备结论

- 做综合 SQL 题时，先看**题目要什么结果**，再看语法怎么写
- 输出字段分散在多表时，优先想到连接
- 题目要“每组一个结果”时，优先检查是否要 `GROUP BY`
- `WHERE` 用于筛原始记录，`HAVING` 用于筛分组统计结果
- “有没有 / 是否存在 / 是否没有”优先想到 `EXISTS / NOT EXISTS`
- 内层返回多个值时，通常更适合 `IN`
- 连接和子查询经常可以表达同一逻辑，但思路入口不同
- 做综合题时，不要只盯语法，要盯“明细 / 分组 / 存在 / 集合 / 统计值”这几个任务类型
- 单选题必须排除干扰项，不能因为包含正确方向就多选
- SQL 写完后要回头检查结果粒度是否与题目一致

### 2.5 高频考点拆解

#### 2.5.1 先判断“结果粒度”比先背语法更重要

所谓结果粒度，可以先理解成：

> 这道题最后是一条条明细，还是“每组只保留一条结果”？

例如：

- “查询学生姓名及成绩” → 明细粒度
- “查询每门课平均成绩” → 分组粒度
- “查询每个系选课人数” → 分组粒度

高频误区：
- 题目明明问“每门课”，却没有分组
- 题目明明是明细查询，却误用了聚合函数

#### 2.5.2 `WHERE` 与 `HAVING` 是综合题高频陷阱

看下面两句话：

- 查询成绩大于 80 的记录
- 查询平均成绩大于 80 的课程

前者是在筛**原始记录**，用 `WHERE`。  
后者是在筛**分组统计结果**，用 `HAVING`。

例如：

```sql
SELECT 课程号, AVG(成绩)
FROM SC
GROUP BY 课程号
HAVING AVG(成绩) > 80;
```

高频误区：
- 把 `AVG(成绩) > 80` 写进 `WHERE`
- 没分清“题目是在筛记录，还是筛统计结果”

#### 2.5.3 连接与子查询经常是“不同写法表达同一逻辑”

例如：

> 查询选修了 C1 的学生姓名

这题既能用连接，也能用子查询。

高频判断思路：
- 若想直接把表连起来、一次拿字段 → 连接
- 若想先查出学号集合、再去外层取姓名 → 子查询

高频误区：
- 把“能用子查询”理解成“只能用子查询”
- 只会背某一种写法，换种表达就不会识别

#### 2.5.4 `IN`、`EXISTS`、`NOT EXISTS` 要按题意分流

这三个很容易混，但关注点不同：

- `IN`：值是否属于集合
- `EXISTS`：是否存在满足条件的记录
- `NOT EXISTS`：是否不存在满足条件的记录

例如：

- “查询选修了 C1 的学生” → 可考虑 `IN`
- “查询有选课记录的学生” → 可考虑 `EXISTS`
- “查询没有选课记录的学生” → 可考虑 `NOT EXISTS`

高频误区：
- 题目问存在性，却误写成统计或集合比较
- 题目明明问集合成员关系，却误用 `EXISTS`

#### 2.5.5 综合题常见组织顺序

很多选择题其实在考你有没有这条固定顺序：

1. 先看题目输出什么
2. 再看需要哪些表
3. 再看是否要连接
4. 再看是否要分组
5. 再看是否要子查询/存在性判断
6. 最后检查条件写在 `WHERE` 还是 `HAVING`

高频误区：
- 一上来就机械写 SQL
- 还没搞清题意，就先套模板

#### 2.5.6 综合 SQL 最后一定要做“结果检查”

SQL 写完并不代表题就做完了。  
最后必须检查：

- 字段对不对
- 行数粒度对不对
- 有没有遗漏必要条件
- 分组是否过多或过少
- 单选题是否排除了所有干扰项

这一步特别像考试中的“验算”。  
很多题不是不会写，而是写得“差一点”。

### 2.6 与前序知识的映射或衔接

Day 7 不是一个全新的独立主题，而是前 4 天 SQL 主线的综合汇总：

- Day 3：给你查询骨架 `SELECT / FROM / WHERE`
- Day 4：给你统计与分组 `GROUP BY / HAVING / COUNT / AVG`
- Day 5：给你多表联动 `JOIN / ON`
- Day 6：给你中间结果联动 `IN / EXISTS / 子查询`
- Day 7：要求你把它们放到同一题里一起判断

可以这样理解：

- **连接查询**：横向把表拼起来
- **分组统计**：把结果按组压缩
- **子查询**：纵向引入中间结果
- **综合 SQL**：根据题意决定这三者怎么组合

### 2.7 易混点速记卡片

- **明细查询**
  - 输出一条条记录
  - 先想字段和表

- **分组统计**
  - 结果常是“每组一条”
  - 先想 `GROUP BY`

- **`WHERE`**
  - 筛原始记录

- **`HAVING`**
  - 筛分组后的统计结果

- **连接**
  - 多表直接关联

- **子查询**
  - 先出中间结果再继续查

- **`IN`**
  - 值属于集合

- **`EXISTS / NOT EXISTS`**
  - 关注是否存在记录
