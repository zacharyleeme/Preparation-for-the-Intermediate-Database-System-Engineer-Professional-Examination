### 2.3 今日核心知识框架

今天要固定下来的，是一套 **“日志恢复题标准分析框架”**：

> **先判故障类型 → 再看日志与事务状态 → 再判断 Undo / Redo → 最后写恢复依据与步骤。**

#### 1）为什么数据库恢复离不开日志

日志（Log）记录了事务执行过程中发生过的关键操作，是数据库进行恢复的依据。  
常见日志记录会包含：

- `START T`：事务开始
- `T 对某数据项的修改记录`：常写成旧值 / 新值形式
- `COMMIT T`：事务提交
- `ABORT T`：事务回滚
- `CHECKPOINT`：检查点

没有日志，系统发生故障后就难以判断：
- 哪些事务已经正式提交；
- 哪些事务尚未完成；
- 哪些修改应该保留；
- 哪些修改应该撤销。

#### 2）恢复题第一步要先判故障类型

恢复题最稳妥的起手，不是直接说 Undo / Redo，而是先区分故障场景。常见故障类型有：

- **事务故障**：某个事务执行失败，需要回滚该事务
- **系统故障**：如突然断电、系统崩溃，内存内容丢失，但外存上的数据库和日志通常还在
- **介质故障**：如磁盘损坏，数据文件本身受损，需要借助备份 + 日志恢复

不同故障，恢复处理的重点不同。

#### 3）Undo 与 Redo 分别解决什么问题

- **Undo（撤销）**：把**未完成 / 未提交事务**已经写入数据库的数据改回去，解决“本不该留下却留下了”的问题
- **Redo（重做）**：把**已提交事务**应当保留但可能尚未真正落盘的数据再做一遍，解决“本该留下却可能没留下”的问题

这两句话是恢复题的核心底座：

> **未提交看 Undo；已提交看 Redo。**

#### 4）立即修改与延迟修改，是恢复判断题的核心分叉点

**立即修改（Immediate Update）** 的特点是：
- 事务尚未提交时，修改后的数据也可能已经写入数据库
- 因此系统崩溃后，既可能要 **Undo 未提交事务**，也可能要 **Redo 已提交事务**

**延迟修改（Deferred Update）** 的特点是：
- 事务未提交前，修改结果通常不真正写回数据库
- 因此崩溃后，对未提交事务一般**不需要 Undo**
- 但对已提交、尚未真正反映到数据库的事务，仍可能需要 **Redo**

#### 5）检查点的作用，不是“多记一个日志名词”，而是缩小恢复范围

检查点（Checkpoint）的核心作用是：
- 把某个时间点之前的恢复信息“阶段性固定下来”
- 使系统故障后不必从日志最开头一直扫描到最后
- 从而缩小恢复分析范围，提高恢复效率

考试里如果问“为什么设置检查点”，最稳妥的回答方向是：
- **缩短恢复时间**
- **减少需要扫描的日志范围**
- **帮助更快确定需要 Undo / Redo 的事务集合**

#### 6）WAL 原则是“日志先于数据”

WAL（Write-Ahead Logging，预写式日志）的基本要求是：

1. 某数据页写回数据库之前，与它对应的日志记录必须先写入稳定存储；
2. 事务提交前，表示该事务提交的日志记录必须先可靠写入。

这条原则的意义在于：
- 即使系统崩溃，恢复系统仍有日志可查；
- 否则数据已经改了，但日志没留下，恢复就失去依据。

#### 7）恢复分析题的稳妥顺序

遇到恢复题，不要一上来凭感觉说 Undo / Redo。更稳的顺序是：

1. 先看故障属于事务故障、系统故障还是介质故障；
2. 再看题目给出的日志片段里，哪些事务有 `COMMIT`，哪些没有；
3. 若题目涉及立即修改 / 延迟修改，先明确采用的是哪种更新策略；
4. 然后判断：
   - 未提交事务是否要 Undo；
   - 已提交事务是否要 Redo；
5. 若题目涉及检查点，再说明从哪里开始分析更合理；
6. 最后用 2～4 句把恢复结论写完整。

#### 8）介质故障与系统故障不要混

- **系统故障**：更多是“内存丢了，但外存数据和日志还在”，恢复常依赖日志
- **介质故障**：更多是“数据库文件本身坏了”，恢复通常要先用备份重装，再结合日志把备份之后的已提交操作补回来

#### 9）下午题真正考的是“依据是否写清”

恢复题常见失分点并不是完全不会，而是：
- 只写“需要撤销 / 需要重做”，不写为什么；
- 只写“设置检查点提高效率”，不写提高在哪里；
- 只写“日志先写”，不写“先于谁写、为什么必须先写”。

所以 Day 15 的重点，不只是“我知道 Undo / Redo 是什么”，而是：

> **我能把恢复题按固定步骤写成能得分的答案。**

### 2.4 必备结论

- 日志是数据库恢复的重要依据，用于记录事务执行过程中的关键操作
- 恢复题第一步先判故障类型，再谈恢复动作
- Undo 用于撤销未提交事务造成的影响
- Redo 用于重做已提交事务应保留但可能尚未真正写入数据库的结果
- 立即修改下，系统故障后常常既要 Undo 未提交事务，也要 Redo 已提交事务
- 延迟修改下，未提交事务通常不必 Undo，但已提交事务仍可能需要 Redo
- 检查点的核心作用是缩小恢复分析范围、缩短恢复时间
- WAL 的核心思想是：日志必须先于对应数据写入稳定存储
- 系统故障通常依赖日志恢复；介质故障通常要结合备份与日志恢复
- 下午题写恢复结论时，必须把“事务状态 + 恢复动作 + 原因”写完整

### 2.5 高频考点拆解

#### 2.5.1 Undo / Redo 判断是恢复题的第一高频点

**核心概念：** Undo 撤销不该留下的结果，Redo 补回本该留下的结果。  
**常见考法：** 给出日志片段，问哪些事务需要 Undo、哪些需要 Redo。  
**高频误区：** 把“已提交事务”也写成 Undo，或把“未提交事务”也写成 Redo。  
**快速判断提示：** 先看有没有 `COMMIT`，再说动作。  
**小例子：** `T1` 有 `COMMIT`，`T2` 没有 `COMMIT`，系统崩溃后通常写成“Redo T1，Undo T2”（在立即修改场景下）。

#### 2.5.2 立即修改与延迟修改最容易混

**核心概念：** 两种更新策略会直接影响恢复动作。  
**常见考法：** 问“某更新策略下是否需要 Undo / Redo”。  
**高频误区：** 一看到恢复就机械写“都要 Undo 和 Redo”。  
**快速判断提示：** 先问自己：未提交事务的数据，提交前有没有可能已经写进数据库？  
**小例子：** 延迟修改下，未提交事务通常还没真正写进数据库，所以一般不需要 Undo。

#### 2.5.3 检查点的作用不能只答“提高效率”

**核心概念：** 检查点帮助缩小恢复范围。  
**常见考法：** 问设置检查点的作用，或问为什么不必从日志开头恢复。  
**高频误区：** 只写“提高恢复效率”，却没有说明“为什么提高”。  
**快速判断提示：** 写“减少需要扫描的日志范围”就更稳。  
**小例子：** 有最近检查点时，恢复分析通常从最近检查点附近开始，而不必从最早日志全量重扫。

#### 2.5.4 WAL 原则常和“为什么先写日志”一起考

**核心概念：** 日志先于数据写入稳定存储。  
**常见考法：** 问 WAL 的含义，或问为什么数据库不能先写数据后补日志。  
**高频误区：** 只记住“先写日志”，却说不出是“先于数据页写回”以及“为了恢复可追溯”。  
**快速判断提示：** 回答里最好同时带上“先于数据写回”和“崩溃后仍有依据”。  
**小例子：** 如果数据已改而日志没写，崩溃后就无法准确判断该改动是否应保留。

#### 2.5.5 介质故障与系统故障要分开答

**核心概念：** 系统故障主要丢失内存状态，介质故障可能损坏数据库文件本身。  
**常见考法：** 问不同故障的恢复思路差异。  
**高频误区：** 把所有故障都写成“直接看日志恢复”。  
**快速判断提示：** 一看到磁盘损坏、数据文件破坏，就要想到“备份 + 日志”。  
**小例子：** 磁盘损坏后，通常先恢复备份，再根据日志重做备份之后的已提交事务。

#### 2.5.6 恢复综合题最怕“只写动作，不写依据”

**核心概念：** 恢复题评分常看“事务状态 + 恢复动作 + 理由”是否完整。  
**常见考法：** 给出日志和故障时刻，让你写恢复过程。  
**高频误区：** 只写“Undo T2，Redo T1”，却不说明为什么。  
**快速判断提示：** 至少写出“因为 T1 已提交、T2 未提交”。  
**小例子：** “T1 已有 `COMMIT`，应保留其结果；T2 无 `COMMIT`，其修改不应保留。”

### 2.6 下午题视角下的题型映射

| 下午题类型 | 常见问法 | 作答关键动作 | 常见丢分点 |
|---|---|---|---|
| 日志恢复判断题 | “哪些事务需要 Undo / Redo” | 先看 `COMMIT`，再看更新策略，最后写恢复动作 | 只写动作，不写依据 |
| 检查点分析题 | “检查点有什么作用 / 从哪里开始恢复” | 先说明检查点意义，再说明可缩小扫描范围 | 只写“提高效率”，不写提高在哪里 |
| WAL 原理题 | “为什么日志必须先写” | 先写日志先于数据，再写崩溃后恢复依据 | 只背缩写，不解释实际意义 |
| 故障分类题 | “这是哪种故障 / 如何恢复” | 先判故障类型，再给恢复思路 | 把系统故障和介质故障混成一类 |
| 恢复综合题 | “给定日志片段，写恢复步骤” | 先分事务状态，再写 Undo / Redo，最后补原因 | 结论有了，但顺序乱、理由漏写 |

### 2.7 标准答题模板 / 采分点提示

#### 模板1：Undo / Redo 判断题
1. 先写系统在何时发生何种故障。  
2. 再根据日志判断哪些事务已经 `COMMIT`，哪些尚未提交。  
3. 若为立即修改，写出“已提交事务需要 Redo，未提交事务需要 Undo”；若为延迟修改，重点说明未提交事务通常不必 Undo。  
4. 最后补一句原因：已提交结果应保留，未提交结果不应保留。

#### 模板2：检查点作用题
1. 先写检查点是日志恢复中的阶段性标记。  
2. 再写发生故障后，可从最近检查点附近开始分析，而不必从日志起点全部扫描。  
3. 最后写结论：这样可以缩小恢复范围、减少扫描日志量、提高恢复效率。

#### 模板3：WAL 原理题
1. 先写 WAL 的含义：日志必须先于对应数据写入稳定存储。  
2. 再写如果数据先写而日志未写，崩溃后恢复将缺少依据。  
3. 最后写结论：因此 WAL 是支持 Undo / Redo 恢复的重要前提。

#### 模板4：恢复综合题
1. 先判故障类型。  
2. 再按日志区分事务是“已提交”还是“未提交”。  
3. 接着结合更新策略判断 Undo / Redo。  
4. 最后用一句话收束：数据库应恢复到“只保留已提交事务结果”的正确状态。

### 2.9 易混点速记卡片

- **Undo**：撤销不该留下的  
  **Redo**：补回本该留下的

- **已提交事务**：通常优先想 Redo  
  **未提交事务**：通常优先想 Undo

- **立即修改**：提交前数据也可能先写库  
  **延迟修改**：提交前通常不真正写库

- **系统故障**：更多靠日志恢复

Undo 用于撤销未提交事务已经造成的影响，解决“本不该留下却留下了”的问题。Redo 用于重做已提交事务应保留但可能尚未真正写入数据库的结果，解决“本该留下却可能没留下”的问题。

**采分点：**
- 写出 Undo 的作用
- 写出 Redo 的作用
- 能区分两者解决的问题不同

**作答提醒：**
- Undo / Redo 不要只翻译字面含义
- 最好补一句“各自解决什么问题”

### 第22题
**参考答案：**  
数据库要求日志先于数据写入稳定存储，是因为一旦系统崩溃，恢复过程必须依赖日志判断哪些修改应撤销、哪些应重做。如果数据已经写了，但日志还没可靠保存，恢复时就可能失去依据。因此 WAL 是支持数据库恢复的重要前提。

**采分点：**
- 写出“日志先于数据写入”
- 写出“崩溃后恢复需要日志作为依据”
- 能写出 WAL 的实际意义

**作答提醒：**
- 不要只写缩写全称
- 重点写“为什么必须这样做”

### 第23题
**参考答案：**  
对 `T3` 的恢复通常不以 Undo 为重点，因为在延迟修改策略下，事务提交前其修改结果通常还未真正写回数据库。由于 `T3` 在 `COMMIT` 前系统就崩溃了，因此其未提交结果一般不必 Undo；若系统需要恢复，也主要是确保未提交结果不要被当成已完成结果保留。

**采分点：**
- 写出“延迟修改下通常不必 Undo 未提交事务”
- 写出原因是“提交前通常未真正写库”
- 结论与更新策略对应一致

**作答提醒：**
- 本题关键不是机械写 Undo / Redo
- 而是先抓住“延迟修改”这个前提

### 第24题
**参考答案：**  
因为检查点已经把某一阶段之前的重要恢复信息做了阶段性固定，发生系统故障后，恢复分析通常可以从最近检查点附近开始，而不必从日志最开头完整扫描。这样做能减少需要检查的日志数量，缩短恢复分析时间。

**采分点：**
- 写出“可从最近检查点附近开始分析”
- 写出“不必从日志起点全量扫描”
- 写出“减少扫描范围 / 缩短恢复时间”

**作答提醒：**
- 不要只写“提高效率”
- 要写出效率提高在哪里

### 第25题
**参考答案：**  
这更属于 **介质故障**。因为磁盘损坏已经影响到数据库文件本身的可用性。常见恢复思路是：先利用备份恢复数据库文件，再结合日志把备份之后已经提交的事务结果重新做回去。

**采分点：**
- 写出“介质故障”
- 写出“数据库文件本身不可用”这一判断依据
- 能概括“备份 + 日志恢复”的思路

**作答提醒：**
- 不要把介质故障答成普通系统故障
- 一看到磁盘损坏，就要想到备份

### 第26题
**参考答案：**  
先看事务提交状态，`T1` 已提交而 `T2` 尚未提交。由于题目给定的是立即修改环境，未提交事务的修改也可能已经部分写入数据库，因此 `T2` 需要 Undo。与此同时，`T1` 虽然已经提交，但其结果未必全部真正写回数据库，因此通常还需要 Redo。恢复完成后，数据库应回到只保留已提交事务结果的正确状态。

**采分点：**
- 顺序完整：提交状态 → Undo / Redo → 恢复目标
- 写出 `T1` Redo、`T2` Undo
- 能写出“立即修改”这一关键依据

**作答提醒：**
- 先看事务状态，再看更新策略
- 最后不要漏写恢复目标

### 第27题
**参考答案：**  
因为检查点时系统中活跃事务为 `T4、T5`，说明它们跨越了检查点，恢复时需要重点确认其后续是否提交。检查点之后 `T6` 又启动并执行，因此它也属于故障前需要纳入恢复分析的事务集合。换言之，恢复时至少要关注“检查点时仍活跃”以及“检查点后新出现”的事务状态。

**采分点：**
- 写出 `T4、T5` 在检查点时处于活跃状态
- 写出 `T6` 是检查点后新出现的事务
- 能说明这些事务都需要纳入恢复分析范围

**作答提醒：**
- 不要把检查点理解成“之前的事务全都不用管”
- 关键看哪些事务在检查点前后仍与故障相关

### 第28题
**参考答案：**  
检查点能帮助减少恢复时间，是因为发生故障后系统通常不必从日志最开始逐条扫描，而可以从最近检查点附近重点分析。这样能缩小需要判断的事务集合和日志范围，因此恢复过程更快、更有针对性。

**采分点：**
- 写出“从最近检查点附近开始分析”
- 写出“缩小日志扫描范围”
- 写出“恢复时间更短”

**作答提醒：**
- “提高效率”要展开写成具体原因

### 第29题
**参考答案：**  
数据库运行中一旦发生故障，风险在于未提交事务的错误修改可能被留下，而已提交事务的正确结果也可能尚未完整落盘。为降低这种风险，需要依靠日志记录事务操作，利用检查点缩小恢复范围，并遵守 WAL 原则保证日志先于数据写入稳定存储。这样做的原因是：恢复系统可以据此判断哪些事务应 Undo、哪些应 Redo，并更快定位恢复起点。综上，日志、检查点与 WAL 共同构成数据库恢复机制的重要基础。

**采分点：**
- 顺序完整：风险 → 手段 → 原因 → 结论
- 至少写出日志、检查点、WAL 中的两个以上关键手段
- 能说明为什么这些手段有效

**作答提醒：**
- 不要只堆术语
- 要把“为什么有效”写出来

### 第30题
**参考答案：**  
立即修改环境下，事务提交前修改结果也可能已经写入数据库，因此系统崩溃后，未提交事务可能需要 Undo，而已提交事务又可能需要 Redo，所以两类动作都更常见。延迟修改环境下，未提交事务的修改通常还没有真正写回数据库，因此对未提交事务往往不必 Undo。两者的差异，关键就在于“提交前数据是否可能已经写库”。

**采分点：**
- 写出立即修改下更容易同时出现 Undo 与 Redo
- 写出延迟修改下未提交事务往往不必 Undo
- 能点出关键差异是“提交前是否可能已写库”

**作答提醒：**
- 比较题要把两边都写到
- 最后最好补一句“关键差异在哪里”
