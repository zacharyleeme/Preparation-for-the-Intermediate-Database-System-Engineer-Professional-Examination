### 2.3 今日核心知识框架

今天必须固定下来的，不只是几个定义，而是一条能用于下午题的 **标准判断链**。

#### 1）先列函数依赖，不要先猜范式

> **范式结论的前提，是你真的写清了依赖关系。**

- 先从题干中找“唯一”“可确定”“由谁决定谁”等描述；
- 把能稳定成立的依赖单独写出来；
- 先写单属性左边的依赖，再看是否存在组合键相关依赖；
- 不要跳过这一步直接写“属于 2NF / 3NF”。

#### 2）先求候选键，再分主属性 / 非主属性

> **不会候选键，后面的部分依赖、传递依赖判断就会飘。**

- 先看哪些属性能唯一标识元组；
- 若多个属性组合才能唯一标识，就要判断组合键是否最小；
- 属于任一候选键的属性是 **主属性**；
- 不属于任何候选键的属性是 **非主属性**。

#### 3）2NF 的关键，是看“部分依赖”

> **只有先出现了组合候选键，才谈得上部分依赖。**

- 若候选键是单属性，则通常不存在部分依赖，2NF 往往天然满足；
- 若候选键是组合属性，要检查是否有非主属性只依赖候选键的一部分；
- 一旦存在，就不满足 2NF。

#### 4）3NF 的关键，是看“传递依赖”与“非主属性之间的决定关系”

> **2NF 解决“只依赖一部分”，3NF 进一步解决“拐个弯依赖上去”。**

- 若候选键决定某个非主属性，而这个非主属性又决定别的非主属性；
- 或题干已明显给出“甲属性能确定乙属性”，但甲本身又不是候选键；
- 则要警惕 3NF 不满足。

#### 5）规范化题最后一定要落到“为什么拆、怎么拆”

> **下午题不是只问结论，而是要问：不规范会怎样，规范化后怎么更稳。**

- 先说明：是否有冗余、插入异常、删除异常、更新异常；
- 再说明：应该把哪类只依赖某一部分的信息拆出去；
- 最后写：拆分后更利于维护、一致性与长期扩展。

### 2.4 必备结论

- 范式题最稳的起点，不是先背 2NF / 3NF 定义，而是先 **列函数依赖并求候选键**。
- 主属性是“属于至少一个候选键的属性”，不是“看起来重要的属性”。
- 若候选键是单属性，则通常不存在部分依赖，因此 2NF 往往天然满足。
- 组合候选键场景下，非主属性只依赖候选键的一部分，属于 **部分依赖**，会破坏 2NF。
- 非主属性依赖于另一个非主属性，通常要警惕 **传递依赖**，会破坏 3NF。
- 3NF 的判断重点，是看“是否存在不该由非主属性来决定的非主属性”。
- BCNF 比 3NF 更严格，核心口径是：**每个决定因素都应当是超键**。
- 冗余最常带来三类异常：**更新异常、插入异常、删除异常**。
- 下午题里“为什么不满足某范式”必须写出 **候选键 / 主码 + 具体依赖 + 违反原因**，不能只写“有冗余”。
- 规范化分解的目标，不只是“表变多”，而是 **减少冗余、消除异常、让依赖关系更稳定**。

### 2.5 高频考点拆解

#### 2.5.1 左边只有一个属性的函数依赖怎么列

**核心概念：** 题干常先要求写“左侧只有一个属性”的函数依赖。  
**常见考法：** 2020 人员分配、2022 冲线记录、2018 患者关系。  
**高频误区：** 把组合键才能决定的信息也硬写成单属性依赖。  
**快速判断提示：** 只有题干明确说明“某属性唯一”或“某属性可确定其他属性”时，才稳妥写入。  
**小例子：** `人员编号 → 人员姓名`，`小区编号 → 物业经理姓名`。

#### 2.5.2 候选键怎么求才不乱

**核心概念：** 候选键 = 最小且能唯一标识元组的属性集合。  
**常见考法：** 让你直接写候选键，或用候选键作为后续范式判断基础。  
**高频误区：** 把超键直接当候选键；或者漏掉“最小性”。  
**快速判断提示：** 先看哪些属性“不会被别人决定”，它们通常更可能出现在候选键里。  
**小例子：** 若“身份证号唯一”“医保卡号唯一”，则两者都可能是候选键。

#### 2.5.3 2NF 的失分点，常在“部分依赖”没写清

**核心概念：** 非主属性只依赖组合候选键的一部分。  
**常见考法：** 判断是否满足 2NF，并用几十字说明原因。  
**高频误区：** 候选键还没求，就先说“有部分依赖”。  
**快速判断提示：** 先确认是不是组合候选键；若不是，基本不用谈部分依赖。  
**小例子：** `(人员编号, 小区编号)` 为键，而 `人员姓名` 只依赖 `人员编号`。

#### 2.5.4 3NF 的失分点，常在“传递依赖”与“冗余价值”没写满

**核心概念：** 主码决定了一个非主属性，而该非主属性又决定另一个非主属性。  
**常见考法：** 问“是否满足 3NF”“不满足的依赖是什么”“为什么会有冗余”。  
**高频误区：** 只写“有传递依赖”，不写具体是哪个依赖导致的。  
**快速判断提示：** 看题干中是否存在“车号决定组别”“手机号决定车手姓名”这种中间决定链。  
**小例子：** `序列号 → 车号`，`车号 → 组别`，则 `序列号 → 组别` 属于传递获得。

#### 2.5.5 规范化题最后一定要写到“异常”和“分解方向”

**核心概念：** 下午题不只看你会判范式，还看你能不能说清“为什么要拆”。  
**常见考法：** 冗余属性、更新异常、删除异常、简述规范化方案。  
**高频误区：** 只写“拆成 3 张表”，不写每张表各保留什么信息。  
**快速判断提示：** 先按“基础实体信息”“业务发生信息”“引用关系信息”三层去拆。  
**小例子：** 冲线记录保留成绩过程，赛车表保留赛车静态信息，车手表保留车手静态信息。

### 2.6 下午题视角下的题型映射

| 下午题类型 | 常见问法 | 作答关键动作 | 常见丢分点 |
|---|---|---|---|
| 候选键 / 范式判断题 | “给出候选键”“判定属于第几范式” | 先列依赖，再求键，再分主非主属性 | 直接写结论，不写依赖依据 |
| 2NF / 3NF 说明题 | “是否满足 2NF / 3NF，并说明原因” | 先写主码，再写部分依赖 / 传递依赖 | 不写主码；不写具体依赖 |
| 冗余 / 异常分析题 | “有哪些冗余”“会出现哪些异常” | 先点出冗余属性，再写更新 / 删除 / 插入异常 | 只写“有冗余”，不写异常类型 |
| 规范化分解题 | “请简述分解方案”“给出分表结果” | 按静态信息 / 业务过程 / 关联信息分层拆表 | 只写表名，不写主外键 |

### 2.7 标准答题模板 / 采分点提示

#### 模板1：候选键 + 最高范式判断题
1. 先写题目中可以稳定成立的函数依赖：……
2. 再写候选键 / 主码为：……
3. 写明哪些属性是主属性，哪些是非主属性：……
4. 判断是否存在部分依赖：……
5. 判断是否存在传递依赖：……
6. 因此该关系模式最高达到 …… 范式。

#### 模板2：50～100 字规范化说明题
1. 先写主码 / 候选键是什么；
2. 再写哪个非主属性只依赖一部分，或哪个依赖导致了传递依赖；
3. 再下结论：因此不满足 2NF / 3NF；
4. 最后写应拆分出的关系方向与价值。

#### 模板3：冗余与异常分析题
1. 先写冗余属性或冗余信息来自哪里；
2. 再写会造成哪些异常：更新异常 / 删除异常 / 插入异常；
3. 最后写应如何将静态信息与业务记录信息分离。

### 2.9 易混点速记卡片

- **候选键**：最小且能唯一标识元组  
  **超键**：能唯一标识元组，但不一定最小

- **主属性**：属于至少一个候选键  
  **非主属性**：不属于任何候选键

- **部分依赖**：非主属性依赖组合键的一部分  
  **传递依赖**：非主属性依赖另一个非主属性

- **2NF**：先排除部分依赖  
  **3NF**：再排除不合规的传递依赖

- **3NF**：比 2NF 更严格  
  **BCNF**：比 3NF 更严格

- **冗余**：同一事实重复存放  
  **异常**：更新 / 插入 / 删除时容易出错或丢信息

### 第1题
**答案：B**  
**解析：** 非主属性依赖组合候选键的一部分，属于部分依赖，直接破坏 2NF。

### 第2题
**答案：B**  
**解析：** 候选键为单属性 A，因此 2NF 通常满足；但存在 `A→B→C` 的传递依赖，故不满足 3NF，最高通常只能达到 2NF。

### 第3题
**答案：A**  
**解析：** 候选键强调“最小”和“唯一标识”两个条件。

### 第4题
**答案：D**  
**解析：** 属于至少一个候选键的属性称为主属性。

### 第5题
**答案：C**  
**解析：** 下午题最稳的判断顺序是：先列函数依赖，再求候选键，然后才能判主非主属性与范式。

### 第6题
**答案：A**  
**解析：** `Sname` 只依赖 `Sno`，`Cname` 只依赖 `Cno`，都属于对组合键的一部分依赖，是典型部分依赖。

### 第7题
**答案：B**  
**解析：** 部分依赖依赖于“组合候选键”的存在；若候选键是单属性，通常不存在部分依赖。

### 第8题
**答案：C**  
**解析：** BCNF 要求每个决定因素都应是超键，因此比 3NF 更严格。

### 第9题
**答案：B**  
**解析：** “物业经理姓名”更像是由小区这一侧决定，而不是由人员编号决定。

**答案：C**  
**解析：** `车号→组别`、`手机号→车手姓名` 容易在冲线记录表中形成传递依赖与冗余存储。

### 第11题
**答案：A**  
**解析：** 若身份证号唯一、医保卡号唯一，则两者都可以单独唯一标识患者，因此候选键至少有两个。

### 第12题

### 第13题
**答案：B**

### 第14题

### 第15题
**答案：B**

### 第18题
**答案：C**  
**解析：** 下午题短文说明不能只给情绪化结论，最稳妥的写法是：主码 / 候选键 → 依赖问题 → 范式结论 → 分解方向。

### 第19题
**参考答案：**  
可较稳定写出的单属性左侧函数依赖包括：
1. 身份证号 → 医保卡号，姓名，性别，民族，出生日期  
2. 医保卡号 → 身份证号，姓名，性别，民族，出生日期

**采分点：**
- 写出“身份证号可确定其他患者属性”
- 写出“医保卡号可确定其他患者属性”
- 不把需要组合才能确定的信息误写成单属性依赖

**作答提醒：**
- 先看题干中“唯一”的字段
- 先写最稳、最明确的依赖

### 第20题
**参考答案：**  
该关系的候选键可写为：`身份证号`、`医保卡号`。  
由于这两个决定因素本身都可作为候选键，其他属性都完全函数依赖于它们，不存在组合键下的部分依赖，也没有题干给出的不合规传递依赖。  
因此该关系至少可稳定达到 3NF，按更严格口径通常也可视为满足 BCNF。

**采分点：**
- 写出全部候选键
- 能说明为什么不存在部分依赖
- 能说明结论不是拍脑袋，而是依赖 + 键共同决定

**作答提醒：**
- 别漏写“为什么”
- 若写 BCNF，最好补一句“决定因素本身是候选键”

### 第21题
**参考答案：**  
这个关系模式把两层信息混装到了一起：
1. 病例本身的基本诊断信息（患者、医生、病情、诊断结论、诊断日期）；
2. 与病例相关的购药 / 用药明细信息（药品条码、服用剂量、药品数量、购买日期）。

这样做更容易导致：
- 同一病例的诊断信息因为多种药品而被重复存储；
- 更新病例基本信息时需要多处同步修改；
- 删除最后一条购药记录时，可能连病例基础信息都被误删。

**采分点：**
- 点出“病例基本信息”与“购药明细信息”混装
- 写出至少 2 类异常或冗余风险
- 说明规范化不是为了拆表而拆表，而是为了减少重复与异常

**作答提醒：**
- 先说混装了什么
- 再说为什么不稳

### 第22题
**参考答案：**  
一种更合理的分解方向可写为：
- 病例基本信息（编号，患者身份证号，主治医生编号，病情描述，诊断结论，诊断日期）
- 购药 / 用药明细（病例编号，药品条码，服用剂量，药品数量，购买日期）

这样拆分后，病例本身的诊断信息与每次用药明细分层保存，更有利于减少冗余、避免异常，也更方便后续统计病例与药品使用情况。

**采分点：**
- 基本信息与明细信息分层
- 至少给出两张表的核心字段方向
- 写出“减少冗余 / 便于维护 / 便于统计”中的至少 1～2 个价值点

**作答提醒：**
- 不要求唯一答案，但要有稳定方向
- 不要把所有字段仍塞回一张表

### 第23题
**参考答案：**  
可较稳定写出的单属性左侧函数依赖包括：
1. 人员编号 → 人员姓名  
2. 小区编号 → 物业经理姓名

**采分点：**
- 写出人员编号决定人员姓名
- 写出小区编号决定物业经理姓名
- 不把“人员职责”误写成只依赖单属性

**作答提醒：**
- “人员职责”对应的是某人在某小区的工作，不宜直接单属性决定

### 第24题
**参考答案：**  
该关系通常**不满足 2NF**。因为若候选键为 `(人员编号, 小区编号)`，则非主属性“人员姓名”只依赖于“人员编号”，“物业经理姓名”只依赖于“小区编号”，都属于对组合键一部分的依赖，即部分依赖，因此不满足 2NF。

**采分点：**
- 写出候选键 / 主码
- 明确指出“人员姓名”与“物业经理姓名”分别只依赖一部分
- 得出“不满足 2NF”的结论

**作答提醒：**
- 别只写“有部分依赖”
- 要把具体依赖说出来

### 第25题
**参考答案：**  
一种更合理的 3NF 分解方向可写为：
- 人员（人员编号，人员姓名）
- 小区（小区编号，物业经理姓名）
- 人员分配（人员编号，小区编号，人员职责）

这样可以把只依赖人员编号的信息、只依赖小区编号的信息与真正的分配关系分开保存，减少冗余并避免异常。

**采分点：**
- 拆出人员信息表
- 拆出小区信息表
- 保留人员与小区的分配关系表

**作答提醒：**
- 这类题的关键不是表名漂亮，而是依赖边界清楚

### 第26题
**参考答案：**  
可参考如下主键 / 外键方向：
- 人员：主键为 `人员编号`
- 小区：主键为 `小区编号`
- 人员分配：主键可为 `(人员编号, 小区编号)`；其中 `人员编号` 是外键，参照人员表；`小区编号` 是外键，参照小区表

**采分点：**
- 写出三个关系模式的主键方向
- 写出分配表中的两个外键
- 不把“人员职责”误当主键

**作答提醒：**
- 主键先保证唯一性
- 外键再表达引用关系

### 第27题
**参考答案：**  
冲线记录表中存在数据冗余。更明显的冗余属性包括：
- `组别`：题干已给出可由 `车号` 确定；
- `车手姓名`：题干已给出 `手机号` 唯一，而姓名可重复，因此车手姓名更适合通过车手信息表维护。

**采分点：**
- 判断出存在冗余
- 列出至少 1～2 个更明显的冗余属性
- 理由要和题干给出的依赖对应上

**作答提醒：**
- 冗余不是“看着多”，而是“能由别的信息稳定推出”

### 第28题
**参考答案：**  
可参考如下设计方向：
1. 冲线记录表主码：`序列号`；外码：`车号` 参照赛车表，`手机号` 参照车手表；
2. 赛车表主码：`车号`；
3. 车手表主码：`手机号`。

**采分点：**
- 序列号是冲线记录的主码
- 车号、手机号的外键方向正确
- 赛车表、车手表的主码分别写稳

**作答提醒：**
- 题干已明确“序列号唯一”“手机号唯一”
- 不要把姓名误写成车手表主码

### 第29题
**参考答案：**  
该冲线记录表通常**不满足 3NF**。因为：
- `序列号 → 车号`，而 `车号 → 组别`；
- `序列号 → 手机号`，而 `手机号 → 车手姓名`。

这说明原表中存在通过非主属性继续决定其他非主属性的情况，容易造成冗余与更新异常，因此不满足更稳妥的 3NF 结构要求。

**采分点：**
- 给出至少一组关键依赖链
- 写明这是导致 3NF 不满足的原因
- 能把“冗余 / 异常”与依赖链联系起来

**作答提醒：**
- 别只写“有传递依赖”
- 要把依赖链写出来

### 第30题
**参考答案：**  
冲线记录表中，`序列号` 可以作为主码，用于唯一标识一条冲线记录。  
但题干又给出了 `车号 → 组别`、`手机号 → 车手姓名` 这样的依赖关系，说明原表中存在通过非主属性继续决定其他属性的情况。  
因此该表不满足更稳妥的 3NF 结构要求，容易造成组别和车手姓名的重复存储，以及更新不一致等问题。  
更合理的做法，是将冲线记录、赛车静态信息、车手静态信息分开保存，例如拆分为冲线记录表、赛车表和车手表。  
这样既能减少冗余，也更利于后续维护、查询与数据一致性控制。

**采分点：**
- 顺序完整：主码 → 依赖问题 → 范式结论 → 分解方向与价值
- 写出至少 1 组关键依赖链
- 有完整收束句

**作答提醒：**
- 模板题最怕只写拆表结论，不写为什么
- 结尾尽量写到“减少冗余 / 一致性 / 维护价值”
