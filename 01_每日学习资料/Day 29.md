### 2.3 今日核心知识框架

#### 主线 A：调度分析

调度题不要先背结论，先按以下顺序做：

1. 找事务：`T1`、`T2`、`T3` 分别做什么；
2. 找共享数据项：如 `A`、`B`、`X`、账户余额、库存数量；
3. 找冲突操作：不同事务、同一数据项、至少一个写；
4. 画冲突边：如果 `T1` 的冲突操作在前，`T2` 在后，则记 `T1 → T2`；
5. 看图是否有环：
   - 无环：冲突可串行化；
   - 有环：不可冲突可串行化。

#### 主线 B：两段锁与死锁

两段锁协议的标准理解：

1. 事务先进入**加锁阶段**；
2. 一旦开始释放锁，就进入**解锁阶段**；
3. 进入解锁阶段后不能再申请新锁；
4. 2PL 能保证可串行化；
5. 但若事务互相持有对方需要的锁，仍可能死锁。

死锁题固定写：

> `T1` 持有 `A` 等 `B`，`T2` 持有 `B` 等 `A`，形成循环等待，因此可能死锁。  
> 更安全的改写是让所有事务按相同顺序申请锁，例如都先申请 `A` 再申请 `B`。

#### 主线 C：隔离级别与并发异常

| 隔离级别 | 至少能避免 | 仍可能出现 |
|---|---|---|
| `Read Uncommitted` | 基本不保证 | 脏读、不可重复读、幻读 |
| `Read Committed` | 脏读 | 不可重复读、幻读 |
| `Repeatable Read` | 脏读、不可重复读 | 一般仍可能幻读 |
| `Serializable` | 脏读、不可重复读、幻读 | 并发性能下降 |

> 今日重点：看到“不能读未提交结果”，最低答案是 **`Read Committed`**；看到“仍可能出现什么”，优先写 **不可重复读 / 幻读**。

#### 主线 D：日志恢复

恢复题固定顺序：

1. 找检查点；
2. 找系统故障 / 崩溃时刻；
3. 列检查点后仍活跃或发生更新的事务；
4. 区分：
   - 已提交但可能未落盘 → `Redo`；
   - 未提交但已写数据 → `Undo`；
   - 检查点前已完成且不在活跃表中 → 通常不处理；
5. 若要求恢复后数据值：
   - `Redo` 用新值；
   - `Undo` 回旧值；
   - 多条日志按时间顺序处理，撤销通常逆序看未提交事务的更新。

### 2.4 必备结论

- 冲突操作必须满足：**不同事务、同一数据项、至少一个写**。
- 冲突图无环，调度才是冲突可串行化。
- 2PL 能保证可串行化，但不能保证一定不死锁。
- 死锁的核心原因是循环等待。
- 避免死锁的常见考试答案是：统一锁申请顺序。
- `Read Committed` 能避免脏读，但仍可能不可重复读 / 幻读。
- 日志恢复题先找检查点，再看提交状态，不要直接背 `Redo / Undo`。
- 已提交但可能未落盘的事务进入 `Redo`。
- 未提交但已经写过数据的事务进入 `Undo`。
- 介质故障优先恢复备份，再结合日志重做。

### 2.5 高频考点拆解

#### 2.5.1 冲突可串行化：下午题常见加分点

**核心概念：** 如果一个并发调度的冲突顺序等价于某个串行调度，则称为冲突可串行化。  
**常见考法：** 给出 `R1(A)、W2(A)、W1(B)、R2(B)` 等序列，要求判断是否可串行化。  
**高频误区：** 只说“看起来有序 / 无序”，不列冲突边。  
**快速判断提示：** 先画 `T1 → T2`、`T2 → T1`，有双向边通常就危险。  
**小例子：** 若对 `A` 有 `T1 → T2`，对 `B` 又有 `T2 → T1`，冲突图成环，不可冲突可串行化。

#### 2.5.2 两段锁：别把“可串行化”和“无死锁”混掉

**核心概念：** 2PL 约束事务加锁 / 解锁阶段，开始释放锁后不能再申请新锁。  
**常见考法：** 判断说法正误，或要求改写成满足 2PL 的调度。  
**高频误区：** 写成“2PL 可以避免死锁”。  
**快速判断提示：** 看到 2PL，先写“可串行化”；看到死锁，写“仍可能循环等待”。  
**小例子：** `T1` 锁 `A` 等 `B`，`T2` 锁 `B` 等 `A`，即使都遵守 2PL，也可能死锁。

#### 2.5.3 死锁改写：统一锁顺序是标准答案

**核心概念：** 多个事务按相反顺序持锁并等待，会形成循环等待。  
**常见考法：** 2017 抢红包、2019 转账汇总类题型常要求给出不死锁调度。  
**高频误区：** 写成“先释放当前锁再去申请其他锁”，破坏 2PL。  
**快速判断提示：** 改写时优先让所有事务按相同顺序拿锁，例如都先 `A` 后 `B`。  
**小例子：** 转账事务与汇总事务都按账户号升序申请锁，可降低循环等待风险。

#### 2.5.4 隔离级别：提交读只解决一层问题

**核心概念：** `Read Committed` 的最低保证是不读未提交数据。  
**常见考法：** 给出“不能读到未提交结果”问最低隔离级别。  
**高频误区：** 把 `Read Uncommitted` 误当最低可用答案，或把残留问题写成死锁 / 阻塞。  
**快速判断提示：** 考隔离级别时，优先回答并发异常名称。  
**小例子：** 汇总交易不能读未提交转账结果：至少 `Read Committed`，但仍可能不可重复读 / 幻读。

#### 2.5.5 检查点恢复：名单比口号重要

**核心概念：** 检查点用于缩小恢复分析范围，恢复时重点看检查点后活跃或更新过的事务。  
**常见考法：** 2020 / 2022 / 2023 下午日志题常给日志表，让列 `Redo / Undo`。  
**高频误区：** 看到提交就都 Redo，看到未提交就都 Undo，却不结合检查点与是否更新。  
**快速判断提示：** 先列事务状态表，再写 `Redo / Undo`。  
**小例子：** 检查点后开始并提交的事务 → Redo；检查点前开始但故障前未提交且检查点后写过数据 → Undo。

### 2.7 标准答题模板 / 采分点提示

#### 模板1：调度可串行化判断

1. 列出涉及的数据项：`A`、`B`、`X`……
2. 找不同事务在同一数据项上的冲突操作，冲突条件是至少一个写；
3. 根据先后顺序画冲突边：如 `T1 → T2`；
4. 判断冲突图是否有环；
5. 无环则冲突可串行化，并给出等价串行顺序；有环则不可冲突可串行化。

#### 模板2：两段锁协议改写

```text
T1: XLock(A) → R/W(A) → XLock(B) → R/W(B) → Commit → Unlock(B) → Unlock(A)
T2: 等待 T1 释放冲突锁 → XLock(A) → R/W(A) → XLock(B) → R/W(B) → Commit → Unlock(B) → Unlock(A)
```

写题时必须出现：
- 加锁；
- 等待 / 串行进入；
- 读写或计算；
- 提交；
- 解锁。

#### 模板3：死锁原因与改写

```text
死锁原因：T1 持有 A 等待 B，T2 持有 B 等待 A，两个事务互相等待对方释放锁，形成循环等待，因此可能死锁。
改写思路：让两个事务按相同顺序申请锁，例如都先申请 A，再申请 B；这样在满足 2PL 的前提下，可以避免相反方向的循环等待。
```

#### 模板4：日志恢复 Redo / Undo 表

| 事务 | 检查点前/后 | 是否写过数据 | 故障前是否提交 | 恢复动作 | 理由 |
|---|---|---|---|---|---|
| T1 |  |  |  | Redo / Undo / 不处理 |  |

固定判断：
- 已提交但可能未落盘 → `Redo`；
- 未提交但已写数据 → `Undo`；
- 检查点前已完成且不在活跃表 → 通常不处理。

#### 模板5：恢复后数据值

1. 先写故障前日志中的最终可见修改；
2. 对 `Redo` 事务保留 / 重做新值；
3. 对 `Undo` 事务回滚到旧值；
4. 若同一数据项有多次更新，按日志顺序与事务状态综合判断。

### 第4题
**答案：C**  
**解析：** 2PL 要求事务一旦开始释放锁，就不能再申请新锁；它能保证可串行化，但不能保证无死锁。

### 第5题
**答案：C**  
**解析：** 死锁的典型原因是多个事务互相等待对方释放资源，形成循环等待。

### 第6题
**答案：A**  
**解析：** 多个事务按相同顺序申请锁，可减少相反方向等待链，降低死锁风险。

### 第7题
**答案：B**  
**解析：** 冲突可串行化常通过构造优先图 / 冲突图判断；图中无环则冲突可串行化。

### 第8题
**答案：C**  
**解析：** 冲突操作条件是：不同事务、同一数据项、至少一个写。不同事务对同一数据项的一读一写构成冲突。

### 第9题
**答案：A**  
**解析：** `Read Committed` 至少能避免脏读。

### 第10题
**答案：B**  
**解析：** 在一般定义下，`Read Committed` 仍可能出现不可重复读与幻读。

### 第11题
**答案：B**  
**解析：** 已提交但可能尚未落盘的事务需要 `Redo`，以保证持久性。

### 第12题
**答案：B**  
**解析：** 未提交但已写过数据的事务需要 `Undo`，以保证原子性。

### 第13题
**答案：A**  
**解析：** 检查点的主要作用是缩小恢复时要扫描和处理的日志范围，提高恢复效率。

### 第14题
**答案：C**  
**解析：** 磁盘损坏、数据文件无法读取，属于介质故障。

### 第15题
**答案：B**  
**解析：** 介质故障通常先从备份恢复数据文件，再结合日志重做，恢复到故障前一致状态。

### 第16题
**答案：C**  
**解析：** 同一条件两次查询，结果集合行数变化，属于幻读。

### 第17题
**答案：A**  
**解析：** 修改数据需要排他锁，防止其他事务同时读写造成不一致。

### 第18题
**答案：A**  
**解析：** `Redo` 对应已提交事务，`Undo` 对应未提交事务；检查点用于恢复效率，不是并发冲突控制。

### 第19题
**参考答案：**
1. 并发执行结束后，`X = 4`；
2. 正确串行结果应为 `X = 3`；
3. 属于 **丢失修改**。因为两个事务都基于旧值 `3` 计算，`T1` 写回 `2` 后，`T2` 又基于旧值写回 `4`，后写覆盖了前写的修改效果。

**采分点：**
- 算对并发最终值；
- 写出正确串行值；
- 点名丢失修改；
- 写出后写覆盖前写。

### 第20题
**参考答案：**
两段锁协议要求事务先进入加锁阶段，再进入解锁阶段；一旦开始释放锁，就不能再申请新锁。这样可以保证冲突操作的先后顺序与某个串行顺序一致，因此保证可串行化。  
但 2PL 不能保证无死锁，因为多个事务仍可能分别持有对方需要的锁，形成循环等待。

**采分点：**
- 加锁阶段 / 解锁阶段；
- 可串行化；
- 循环等待；
- 不保无死锁。

### 第21题
**参考答案：**
1. 至少设置 **`Read Committed`（提交读）**；
2. 至少能避免 **脏读**；
3. 一般仍可能出现 **不可重复读** 和 **幻读**。

**采分点：**
- 隔离级别名称正确；
- 能防脏读；
- 残留问题写成不可重复读 / 幻读。

### 第22题
**参考答案：**
1. `T1` 持有 `A` 的排他锁等待 `B`，`T2` 持有 `B` 的排他锁等待 `A`，两个事务互相等待对方释放锁，形成循环等待，因此可能死锁；
2. 改写思路：让两个事务按相同顺序申请锁，例如都先申请 `A`，再申请 `B`，在满足 2PL 的前提下避免相反方向的等待链。

**采分点：**
- 写出循环等待；
- 写出统一锁申请顺序；
- 不破坏 2PL。

### 第23题
**参考答案：**
一种思路是让第一个响应者先获得对 `X` 的排他锁，再完成读取、判断、写入、提交和解锁，其他抢红包人等待其释放锁后再读取。示例：

```text
XLock1(X)
a1 = R1(X)
若 X 为空，则 W1(b1, X)
Commit1
ULock1(X)

XLock2(X)
a2 = R2(X)
发现 X 已非空，不再写入
Commit2 / Rollback2
ULock2(X)

XLock3(X)
a3 = R3(X)
发现 X 已非空，不再写入
Commit3 / Rollback3
ULock3(X)
```

**采分点：**
- 第一个响应者先拿锁；
- 使用排他锁保护读写判断；
- 后续事务等待；
- 满足 2PL，提交后再解锁；
- 不形成循环等待。

### 第24题
**参考答案：**
1. `R1(A)` 与 `W2(A)` 冲突，方向为 `T1 → T2`；`R2(B)` 与 `W1(B)` 冲突，方向为 `T2 → T1`；
2. 冲突图中有 `T1 → T2` 和 `T2 → T1`；
3. 图中有环，因此该调度不是冲突可串行化调度。

**采分点：**
- 找出两个冲突数据项；
- 画出双向边；
- 判断有环，不可冲突可串行化。

### 第25题
**参考答案：**
一种满足 2PL 且降低死锁风险的写法：

```text
T1: SLock(A) → SLock(B) → R(A) → R(B) → 计算汇总 → Commit → Unlock(B) → Unlock(A)
T2: 等待 T1 释放 A、B 后
T2: XLock(A) → XLock(B) → R(A) → W(A) → R(B) → W(B) → Commit → Unlock(B) → Unlock(A)
```

也可让 `T2` 先执行，但两个事务都应按相同顺序申请 `A`、`B` 的锁。

**采分点：**
- 按相同顺序申请锁；
- 体现等待 / 串行进入；
- 提交后解锁；
- 满足 2PL。

### 第26题
**参考答案：**
1. 不可重复读：同一事务两次读取同一行库存，读到不同值；
2. 脏读：读取了另一个事务尚未提交的数据，而该事务后来回滚；
3. 幻读：相同条件两次查询，结果集合行数发生变化。

**采分点：**
- 术语正确；
- 理由与现象对应；
- 区分同一行变化与集合行数变化。

### 第27题
**参考答案：**
- `T1`：`Redo`。检查点后开始，写过数据，故障前已提交；
- `T2`：`Undo`。检查点前开始但检查点后继续写过数据，故障前未提交；
- `T3`：通常不处理或仅作为活跃未提交但无写事务处理。因为未写数据，无需撤销数据项更新；
- `T4`：通常不处理。检查点前已提交，且不在检查点后活跃事务列表中。

**采分点：**
- 已提交写事务 Redo；
- 未提交写事务 Undo；
- 无写事务不作为重点撤销对象；
- 检查点前稳定完成事务通常不处理。

### 第28题
**参考答案：**
1. `T1` 已提交，应 `Redo`；`T2` 未提交，应 `Undo`；
2. `X`：保留 / 重做 `T1` 的新值 `80`；
3. `Y`：撤销 `T2` 的更新，恢复为 `50`；
4. `Z`：撤销 `T2` 的更新，恢复为 `30`。

**采分点：**
- T1 Redo；
- T2 Undo；
- X 用新值；
- Y / Z 回旧值。

### 第29题
**参考答案：**
1. 事务故障：首要动作是回滚该事务 / 对该事务执行 Undo；
2. 系统故障：重启后依据日志对未提交事务 Undo，对已提交但未持久化事务 Redo；
3. 介质故障：先恢复备份，再结合日志进行重做恢复。

**采分点：**
- 三类故障名称准确；
- 恢复动作匹配；
- 介质故障先备份后日志。

### 第30题
**参考答案：**
- `Redo`：`T1`、`T3`。二者故障前已提交，且存在检查点后相关活动 / 写操作，需要保证结果持久化；
- `Undo`：`T2`。检查点后开始，写过 `B`，故障前未提交；
- 通常不处理：`T4`。检查点前已提交，检查点后无活动；
- 判断规则：**已提交但可能未落盘的事务 Redo，未提交但写过数据的事务 Undo，检查点前已稳定完成且检查点后无活动的事务通常不处理。**

**采分点：**
- Redo 列表正确；
- Undo 列表正确；
- 不处理列表正确；
- 写出判断规则。
