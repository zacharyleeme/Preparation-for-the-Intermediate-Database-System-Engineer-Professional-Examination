### 2.3 今日核心知识框架

今天要固定下来的，是一套 **“事务恢复综合题标准分析框架”**：

> **先判故障类型 → 再定恢复对象集合 → 再判 Undo / Redo → 最后写恢复目标与依据。**

#### 1）恢复综合题先分“故障类型”，不要直接跳到 Undo / Redo

恢复题最常见的失误之一，就是看到日志片段后立刻开始判 Undo / Redo，却没先分清：
- 是事务故障；
- 是系统故障；
- 还是介质故障。

三类故障的恢复重点不同：
- **事务故障**：关注单个事务的回滚；
- **系统故障**：关注内存信息丢失，但数据库和日志通常仍可用；
- **介质故障**：关注数据库文件本身损坏，需要先恢复备份，再结合日志处理后续已提交事务。

#### 2）恢复综合题第二步要先划定“恢复对象集合”

Day 15 已经说明，检查点题不能只答“从检查点附近开始分析”，还必须回答：

> **哪些事务必须被纳入恢复判断集合？**

在有检查点的系统故障题里，至少要关注两类事务：
1. **检查点时仍活跃的事务**
2. **检查点后新启动并在故障前执行的事务**

只有先把恢复对象集合找准，后面的 Undo / Redo 才不会漏判。

#### 3）恢复综合题第三步才是“提交状态 + 更新策略 + Undo / Redo”

对象集合确定后，再看：
- 哪些事务已提交；
- 哪些事务未提交；
- 题目采用的是立即修改还是延迟修改。

核心判断主线是：
- **已提交事务**：重点考虑是否需要 **Redo**
- **未提交事务**：重点考虑是否需要 **Undo**
- **立即修改**：更容易同时出现 Undo 与 Redo
- **延迟修改**：未提交事务通常不必 Undo，但已提交事务仍可能要 Redo

#### 4）检查点的价值，不只是“提高效率”，而是“缩小分析范围”

检查点真正该写出来的，不是空泛的“效率更高”，而是：
- 不必从日志最开头全量扫描；
- 可以从最近检查点附近开始重点分析；
- 能更快锁定与故障相关的事务集合；
- 从而缩短恢复分析时间。

#### 5）WAL、日志、检查点、备份是“协同关系”，不是互相替代

很多综合题容易把这几个概念拆散来背。更稳的理解方式是：

- **WAL**：保证日志先于数据写入稳定存储，让恢复有依据；
- **日志**：记录事务关键操作，是恢复判断 Undo / Redo 的依据；
- **检查点**：帮助缩小恢复分析范围；
- **备份**：在介质故障时恢复数据库文件基础状态。

它们之间不是“谁替代谁”，而是：

> **WAL 保证日志可靠，日志支撑恢复判断，检查点缩小分析范围，备份解决介质损坏底座问题。**

#### 6）介质故障题的标准顺序与系统故障题不同

- **系统故障题**：通常重点写日志、检查点、提交状态、Undo / Redo
- **介质故障题**：通常重点写“先恢复备份 → 再利用日志重做备份之后已提交事务 → 未提交事务结果不保留”

因此，一看到“磁盘损坏、数据库文件本身不可用”，就不要再按普通系统故障的口径去答。

#### 7）下午题得分关键是“把依据写完整”

恢复综合题常见失分，不是大方向完全错，而是：
- 只写“Redo T1，Undo T2”，没写为什么；
- 只写“检查点提高效率”，没写提高在哪里；
- 只写“备份 + 日志恢复”，没写谁先谁后、为什么这样做；
- 只写过程，不写最后一句恢复目标。

所以今天的重点是：

> **让每道恢复题答案都尽量具备“对象、动作、依据、结论”四个层次。**

### 2.4 必备结论

- 恢复综合题的第一步应先判故障类型，而不是直接判断 Undo / Redo
- 检查点场景下，至少要关注“检查点时仍活跃的事务”和“检查点后新出现的事务”
- 已提交事务通常优先考虑 Redo，未提交事务通常优先考虑 Undo
- 立即修改下，系统故障后常更容易同时出现 Undo 与 Redo
- 延迟修改下，未提交事务通常不必 Undo，但已提交事务仍可能需要 Redo
- 检查点的核心作用是缩小恢复分析范围，而不是笼统地写“提高效率”
- WAL 的核心意义是：日志必须先于对应数据写入稳定存储，以保证崩溃后恢复有依据
- 系统故障通常更多依赖日志与检查点；介质故障通常要结合备份与日志恢复
- 介质故障题的标准顺序通常是：先恢复备份，再重做备份后已提交事务
- 恢复题的标准收束句应回到：**数据库应恢复到只保留已提交事务结果的正确状态**

### 2.5 高频考点拆解

#### 2.5.1 检查点场景下“恢复对象集合”是综合题第一高频点

**核心概念：** 检查点题不仅要看从哪里开始分析日志，还要看哪些事务必须纳入恢复判断。  
**常见考法：** 给出“检查点时活跃事务 + 检查点后事务活动”，问恢复时为什么要关注某些事务。  
**高频误区：** 只写“从检查点附近开始分析”，漏掉“检查点时仍活跃 + 检查点后新出现”这两类事务。  
**快速判断提示：** 先分两类事务，再谈日志范围。  
**小例子：** 检查点时 `T1、T2` 活跃，之后 `T3` 启动，则故障后至少要关注 `T1、T2、T3` 的状态。

#### 2.5.2 立即修改与延迟修改在综合题里不能只背定义

**核心概念：** 两种策略的关键差别是“提交前数据是否可能已写库”。  
**常见考法：** 给出系统崩溃场景，让你判断为什么有的事务需要 Undo，有的事务不需要。  
**高频误区：** 只写“立即修改 / 延迟修改”的名字，却不写它对恢复动作的影响。  
**快速判断提示：** 先问自己：未提交事务的结果，崩溃前有可能已经写到数据库里吗？  
**小例子：** 延迟修改下，未提交事务通常还未真正写库，因此一般不需要 Undo。

#### 2.5.3 介质故障题要敏感抓住“数据库文件本身不可用”

**核心概念：** 介质故障与系统故障的最大区别，在于数据库文件本身可能已经损坏。  
**常见考法：** 问“这是哪类故障”“如何恢复”。  
**高频误区：** 一看到恢复题就直接答 Undo / Redo，忽略了介质故障应先恢复备份。  
**快速判断提示：** 只要出现“磁盘损坏、数据文件不可用”，先想到备份。  
**小例子：** 晚上 22:00 有备份，22:00 之后事务继续运行，磁盘损坏后应先恢复 22:00 的备份，再根据日志补回之后已提交的事务。

#### 2.5.4 综合题的高频失分点是“动作有了，依据没写”

**核心概念：** 下午题评分看的是依据是否完整，而不是只看最终动作。  
**常见考法：** 给出日志片段和故障时刻，让你写恢复分析过程。  
**高频误区：** 只写“Redo T1，Undo T2”，不写“因为 T1 已提交、T2 未提交”。  
**快速判断提示：** 每个恢复动作后都补一句“为什么”。  
**小例子：** “T1 已有 `COMMIT`，其结果应保留；T2 无 `COMMIT`，其修改不应保留。”

#### 2.5.5 恢复机制协同题要写出“各自作用 + 为什么合在一起有效”

**核心概念：** WAL、日志、检查点、备份各自有不同分工。  
**常见考法：** 问“这些机制如何共同支持恢复”。  
**高频误区：** 只堆术语，不写关系。  
**快速判断提示：** 按“WAL 保日志可靠 → 日志给恢复依据 → 检查点缩范围 → 备份解介质故障”去写。  
**小例子：** 如果只有备份没有日志，就无法补回备份之后已提交事务的结果。

#### 2.5.6 恢复标准答案最后一定要收束到“恢复目标”

**核心概念：** 恢复题的最后一句不是装饰，而是结论收束。  
**常见考法：** 综合表达题、模板题、简答题。  
**高频误区：** 前面分析了一堆，最后没有说明数据库最终应恢复到什么状态。  
**快速判断提示：** 最后补一句“数据库应恢复到只保留已提交事务结果的正确状态”。  
**小例子：** 即使前面写对了所有 Undo / Redo，少掉这句仍可能丢掉结尾采分点。

### 2.6 下午题视角下的题型映射

| 下午题类型 | 常见问法 | 作答关键动作 | 常见丢分点 |
|---|---|---|---|
| 系统故障恢复综合题 | “哪些事务需要 Undo / Redo” | 先判故障类型，再定事务集合，再看提交状态与更新策略 | 只写 Undo / Redo，不写依据 |
| 检查点分析题 | “为什么要关注这些事务 / 从哪里开始恢复” | 先分活跃事务与新事务，再写从检查点附近分析 | 只写“提高效率”，不写事务集合 |
| 介质故障恢复题 | “如何恢复数据库” | 先恢复备份，再利用日志处理备份后事务 | 把介质故障答成普通系统故障 |
| 恢复机制协同题 | “WAL、日志、检查点、备份如何配合” | 分别写作用，再写为什么合起来有效 | 只堆概念，不写逻辑关系 |
| 综合表达题 | “写出完整恢复分析过程” | 故障类型 → 事务集合 → Undo / Redo → 恢复目标 | 句序乱、结尾漏写总结句 |

### 2.7 标准答题模板 / 采分点提示

#### 模板1：系统故障恢复综合题
1. 先写故障类型为系统故障。  
2. 再写恢复时应关注检查点时仍活跃的事务，以及检查点后新出现的事务。  
3. 再根据日志判断哪些事务已提交、哪些未提交。  
4. 若为立即修改，写“已提交事务通常需要 Redo，未提交事务通常需要 Undo”；若为延迟修改，说明未提交事务通常不必 Undo。  
5. 最后写恢复目标：数据库应恢复到只保留已提交事务结果的正确状态。

#### 模板2：介质故障恢复题
1. 先写故障类型为介质故障，数据库文件本身不可用。  
2. 再写应先利用最近备份恢复数据库基础状态。  
3. 然后根据日志重做备份之后已经提交的事务。  
4. 未提交事务的结果不应被保留。  
5. 最后总结：备份解决数据文件损坏，日志保证备份后已提交结果可恢复。

#### 模板3：恢复机制协同表达题
1. 先写当前风险：系统故障或介质故障可能导致未提交结果残留、已提交结果丢失。  
2. 再写 WAL 保证日志先于数据写入稳定存储，使恢复有依据。  
3. 再写日志用于判断哪些事务应 Undo / Redo，检查点用于缩小恢复分析范围。  
4. 若涉及介质故障，再补写备份用于恢复数据库文件基础状态。  
5. 最后写结论：这些机制共同作用，帮助数据库恢复到正确状态。

### 2.9 易混点速记卡片

- **系统故障**：数据库和日志通常还在  
  **介质故障**：数据库文件本身可能坏了

- **检查点时仍活跃事务**：必须继续看  
  **检查点后新出现事务**：也必须继续看

- **立即修改**：提交前数据可能已写库  
  **延迟修改**：提交前通常未真正写库

- **Redo**：补回本该留下的

**解析：** 立即修改与延迟修改在恢复题里的关键差异，是“提交前数据是否可能已经写库”。

### 第15题
**答案：D**  
**解析：** 给出检查点后，恢复分析通常至少要覆盖“检查点时仍活跃的事务”和“检查点后新出现的事务”。

### 第16题
**答案：A**  
**解析：** 数据库恢复的核心目标之一，是让数据库最终只保留已提交事务结果，回到一致的正确状态。

### 第17题
**答案：C**  
**解析：** 下午恢复综合题常见失分点之一，就是只写 Undo / Redo 的最终动作，却不写对应依据。

### 第18题
**答案：B**  
**解析：** Day 16 的核心推进，是从恢复基本判断，进一步进入恢复综合分析与案例表达。

### 第19题
**参考答案：**  
恢复分析时至少应重点关注 `T1、T2、T3`。其中，`T1、T2` 是检查点时仍活跃的事务，说明它们跨越了检查点，需要继续确认其后续状态；`T3` 虽然是在检查点后启动，但它在系统崩溃前已经执行，因此同样属于恢复分析对象。

**采分点：**
- 写出 `T1、T2` 是检查点时仍活跃的事务
- 写出 `T3` 是检查点后新出现并已执行的事务
- 能说明三者都需要纳入恢复分析范围

**作答提醒：**
- 先分事务集合
- 再写原因
- 不要只写“从检查点附近开始分析”

### 第20题
**参考答案：**  
`T1` 应优先考虑 **Redo**，因为它已经提交；`T2` 和 `T3` 应优先考虑 **Undo**，因为它们尚未提交。由于系统采用立即修改策略，未提交事务的修改可能已经部分写入数据库，因此需要撤销；已提交事务的结果应保留，因此需要考虑重做。

**采分点：**
- 写出 `T1` Redo
- 写出 `T2、T3` Undo
- 能说明依据是“提交状态 + 立即修改策略”

**作答提醒：**
- 先看是否提交
- 再结合更新策略
- 最后补为什么

### 第21题
**参考答案：**  
这道题通常不必从日志最开头全部扫描，是因为有最近检查点，可以从检查点附近开始重点分析，缩小需要扫描的日志范围。`T3` 虽然不是检查点时的活跃事务，但它是在检查点后启动并在系统崩溃前执行的事务，因此同样必须纳入恢复分析范围。

**采分点：**
- 写出“可从最近检查点附近开始分析”
- 写出“缩小日志扫描范围”
- 写出 `T3` 也属于恢复对象集合

**作答提醒：**
- 同时回答“从哪里开始看”和“哪些事务要看”

### 第22题
**参考答案：**  
本题属于系统故障恢复综合分析题。恢复时应先关注检查点时仍活跃的事务 `T1、T2`，以及检查点后新出现的事务 `T3`。根据日志可见，`T1` 已提交，而 `T2、T3` 尚未提交。由于系统采用立即修改策略，因此通常对 `T1` 进行 Redo，对 `T2、T3` 进行 Undo。恢复完成后，数据库应回到只保留已提交事务结果的正确状态。

**采分点：**
- 顺序完整：故障类型 → 事务集合 → Undo / Redo → 恢复目标
- 写出 `T1` Redo、`T2/T3` Undo
- 句子完整、依据清楚

**作答提醒：**
- 别漏最后一句恢复目标

### 第23题
**参考答案：**  
这更属于 **介质故障**。因为题目明确说明“磁盘损坏导致数据库文件本身不可用”，这已经不是单纯内存信息丢失的系统故障，而是存储介质层面的数据文件损坏。

**采分点：**
- 写出“介质故障”
- 写出“数据库文件本身不可用”这一依据
- 能区分它与普通系统故障不同

**作答提醒：**
- 看到磁盘损坏、数据文件不可用，要先想到介质故障

### 第24题
**参考答案：**  
恢复步骤通常可概括为：先利用最近一次完整备份恢复数据库基础状态；再读取备份之后的日志记录；然后把备份之后已经提交的事务结果重做回去；未提交事务的结果不应保留。这样才能把数据库恢复到正确状态。

**采分点：**
- 写出“先恢复备份”
- 写出“再根据日志处理备份之后事务”
- 写出“已提交保留、未提交不保留”的恢复方向

**作答提醒：**
- 介质故障题通常先写备份，再写日志

### 第25题
**参考答案：**  
仅靠 22:00 的备份还不足以完成这次恢复，因为备份只能把数据库恢复到 22:00 时的状态，无法直接补回 22:00 之后已经提交事务的结果。若没有日志，就无法判断并恢复备份之后那些应被保留的事务修改。

**采分点：**
- 写出“备份只能恢复到备份时刻状态”
- 写出“备份之后已提交事务还需要日志支持恢复”
- 说明为什么备份单独不够

**作答提醒：**
- 不要只写“需要日志”，要写“为什么需要”

### 第26题
**参考答案：**  
`T4` 的结果应被保留，因为它已经提交；在恢复过程中，应通过日志把它的结果重做回数据库。`T5` 的结果不应保留，因为它尚未提交，其修改不能作为最终正确结果留下。

**采分点：**
- 写出 `T4` 应保留
- 写出 `T5` 不应保留
- 能说明依据分别是“已提交 / 未提交”

**作答提醒：**
- 先分提交状态
- 再写保留或不保留

### 第27题
**参考答案：**  
这句话至少缺了两层关键意思：第一，恢复分析时不仅要从检查点附近开始扫描，还要明确哪些事务属于恢复对象，尤其是检查点时仍活跃的事务；第二，还要把检查点后新启动并在故障前执行的事务纳入恢复分析，而不是只谈日志扫描起点。

**采分点：**
- 写出“检查点时仍活跃的事务要纳入恢复分析”
- 写出“检查点后新出现的事务也要纳入恢复分析”
- 指出原句只回答了扫描起点，没回答恢复对象集合

**作答提醒：**
- 检查点题常常同时考“范围”和“对象”

### 第28题
**参考答案：**  
立即修改环境下，事务提交前数据也可能已经写入数据库，因此系统崩溃后，未提交事务可能需要 Undo，而已提交事务又可能需要 Redo，所以更容易同时出现两类恢复动作。延迟修改环境下，未提交事务的修改通常在提交前还未真正写库，因此对未提交事务往往不必 Undo。两者的关键差异，在于提交前数据是否可能已经写入数据库。

**采分点：**
- 写出立即修改下更容易同时出现 Undo 与 Redo
- 写出延迟修改下未提交事务往往不必 Undo
- 点出关键差异是“提交前数据是否可能已写库”

**作答提醒：**
- 比较题两边都要写到
- 最后再补一句关键差异

### 第29题
**参考答案：**  
数据库发生故障时，风险在于未提交事务的错误结果可能被留下，而已提交事务的正确结果也可能尚未完整落盘。WAL 保证日志先于数据写入稳定存储，使系统崩溃后恢复仍有依据；日志记录事务关键操作，用于判断哪些事务应 Undo、哪些事务应 Redo；检查点用于缩小恢复分析范围，提高恢复效率；而备份则在介质故障时提供数据库文件的基础恢复点。正因为这些机制分工不同、相互配合，数据库才可能在不同故障场景下恢复到正确状态。

**采分点：**
- 写出日志、检查点、WAL、备份各自作用
- 能说明为什么它们合起来有效
- 结尾回到恢复正确状态这一结论

**作答提醒：**
- 不要只堆术语
- 要写“为什么协同有效”

### 第30题
**参考答案：**  
回答恢复综合题时，首先应判断题目属于事务故障、系统故障还是介质故障。其次要明确哪些事务属于恢复分析对象，例如检查点时仍活跃的事务和检查点后新出现的事务。然后根据日志判断事务的提交状态，并结合更新策略分析哪些事务应 Undo、哪些事务应 Redo。最后说明恢复目标是使数据库回到只保留已提交事务结果的正确状态。

**采分点：**
- 顺序完整：故障类型 → 事务集合 → Undo / Redo → 恢复目标
- 写出“提交状态 + 更新策略”这一判断依据
- 结尾有标准恢复目标句

**作答提醒：**
- 模板题关键在句序稳定
- 别漏掉“事务集合”这一步
