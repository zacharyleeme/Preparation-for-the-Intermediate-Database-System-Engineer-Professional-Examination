### 2.2 为什么 Day 5 进入连接查询

Day 3 学的是单表基础查询，Day 4 学的是单表或分组统计。  
但真实考试里，很多 SQL 题并不会只给你一张表，而是会考：

- 学生表和选课表如何一起查
- 课程表和成绩表如何联合判断
- 哪些学生选了哪门课
- 哪门课程是谁教的

这就意味着：仅会对单表做 `SELECT / WHERE / GROUP BY` 还不够，必须继续进入**多表连接查询**。

原因主要有三点：

1. **连接查询是 SQL 综合题的核心骨架之一**
2. 上午选择题经常直接考连接类型、连接条件、结果行数或连接语义
3. 下午分析题、数据库设计补充问答、SQL 综合题，常常都离不开“多表关联”这一步

简单说：

- Day 3：学会从一张表里查数据
- Day 4：学会在一张表或若干分组里做统计
- Day 5：学会把**多张表关联起来查数据**

### 2.3 今日核心知识框架

连接查询最核心的问题，不是先背语法，而是先想清楚下面三个问题：

1. **要查哪些表**
2. **这些表靠什么字段关联**
3. **连接后要保留哪些记录、输出哪些列**

最基础的连接查询写法可先理解为：

```sql
SELECT 列名
FROM 表1, 表2
WHERE 表1.关联字段 = 表2.关联字段;
```

更规范、更常见的写法是：

```sql
SELECT 列名
FROM 表1
JOIN 表2
ON 表1.关联字段 = 表2.关联字段;
```

可以先用一句人话理解：

- `FROM / JOIN`：本次要把哪几张表放在一起
- `ON`：这些表之间按什么条件关联
- `SELECT`：最终显示哪些字段
- `WHERE`：连接完成后再做进一步筛选

思考顺序建议始终是：

1. 先确定主表和关联表
2. 再找关联字段
3. 再判断连接类型
4. 最后确定输出字段和附加条件

### 2.4 必备结论

- 多表查询的核心是：**表与表之间必须有明确的关联条件**
- 没有连接条件的多表查询，往往会形成**笛卡尔积**
- `JOIN ... ON ...` 用于表达连接关系
- **内连接（INNER JOIN）**：只保留能匹配上的记录
- **左外连接（LEFT JOIN）**：保留左表全部记录，右表匹配不上则补空值
- **右外连接（RIGHT JOIN）**：保留右表全部记录，左表匹配不上则补空值
- **自连接**：同一张表当作两张表来连，通常需要起别名
- 多表查询时，字段名前常需要带表名或别名，避免歧义
- 连接条件通常写在 `ON` 中；结果过滤条件通常写在 `WHERE` 中
- SQL 中的连接思想，与关系代数中的“连接运算”密切对应

### 2.5 高频考点拆解

#### 2.5.1 什么是连接查询

设有三张常见表：

- Student(学号, 姓名, 所在系)
- SC(学号, 课程号, 成绩)
- Course(课程号, 课程名, 学分)

如果只看 Student 表，只能知道学生信息；
如果只看 SC 表，只能知道某个学号选了某门课并取得多少分；
如果想知道“张三选了什么课、成绩是多少”，就必须把 Student 和 SC 关联起来。

例如：

```sql
SELECT Student.姓名, SC.课程号, SC.成绩
FROM Student
JOIN SC
ON Student.学号 = SC.学号;
```

这条语句的本质是：

- 先把 Student 和 SC 放在一起
- 按“学号相同”进行匹配
- 然后输出姓名、课程号、成绩

高频误区：

- 连接不是把两张表机械拼接
- 连接必须有“谁和谁对应”的条件
- 没有关联字段时，结果往往会爆炸式增加

#### 2.5.2 等值连接与内连接

最基础、最常见的连接类型就是按相等条件进行匹配：

```sql
SELECT Student.姓名, SC.成绩
FROM Student
JOIN SC
ON Student.学号 = SC.学号;
```

这里的核心就是：

```sql
Student.学号 = SC.学号
```

这种按相等条件匹配的连接，常称为**等值连接**。  
而在日常 SQL 题里，最常考的连接就是**内连接**。

内连接的特点：

- 只保留两边都能匹配上的记录
- 任一侧找不到对应记录，就不会出现在结果中

因此：

- “只要匹配成功的部分” → 优先想到内连接
- “要保留一边全部记录” → 优先想到外连接

#### 2.5.3 左外连接与右外连接

如果题目要求：

- 列出**所有学生**以及他们的成绩
- 即使某个学生没有成绩，也要显示出来

这时就不能只用内连接，因为没有成绩的学生在内连接里会被过滤掉。

应考虑左外连接：

```sql
SELECT Student.姓名, SC.成绩
FROM Student
LEFT JOIN SC
ON Student.学号 = SC.学号;
```

含义：

- 以 Student 为左表
- 左表的全部记录都保留
- SC 中匹配不上的部分显示为 `NULL`

同理：

- `LEFT JOIN`：保留左表全部
- `RIGHT JOIN`：保留右表全部

高频误区：

- 不要把“左表”“右表”理解成现实意义上的主次
- 左右只看 SQL 写法中谁在左、谁在右
- 外连接的核心不是“更多字段”，而是“保留哪一边的全部记录”

#### 2.5.4 自连接

有些题表面上只有一张表，但实际上要把一张表看成两个角色。

例如员工表：

- Employee(员工号, 姓名, 上级编号)

如果要查询“员工姓名及其上级姓名”，就要把 Employee 当成两张表使用：

```sql
SELECT e1.姓名 AS 员工姓名, e2.姓名 AS 上级姓名
FROM Employee e1
LEFT JOIN Employee e2
ON e1.上级编号 = e2.员工号;
```

这里：

- `e1` 表示“员工自己”
- `e2` 表示“员工的上级”

这就是**自连接**。

高频误区：

- 自连接不是特殊新语法，本质还是连接
- 关键在于给同一张表起不同别名
- 若不用别名，很难区分字段来自哪个角色

#### 2.5.5 连接条件与筛选条件不要混淆

看下面这条语句：

```sql
SELECT Student.姓名, SC.成绩
FROM Student
JOIN SC
ON Student.学号 = SC.学号
WHERE SC.成绩 >= 60;
```

要分清：

- `ON Student.学号 = SC.学号`：这是连接条件
- `WHERE SC.成绩 >= 60`：这是结果筛选条件

连接条件决定：**两张表怎样匹配**  
筛选条件决定：**匹配之后哪些记录保留**

考试很喜欢在这里出混淆题。

#### 2.5.6 连接与笛卡尔积的关系

如果写成：

```sql
SELECT *
FROM Student, SC;
```

而没有任何 `WHERE Student.学号 = SC.学号` 之类的条件，
结果就会变成：

- Student 的每一行
- 与 SC 的每一行
- 两两组合

这就是**笛卡尔积**。

如果 Student 有 10 行，SC 有 20 行，结果就可能变成 200 行。

因此必须记住：

- 多表一起出现，不代表已经连接成功
- 没有连接条件时，优先警惕笛卡尔积
- 连接查询题要第一时间找“关联字段”

### 2.6 SQL 与关系代数的对应直觉

Day 2 已经学过连接的关系代数思想，Day 5 要把它落到 SQL 写法上。

例如：

```sql
SELECT Student.姓名, SC.成绩
FROM Student
JOIN SC
ON Student.学号 = SC.学号;
```

可以理解为：

1. 先把 Student 与 SC 按学号对应起来
2. 得到连接结果
3. 再投影出“姓名、成绩”这两列

从直觉上说：

- `JOIN ... ON ...` ≈ 关系代数中的连接
- `SELECT 字段` ≈ 投影
- `WHERE 条件` ≈ 选择

也就是说，Day 5 本质上是在继续把 Day 2 的关系代数思想，翻译成考试中更高频的 SQL 形式。

### 2.7 易混点速记卡片

- **内连接**
  - 只保留匹配成功的记录
  - 关键词：交集直觉

- **左外连接**
  - 保留左表全部记录
  - 关键词：左边全保留

- **右外连接**
  - 保留右表全部记录
  - 关键词：右边全保留

- **连接条件 vs 筛选条件**
  - `ON`：谁和谁匹配
  - `WHERE`：匹配后保留哪些结果

- **连接 vs 笛卡尔积**
  - 有连接条件：正常关联
  - 无连接条件：行数暴增风险

- **自连接**
  - 一张表扮演两个角色
  - 关键词：同表不同别名

### 第1题

**答案：A**

### 第8题
**答案：A**
