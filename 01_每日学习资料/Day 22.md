### 2.3 今日核心知识框架

今天要固定下来的，不只是“看懂题”，而是把数据库设计题里的 **记录型关系模式判断与表达** 压成固定作答链。

#### 1）什么时候不能只加字段

> **先看会不会多次发生，再看要不要单独统计，再看是否需要保留历史。**

- 若同一主业务对象下，某类信息可能多次出现，如多次维修、多次支付、多种商品明细、多组收货地址；
- 若每次出现都要记录时间、金额、内容、状态、责任人等独立信息；
- 若后续还要做查询、统计、审计、对账、追踪；

则通常不能只加一个字段，更稳妥的是建立 **独立记录型关系模式**。

#### 2）订单头 / 订单明细判断路径

> **先分“订单整体信息”与“订单中每个商品的发生信息”。**

- 订单头信息：订单编号、顾客编号、下单时间、订单状态等；
- 订单明细信息：商品编号、数量、单价、小计等；
- 若一个订单可包含多个商品，则数量等信息通常不是订单静态属性，而是 **订单—商品关系中的明细属性**。

#### 3）主键 / 外键 / 组合键表达路径

> **先找唯一标识，再找引用方向，再检查是否需要组合键。**

- 主键用于唯一标识本表记录；
- 外键用于保持实体之间的引用关系；
- 若单个字段不能唯一标识，如“楼编号 + 公寓号”“订单编号 + 商品编号”，则要考虑组合键；
- 对记录型关系模式，还要分清：是采用“业务自然组合键”，还是增加“记录编号 / 支付编号 / 维修编号”这类人工主键。

#### 4）扩展需求改图改模路径

> **先判断新需求引入了什么新业务行为，再判断影响哪些对象 / 联系 / 关系模式。**

例如：
- “实体店销售也增加送货上门服务” → 不是只加一个标志字段，而是要考虑订单、配送、顾客 / 收货信息、下单主体与统一管理链条；
- “记录每次维修历史” → 不是只在故障表里加维修内容，而是要新增维修记录；
- “记录详细支付情况” → 不是只在快递表里加金额，而是要新增支付记录模式。

#### 5）数据库设计题标准收束句路径

> **最后一句不只写“便于管理”，而要写到“反映业务语义 + 支持查询统计 + 统一管理 + 扩展性”。**

标准收束方向可固定成：
- 更能真实反映业务对象与业务行为；
- 更利于后续查询、统计、对账、追踪；
- 更适合和其他业务场景统一管理；
- 更有利于后续扩展，而不是把复杂信息硬塞回主表。

### 2.4 必备结论

- 只要题干出现“多次维修 / 多次支付 / 多组地址 / 多个商品明细”，就要优先警惕是否需要独立关系模式。
- 记录型关系模式的判断核心，不是“看起来复杂”，而是“会不会多次发生、要不要保留历史、要不要单独统计”。
- 订单头信息与订单明细信息通常不能混成一张只按订单编号唯一的表。
- 商品数量通常属于订单—商品这次业务发生时的属性，更适合放在订单明细或联系模式中。
- 支付记录类题目里，支付编号更适合作为唯一标识之一；快递编号更适合作为关联外键之一。
- 维修记录类题目里，不能只保留故障内容，要能表达“每次维修”的独立信息。
- 扩展需求题必须同时考虑 **ER 图层面的业务关系** 与 **关系模式层面的结构变化**。
- 主键、外键、组合键、联系类型、联系属性，下午题里最好显式写出，不要假定阅卷人会替你补全。
- 标准答案不只要写“怎么设计”，还要写“为什么不能只按原方式设计”。
- 收束句尽量写到：**更能反映业务语义，并支持后续查询、统计、扩展与统一管理。**

### 2.5 高频考点拆解

#### 2.5.1 多次发生信息何时独立建模

**核心概念：** 同一主业务对象下某类记录会重复发生，且每次都带独立信息。  
**常见考法：** 多次维修、快递费支付、订单商品明细、配送记录。  
**高频误区：** 只在原表上多加一个字段。  
**快速判断提示：** 看是否要记录“时间、金额、内容、状态、责任人”等每次不同的信息。  
**小例子：** 快递费支付（支付编号，快递编号，支付金额，支付时间，支付方式，付款方）。

#### 2.5.2 订单头与订单明细别混表

**核心概念：** 订单整体属性和订单中的每个商品明细不是同一层级。  
**常见考法：** 一份订单可包含多个商品，并记录数量、价格、小计。  
**高频误区：** 在订单表里直接加商品字段或多个商品列。  
**快速判断提示：** 先问：这个属性属于“整张订单”，还是属于“订单里的某个商品项”？  
**小例子：** 订单（订单编号，顾客编号，下单时间，订单状态）；订单明细（订单编号，商品编号，数量，单价）。

#### 2.5.3 主键外键表达要落到具体字段

**核心概念：** 下午题不只要说“有关联”，还要写清“谁作主键，谁作外键”。  
**常见考法：** 支付记录、维修记录、合同模式、收货地址模式。  
**高频误区：** 只写新增模式名称，不写关键字段与参照方向。  
**快速判断提示：** 先写本表唯一标识，再写引用了谁。  
**小例子：** 支付记录中“支付编号”为主键，“快递编号”为外键。

#### 2.5.4 扩展需求要写到统一管理链条

**核心概念：** 扩展需求引入的是新业务链，而不只是新字段。  
**常见考法：** 实体店也做送货上门、增加维修历史、增加多方式支付。  
**高频误区：** 只写“增加配送字段 / 增加支付字段”，没有写到统一订单、配送、统计流程。  
**快速判断提示：** 看新需求是否会改变后续下单、配送、结算、统计、查询流程。  
**小例子：** 实体店送货上门后，销售记录不再只是线下完成，而要纳入订单与配送统一管理。

#### 2.5.5 收束句必须体现设计价值

**核心概念：** 数据库设计题最后一句是得分点，不是礼貌性结尾。  
**常见考法：** “这样设计有什么好处”“为什么更合理”。  
**高频误区：** 只写“便于管理”。  
**快速判断提示：** 至少补 2 个价值：查询 / 统计 / 统一管理 / 扩展 / 真实反映业务语义。  
**小例子：** “这样既能真实反映业务对象与业务行为，也便于后续查询、统计与统一管理。”

### 2.7 标准答题模板 / 采分点提示

#### 模板1：记录型关系模式判断题
1. 先写该信息会在同一主业务对象下多次发生。  
2. 再写每次发生都要记录独立信息，如时间、金额、内容、状态。  
3. 说明若只加一个字段，会丢失历史或不利于统计。  
4. 再给出新增关系模式的核心字段。  
5. 最后补一句：这样更利于后续查询、统计、追踪与统一管理。

#### 模板2：订单头 / 订单明细题
1. 先区分订单整体信息与订单明细信息。  
2. 说明一份订单可包含多个商品，因此不能只按订单编号唯一压成一张表。  
3. 给出订单头模式与订单明细模式的核心字段。  
4. 说明数量、单价等信息属于明细层。  
5. 最后补一句：这样更符合业务语义，也便于后续统计与配送管理。

#### 模板3：扩展需求统一管理题
1. 先说明扩展需求引入了新的业务行为或业务链条。  
2. 再写受影响的核心对象 / 联系 / 关系模式。  
3. 说明为什么不能只加一个标志字段。  
4. 给出新增模式或修改后的结构方向。  
5. 最后补一句：这样才能把不同来源的业务纳入统一管理与后续查询统计。

### 2.9 易混点速记卡片

- **联系属性**：某次业务发生时的信息  
  **独立关系模式**：同一主业务下可多次发生、需单独记录和统计的信息

  **订单明细**：订单里每个商品项的信息

- **主键**：唯一标识本表记录  
  **外键**：指向其他关系模式的引用字段

- **组合键**：单字段不够唯一时的联合标识  
  **人工主键**：如支付编号、维修编号等记录标识

- **只加字段**：结构通常不够稳定  
  **独立建模**：更利于保留历史、统计分析、统一管理

#### 第30题（案例组 3 / 小问 4）

**题型归属：** 通用模板复写 / 数据库设计题综合标准句序  
**来源：2016～2023 真题主线改编（数据库设计通用模板）**

请用“**先抽主业务对象 → 再识别记录型关系模式 → 再写主键外键与明细结构 → 最后写统一管理与统计价值**”的顺序，写一段 **4～6 句** 的数据库设计题通用标准答案。

**作答区：**

- 然后检查相关实体对象是否会有一对多联系，是否需要保留历史数据

- 总结这样设计可以更好反应业务语义，有利于长期查询统计，和统一管理
