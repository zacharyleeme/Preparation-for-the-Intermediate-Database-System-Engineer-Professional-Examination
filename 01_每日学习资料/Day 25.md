### 2.3 今日核心知识框架

今天要固定下来的，是一条适用于上午选择题与下午 SQL 小问的**统一下手链路**：

> **先看题目要什么结果 → 再判断这是 DDL / 查询 / 插入删除 / 视图哪一类 → 再选主表与连接 → 再补筛选与分组 → 最后检查约束、空值、重复值与结果边界。**

#### 1）先分清这题是在考“建表”还是“查数据”

SQL 题最容易乱的第一步，就是把不同任务混着做。你要先分清：

- 题目是在补 **`CREATE TABLE` / 约束**？
- 还是在写 **`SELECT` 查询**？
- 还是在写 **`INSERT ... SELECT` / `DELETE` / `UPDATE`**？
- 还是在写 **视图**？

只要类型分错，后面语法再熟也会跑偏。

#### 2）DDL 题先盯“唯一标识、引用关系、是否允许为空”

遇到 `CREATE TABLE` 或补约束题，最稳的顺序是：

1. 先找哪几个字段唯一标识一条记录；
2. 再看哪个字段引用别的表；
3. 再看哪些字段必须非空；
4. 再看数值范围、状态值范围等是否需要 `CHECK`；
5. 最后才是补默认值或其他细节。

也就是：

> **先主键，再外键，再非空，再检查条件。**

#### 3）查询题先定“输出粒度”

写 SQL 查询时，最容易犯的错误不是语法拼错，而是**没先定结果行到底代表什么**。

例如：
- 结果行代表“每个学生 1 行”？
- 还是“每门课 1 行”？
- 还是“每个部门 1 行”？
- 还是“每个订单明细 1 行”？

只要输出粒度没定稳，后面就容易：
- 连错表
- 多计或少计
- `GROUP BY` 写错
- 聚合字段和非聚合字段混乱

#### 4）连接题先选主表，再选连接方式

最稳的判断顺序：

1. 题目最想保留哪一侧的记录？
2. 那一侧通常就是主表 / 左表；
3. 如果只要匹配记录，用 `INNER JOIN`；
4. 如果要“即使没有匹配也保留”，用 `LEFT JOIN`；
5. 若用了 `LEFT JOIN`，就要特别小心：**不要再把右表筛选条件随手写进 `WHERE`，否则容易把外连接写回内连接效果。**

#### 5）统计题先分清 `WHERE` 和 `HAVING`

高频稳定结论：
- **`WHERE`**：分组前筛选行
- **`HAVING`**：分组后筛选组

只要题目出现“每个部门人数大于 3”“平均成绩不低于 80”这类话，就要优先联想到：

> **先 `GROUP BY`，再 `HAVING`。**

#### 6）“没有记录”类题优先想 `NOT EXISTS` 或外连接判空

常见问法：
- 没有选课的学生
- 没有员工的部门
- 没有参加社团的成员
- 没有发生交易的客户

这类题最稳的两个入口：
1. `NOT EXISTS`
2. `LEFT JOIN ... WHERE 右表主键 IS NULL`

其中：
- 想强调“**不存在匹配记录**”，`NOT EXISTS` 很稳；
- 想顺便保留左表并观察匹配情况，`LEFT JOIN` 更直观。

#### 7）插入与视图题，先想“来源结果是什么”

`INSERT ... SELECT` 的核心，不是 `INSERT`，而是：

> **你到底要把哪一个查询结果批量写进目标表。**

而视图题要先稳住：

> **视图保存的是查询定义，不是物理复制一份数据表。**

### 2.4 必备结论

- `PRIMARY KEY` 用于唯一标识元组，且不能为空。
- `FOREIGN KEY` 的核心作用，是维护引用表与被引用表之间的参照完整性。
- 需要限制数值范围或状态范围时，优先想到 `CHECK`。
- 题目要求“保留左侧所有记录”时，优先考虑 `LEFT JOIN`。
- `WHERE` 在分组前筛选行，`HAVING` 在分组后筛选组。
- “没有相关记录”的题，优先想到 `NOT EXISTS` 或 `LEFT JOIN ... IS NULL`。
- `INSERT INTO ... SELECT ...` 用于把查询结果批量插入目标表。
- 视图保存的是查询定义，不是重新复制一张物理表。
- `UPDATE` / `DELETE` 若缺少 `WHERE`，通常会影响全部记录，必须格外谨慎。
- SQL 下午题的最稳句序是：**先定输出粒度，再定表连接，再补筛选，最后检查分组、空值与重复值边界。**

### 2.5 高频考点拆解

#### 2.5.1 `PRIMARY KEY`、`FOREIGN KEY`、`NOT NULL`、`CHECK` 的边界

**核心概念：** 这四类约束分别控制“唯一标识”“引用关系”“是否允许空值”“取值范围 / 规则”。  
**常见考法：** 补 `CREATE TABLE`、补完整性约束、判断哪种约束最合适。  
**高频误区：** 把“必须存在于另一张表”的关系误写成 `NOT NULL`，或把“成绩 0～100”误写成主键问题。  
**快速判断提示：** 先问自己：这是在管“唯一”“引用”“空值”还是“范围”。  
**小例子：** `Grade BETWEEN 0 AND 100` 更像 `CHECK`，`SC.Sno` 引用 `Student.Sno` 更像 `FOREIGN KEY`。

#### 2.5.2 `LEFT JOIN` 不是“写了就高级”，而是“真的要保留左表行”

**核心概念：** `LEFT JOIN` 的价值，是即使右表没有匹配，也保住左表当前行。  
**常见考法：** “列出所有部门及人数（包括 0 人部门）”“列出所有课程及选课人数（包括 0 人课程）”。  
**高频误区：** 明明要保留空匹配行，却用了 `INNER JOIN`；或者先用了 `LEFT JOIN`，后面又在 `WHERE` 里把右表过滤掉。  
**快速判断提示：** 题目只要出现“包括没有……的”，就先看是否要 `LEFT JOIN`。  
**小例子：** “所有部门及员工数（含没有员工的部门）”最稳是 `Dept LEFT JOIN Emp`。

#### 2.5.3 统计题先分组，再 `HAVING`

**核心概念：** 聚合条件不属于单行条件，而属于分组后的结果条件。  
**常见考法：** “每个学生选课门数”“平均工资大于 8000 的部门”“参加社团人数不少于 2 的系”。  
**高频误区：** 把 `COUNT(*) > 3` 或 `AVG(Salary) > 8000` 写进 `WHERE`。  
**快速判断提示：** 只要条件里有聚合函数，就优先检查是不是该放在 `HAVING`。  
**小例子：** `GROUP BY DeptNo HAVING COUNT(*) >= 2`。

#### 2.5.4 “没有记录”类 SQL，要稳住两个入口

**核心概念：** 判断“缺失匹配”的本质，是判断某条相关记录是否不存在。  
**常见考法：** 没有选课的学生、没有员工的部门、没有加入社团的成员。  
**高频误区：** 用普通连接后再硬拼条件，导致逻辑混乱；或者把“没有”写成了“统计等于 0”但没分清统计粒度。  
**快速判断提示：** 优先从 `NOT EXISTS` 与 `LEFT JOIN ... IS NULL` 两条路里选更顺手的一条。  
**小例子：** `WHERE NOT EXISTS (SELECT 1 FROM SC WHERE SC.Sno = Student.Sno)`。

#### 2.5.5 `INSERT ... SELECT` 的关键，不是写 `INSERT`，而是先写对 `SELECT`

**核心概念：** 目标表接收的是一个查询结果集。  
**常见考法：** 将符合条件的员工插入某张汇总表、将某月数据导入备份表。  
**高频误区：** 只想着目标表字段，不先确认来源查询的列顺序与筛选条件。  
**快速判断提示：** 先独立写出 `SELECT`，再把它套进 `INSERT INTO ...`。  
**小例子：** `INSERT INTO HighSalary(EmpNo, Ename) SELECT EmpNo, Ename FROM Emp WHERE Salary > 8000;`

#### 2.5.6 视图题先稳住“保存定义，不复制数据”

**核心概念：** 视图本质是一个命名查询结果的定义。  
**常见考法：** 建立统计视图、简化复杂查询、限制用户访问列。  
**高频误区：** 把视图当成新建一张实际存储全部数据的物理表。  
**快速判断提示：** 先问自己：题目是要“方便后续查询 / 封装逻辑”，还是要“真正存数据”。前者更像视图。  
**小例子：** `CREATE VIEW V_ClubMemberCount AS SELECT ...`。

### 2.7 标准答题模板 / 采分点提示

#### 模板1：`CREATE TABLE` / 关键约束补全题
1. 先写清哪几个字段唯一标识一条记录。  
2. 再写主键：单主键还是复合主键。  
3. 再写引用关系：哪些字段要 `REFERENCES` 哪张表。  
4. 再写必须非空的字段与取值范围。  
5. 最后检查：是否把“唯一、引用、非空、范围”四类约束都覆盖了。

#### 模板2：连接 + 查询题
1. 先看结果要输出哪些字段。  
2. 再确定这些字段分别来自哪几张表。  
3. 根据主外键关系写 `JOIN`。  
4. 再补 `WHERE` 行筛选条件。  
5. 最后检查：是否多连表、漏连表、漏条件。

#### 模板3：分组统计 / 无记录判断题
1. 先写结果粒度：每个谁 1 行。  
2. 若题目要求统计，再写 `GROUP BY`。  
3. 若题目要求“人数不少于 / 平均值不低于”，再写 `HAVING`。  
4. 若题目要求“没有……”，优先改写为 `NOT EXISTS` 或 `LEFT JOIN ... IS NULL`。  
5. 最后检查：统计口径是否去重、空匹配是否保留、是否误把聚合条件写进 `WHERE`。

### 第1题
**答案：A**  
**解析：** `PRIMARY KEY` 的核心作用是唯一标识元组，且不能为空。`FOREIGN KEY` 管引用关系，`CHECK` 管取值规则，`DEFAULT` 只是默认值。

### 第2题
**答案：B**  
**解析：** `SC.Sno` 必须来自 `Student.Sno`，这是典型的参照完整性约束，应使用 `FOREIGN KEY`。

### 第3题
**答案：C**  
**解析：** 限制成绩范围 0～100，最直接的约束是 `CHECK (Grade BETWEEN 0 AND 100)`。这不是主键问题，也不是引用关系问题。

### 第4题
**答案：A**  
**解析：** 题干明确要求“包括没有员工的部门”，因此必须保留 `Dept` 的全部行，最稳妥是 `Dept LEFT JOIN Emp`。

### 第5题
**答案：C**  
**解析：** `COUNT(*) > 3` 属于聚合后的条件，应该写在 `HAVING` 中，而不是 `WHERE` 中。

### 第6题
**答案：B**  
**解析：** “每个学生的选课门数”说明结果粒度是“每个学生 1 行”；“未选课学生也要保留”说明应从 `Student` 出发做 `LEFT JOIN`，再按学生分组。

### 第7题
**答案：D**  
**解析：** “没有任何选课记录”的本质是“不存在匹配的 `SC` 记录”，`NOT EXISTS` 是最稳的入口之一。

### 第8题
**答案：B**  
**解析：** `INSERT INTO ... SELECT ...` 用于把一个查询结果集批量写入目标表，是下午题高频写法。

### 第9题
**答案：C**  
**解析：** 视图本质是查询定义，不是重新复制一张物理表。它常用于封装复杂查询或限制访问范围。

### 第10题
**答案：A**  
**解析：** 子表 `SC` 仍引用该学生时，直接删除 `Student` 中对应记录会优先触发参照完整性问题，这是本题最关键的检查点。

### 第12题
**答案：D**  
**解析：** `UPDATE` 缺少 `WHERE` 时，通常会影响所有记录，所以写更新语句时必须特别注意筛选条件是否完整。

### 第13题
**答案：C**  
**解析：** `LEFT JOIN` 的价值是保留左表空匹配行；如果把右表条件直接写进 `WHERE`，就可能把这些本来要保留的空匹配行又过滤掉。

### 第14题
**答案：B**  
**解析：** 平均工资属于聚合结果，因此应先 `GROUP BY Dno`，再用 `HAVING AVG(Salary) > 8000` 过滤。

### 第15题
**答案：D**  
**解析：** `UNIQUE` 用于限制重复值，但它不等同于主键；`PRIMARY KEY` 还承担“唯一标识元组”的角色，并且不能为空。

### 第16题
**答案：A**  
**解析：** `COUNT(*)` 最稳的理解，是统计满足查询条件后的结果行数；它经常与 `GROUP BY` 联用。

### 第17题
**答案：C**  
**解析：** SQL 下午题最不应跳过的第一步，是先确定结果粒度与输出字段。否则后面连表、筛选、分组都容易一起漂。

### 第19题
**参考答案：**
```sql
CREATE TABLE SC (
    Sno CHAR(10) NOT NULL,
    Cno CHAR(10) NOT NULL,
    Grade INT CHECK (Grade BETWEEN 0 AND 100),
    PRIMARY KEY (Sno, Cno),
    FOREIGN KEY (Sno) REFERENCES Student(Sno),
    FOREIGN KEY (Cno) REFERENCES Course(Cno)
);
```

**采分点：**
- 写出复合主键 `(Sno, Cno)`
- 写出两个外键引用关系
- 写出 `Grade` 的取值约束或非空意识

**作答提醒：**
- 先主键，再外键，再范围约束
- 这题重点不是数据类型花样，而是约束方向写对

### 第20题
**参考答案：**
```sql
SELECT S.Sname, C.Cname, SC.Grade
FROM Student S
JOIN SC ON S.Sno = SC.Sno
JOIN Course C ON SC.Cno = C.Cno
WHERE SC.Grade >= 80;
```

**采分点：**
- 输出字段来自三张表：学生名、课程名、成绩
- 写出两条连接条件
- 写出 `Grade >= 80` 的筛选条件

**作答提醒：**
- 先确定结果字段
- 再确定要连接哪几张表
- 再补成绩筛选条件

### 第21题
**参考答案：**
```sql
SELECT S.Sno, S.Sname, COUNT(SC.Cno) AS CourseCount
FROM Student S
LEFT JOIN SC ON S.Sno = SC.Sno
GROUP BY S.Sno, S.Sname;
```

**采分点：**

- 结果粒度是“每个学生 1 行”

**作答提醒：**

DELETE FROM Student S

**参考答案：**
```sql
SELECT D.Dno, D.Dname, COUNT(E.EmpNo) AS EmpCount
FROM Dept D
LEFT JOIN Emp E ON D.Dno = E.Dno
GROUP BY D.Dno, D.Dname;
```

**采分点：**
- 从 `Dept` 出发做 `LEFT JOIN`
- 统计员工人数
- 保留没有员工的部门

**作答提醒：**
- “包括没有员工的部门”是最强提示词
- 先保住左表，再统计右表匹配数

### 第24题
**参考答案：**
```sql
INSERT INTO HighSalary (EmpNo, Ename)
SELECT EmpNo, Ename
FROM Emp
WHERE Salary > 8000;
```

**采分点：**
- 写出 `INSERT INTO 目标表`
- `SELECT` 出符合条件的员工号与姓名
- 写出工资筛选条件

**作答提醒：**
- 先独立把 `SELECT` 写对
- 再把它包装到 `INSERT INTO ...` 里

### 第25题
**参考答案：**  
`INNER JOIN` 只会保留两张表中能够成功匹配的记录，因此没有员工的部门会直接消失。  
题目既然要求“包括没有员工的部门”，就必须先把部门表全部保留下来，因此更稳妥的是从 `Dept` 出发使用 `LEFT JOIN`。  
这样即使员工侧没有匹配记录，部门行仍然能保留，只是员工侧字段为 `NULL`。

**采分点：**
- 说明 `INNER JOIN` 会丢失无匹配部门
- 说明 `LEFT JOIN` 可以保留左表全部记录
- 提到右表无匹配时对应字段为 `NULL`

**作答提醒：**
- 这题重点是把“为什么用外连接”说成可得分短句
- 不要只写“因为外连接更合适”这种空话

### 第26题
**参考答案：**
```sql
SELECT D.Dno, D.Dname
FROM Dept D
LEFT JOIN Emp E ON D.Dno = E.Dno
WHERE E.EmpNo IS NULL;
```

**采分点：**
- 先 `LEFT JOIN`
- 再用 `WHERE E.EmpNo IS NULL` 判断无匹配
- 输出部门号与部门名

**作答提醒：**
- 这题和第23题是一组：第23题保留全部部门，第26题进一步筛出“没有员工”的部门

### 第27题
**参考答案：**
```sql
CREATE VIEW V_ClubMemberCount AS
SELECT C.ClubNo, C.ClubName, COUNT(J.Sno) AS MemberCount
FROM Club C
LEFT JOIN JoinClub J ON C.ClubNo = J.ClubNo
GROUP BY C.ClubNo, C.ClubName;
```

**采分点：**
- 使用 `CREATE VIEW`
- 从 `Club` 出发做 `LEFT JOIN`
- 按社团分组统计人数
- 保留没有成员的社团

**作答提醒：**
- 视图题先稳住“保存查询定义”
- 统计题仍要先定结果粒度：每个社团 1 行

### 第28题
**参考答案：**
```sql
SELECT M.Sno, M.Sname
FROM Member M
WHERE NOT EXISTS (
    SELECT 1
    FROM JoinClub J
    WHERE J.Sno = M.Sno
);
```

**采分点：**
- 使用 `NOT EXISTS`
- 内外层通过 `Sno` 正确关联
- 输出成员学号与姓名

**作答提醒：**
- “没有参加任何社团”本质上就是“不存在匹配的 `JoinClub` 记录”

### 第29题
**参考答案：**
```sql
SELECT M.DeptNo, COUNT(DISTINCT M.Sno) AS MemberCount
FROM Member M
JOIN JoinClub J ON M.Sno = J.Sno
GROUP BY M.DeptNo
HAVING COUNT(DISTINCT M.Sno) >= 2;
```

**采分点：**
- 按 `DeptNo` 分组
- 对参加社团的学生按学号去重计数
- 使用 `HAVING` 过滤人数不少于 2 的系别

**作答提醒：**
- 题干说“按学号去重”，就要检查是否需要 `DISTINCT`
- 题干说“人数不少于 2”，就要优先想到 `HAVING`

### 第30题
**参考答案：**  
做 SQL 下午题时，第一步要先看结果到底要按什么粒度输出，也就是每一行代表“每个学生”“每个部门”还是“每个社团”。  
第二步再根据结果字段来自哪几张表，确定主表、连接关系以及该用 `INNER JOIN` 还是 `LEFT JOIN`。  
第三步补行级筛选条件，凡是单条记录层面的限制优先放在 `WHERE`。  
如果题目还要求统计或比较聚合结果，再补 `GROUP BY` 与 `HAVING`。  
最后要检查是否误丢了空匹配记录、是否需要去重，以及结果字段与分组粒度是否一致。

**采分点：**
- 写出“输出粒度”作为第一步
- 写出主表 / 连接判断
- 写出 `WHERE`、`GROUP BY`、`HAVING` 的顺序
- 写出结果检查意识

**作答提醒：**
- 这是今天最重要的标准句序题
- 写得短没关系，但顺序一定要稳
