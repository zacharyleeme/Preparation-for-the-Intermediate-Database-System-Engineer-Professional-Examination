### 2.3 今日核心知识框架

今天要固定下来的，是一条比 Day 23 更完整的**规范化综合题标准链路**：

> **先列函数依赖 → 求候选键 → 判 2NF / 3NF / BCNF → 找出问题依赖 → 给出分解方案 → 判无损连接 → 判保持函数依赖 → 做综合评价。**

#### 1）为什么“会判范式”之后还必须继续看分解质量

Day 23 主要解决的是：
- 候选键是谁；
- 哪些属性是主属性 / 非主属性；
- 是否存在部分依赖、传递依赖；
- 最高大致到 2NF / 3NF 哪一层。

但考试不会停在“它最高到几范式”。下午题真正拉分的地方通常还会继续问：
- **那你打算怎么分？**
- **这样分有没有把原关系搞坏？**
- **原来的依赖还能不能继续检查？**
- **3NF 与 BCNF 的方案，为什么有时不能等同处理？**

所以 Day 24 的重点，不是再背一遍定义，而是把前一天的判断结果推进成“**可落笔的分解质量判断**”。

#### 2）二元无损连接，第一眼先看公共属性

无损连接（Lossless Join）最常见的下午题判定口径，仍然是二元分解：

若 `R` 被分解为 `R1` 和 `R2`，则通常先看：

> **`R1 ∩ R2` 是否能函数决定 `R1` 或 `R2` 的全部属性。**

最稳的人话写法是：

> **公共属性若能定住其中一边，这个二元分解通常可判为无损连接。**

这也是 Day 23 第 13 题必须先补做的原因——它是今天后续所有“分解是否可靠”的桥梁动作。

#### 3）保持函数依赖，不是“看起来差不多”就算保留

保持函数依赖（Dependency Preservation）看的不是“感觉这些属性还在”，而是：

> **原关系模式中的关键依赖，在分解后的子模式里，是否还能被直接表达，或者由投影依赖联合推出。**

最稳的判断顺序是：
1. 先把原依赖逐条列出；
2. 再看每条依赖在分解后的哪个子模式里被完整保留；
3. 若没被完整保留，再看能否由各子模式投影依赖联合推出；
4. 若关键依赖被拆散且无法恢复，就不能写成“保持函数依赖”。

#### 4）3NF 与 BCNF 的区别，要落到“左部是不是超键”

Day 23 已经让你开始接触 2NF / 3NF 的判断；Day 24 继续往上，就是 **BCNF（巴斯-科德范式）**。

最稳的记法：
- **3NF**：更强调消除部分依赖、传递依赖，并尽量兼顾依赖保持；
- **BCNF**：更严格，只要存在某条**非平凡依赖**，其左部不是超键，就不能通过。

所以判断 BCNF 时，最常见的失分不是不会定义，而是：
- 没先把候选键求对；
- 没分清“左部是不是超键”；
- 忘了 3NF 有时会放行“右部是主属性”的情况，而 BCNF 不会。

#### 5）为什么“满足 3NF”不等于“满足 BCNF”

这是今天必须彻底稳住的高频坑点。

若某依赖 `X → A` 中：
- `X` 不是超键；
- 但 `A` 恰好是主属性；

那么该关系模式**可能仍满足 3NF**，但**不满足 BCNF**。

所以考试里一旦出现“**3NF 但不满足 BCNF**”这种题，不要慌，先做两件事：
1. 求出全部候选键；
2. 看被依赖出来的右部属性是不是主属性。

#### 6）联系属性不能塞回实体表，这一点今天要继续压实

Day 23 最明显的一条结构性错误，是把“某人在哪个小区负责什么工作”这类**联系属性**，错放回了“人员信息表”。

Day 24 必须继续稳住这条规则：

> **只要某个属性描述的是“两个实体在某个业务场景下发生的角色、职责、数量、明细、状态”，它更可能属于联系关系，而不是某个实体的静态属性。**

例如：
- 订单中的“数量”更像订单—商品明细关系属性；
- 项目中的“分工角色”更像项目—成员分工关系属性；
- 人员在某小区的“负责工作”更像人员—小区关系属性。

#### 7）链式分解的无损说明，不写“三表总交集”

多表分解时，当前最要避免的写法是：

> `R1 ∩ R2 ∩ R3 → Ri`

这种写法太粗，会把本应逐段说明的链路偷换掉。

更稳的写法是：
- 先说明 `R1` 与 `R2` 的公共属性及依赖依据；
- 再说明 `R2` 与 `R3` 的公共属性及依赖依据；
- 若还更长，再继续逐段说明；
- 最后补一句“该分解沿原依赖链展开，可视为合理无损”。

#### 8）下午题最稳的标准句序

今天模板题仍然要强制按下面顺序写：

1. 先写原依赖 / 原关系问题
2. 再写候选键或最高范式判断
3. 再写分解结果
4. 再判保持函数依赖
5. 再判无损连接
6. 最后写“减少冗余 / 避免异常 / 结构更清晰 / 便于维护”等价值结论

只要句序稳，哪怕一两句没写满，过程分也更容易保住。

### 2.4 必备结论

- 二元分解判无损连接时，优先看 **公共属性能否决定某个子模式全部属性**。
- **有公共属性** 不等于 **一定无损**；关键是公共属性能不能“定住一边”。
- 保持函数依赖必须逐条核对原依赖是否在分解后被保留或可推出，不能凭感觉写“还在”。
- 3NF 分解通常更容易兼顾 **无损连接 + 保持函数依赖 + 答题可解释性**。
- BCNF 比 3NF 更严格，核心口径是：**每个非平凡依赖的左部都应是超键**。
- “满足 3NF”不一定“满足 BCNF”；若左部不是超键但右部是主属性，3NF 可能仍通过。
- 多个单属性候选键，不应误写成一个组合候选键。
- 联系属性属于业务关系本身，不能随手塞回某个实体静态信息表。
- 链式分解说明无损时，应按相邻子模式逐段写，不要偷换成三表总交集。
- 下午题模板结尾最好补到：**减少冗余、降低更新异常、提高一致性、便于维护**。

### 2.5 高频考点拆解

#### 2.5.1 多个单属性候选键，最容易被误写成一个组合键

**核心概念：** 若题干明确给出两个属性都能单独唯一标识元组，那么它们是两个候选键，而不是一个必须组合起来的键。  
**常见考法：** 患者身份证号 / 医保卡号类题、编号唯一类题。  
**高频误区：** 为了“保险”，把两个已经各自唯一的属性强行写成组合候选键。  
**快速判断提示：** 先问自己：这个属性单独能不能推出全部属性？若能，它本身就是候选键。  
**小例子：** 身份证号唯一、医保卡号唯一，则候选键至少是 **身份证号、医保卡号** 两个单独候选键。

#### 2.5.2 “3NF 但不满足 BCNF”是下午题高频陷阱

**核心概念：** 3NF 和 BCNF 的通过门槛不同。  
**常见考法：** 给出 `AB → C`、`C → B` 这类依赖，让你判断最高范式。  
**高频误区：** 一看到左部不是超键，就直接写“连 3NF 都不满足”。  
**快速判断提示：** 先求候选键，再看依赖右部是不是主属性。  
**小例子：** 若候选键为 `AB`、`AC`，则 `B`、`C` 都是主属性；依赖 `C → B` 左部不是超键，因此不满足 BCNF，但仍可能满足 3NF。

#### 2.5.3 无损连接题的起手式是“公共属性能否定住一边”

**核心概念：** 二元分解的无损连接不是看表名、不是看有没有公共字段，而是看公共属性是否足以决定某一子模式。  
**常见考法：** 给出 `R1`、`R2`，让你判断分解是否无损。  
**高频误区：** 只写“有公共属性，所以无损”。  
**快速判断提示：** 先写 `R1 ∩ R2 = ...`，再写这个公共属性能否决定 `R1` 或 `R2`。  
**小例子：** `R1(订单号, 商品号, 数量)` 与 `R2(商品号, 商品名, 单价)`，公共属性是 `商品号`；若 `商品号` 能决定 `R2` 的其他属性，则可判无损。

#### 2.5.4 保持函数依赖题要逐条列原依赖，再逐条对照

**核心概念：** 保持依赖看的是“原有约束还能不能被保留 / 推出”。  
**常见考法：** 给出分解方案，问是否保持函数依赖。  
**高频误区：** 只检查了一条依赖，就把整题写成“保持依赖”。  
**快速判断提示：** 把原依赖一条条列出来，再看每条依赖在哪个子模式中被完整保留。  
**小例子：** 若原依赖 `AB → C` 被分解后没有任何一个子模式同时保留 `A、B、C`，且又无法由投影依赖推出，就不能写成“保持函数依赖”。

#### 2.5.5 联系属性放错位置，会把后续整张表一起带偏

**核心概念：** 联系属性描述的是“某个实体在某个关系中的角色 / 职责 / 数量 / 明细”，不属于实体静态属性。  
**常见考法：** 人员分配、项目分工、订单明细、选课成绩这类题。  
**高频误区：** 把“职责、角色、数量、成绩”等本应挂在联系关系上的属性，塞进了人员表 / 项目表 / 商品表。  
**快速判断提示：** 先问自己：这个属性是“对象本身固定带着的”，还是“对象与对象发生关系时才出现的”。  
**小例子：** “分工角色”应挂在“项目—成员分工”关系中，而不是单独挂在“成员”表里。

#### 2.5.6 链式分解说明无损时，按相邻子模式逐段写更稳

**核心概念：** 三表及以上分解，不宜偷换成一行总交集。  
**常见考法：** 给出 `R1、R2、R3` 三张表，要求说明是否可视为合理无损。  
**高频误区：** 直接写 `R1 ∩ R2 ∩ R3` 的总交集口径。  
**快速判断提示：** 先写 `R1 ∩ R2`，再写 `R2 ∩ R3`，最后补一句“沿依赖链展开”。  
**小例子：** `R1(A,B)`、`R2(B,C)`、`R3(C,D)`，应先写公共属性 `B`，再写公共属性 `C`，而不是三表一起写。

| 下午题类型 | 常见问法 | 作答关键动作 | 常见丢分点 |
|---|---|---|---|
| 3NF / BCNF 判断题 | “该关系模式最高达到几范式” | 先求候选键，再看左部是否为超键、右部是否为主属性 | 候选键没求稳就直接猜范式 |
| 无损连接判断题 | “该分解是否无损连接，并说明理由” | 先写公共属性，再写公共属性能否决定某个子模式 | 只写“无损”，不写公共属性与依据 |
| 保持函数依赖判断题 | “该分解是否保持原函数依赖” | 先列原依赖，再逐条对照子模式是否保留 | 只检查一条依赖就下总结论 |
| 联系属性归位题 | “该属性应放在哪个关系模式中” | 先分实体静态属性与联系属性，再决定表位置 | 把职责 / 数量 / 角色塞回实体表 |
| 综合评价题 | “为什么该分解更合理” | 同时写范式提升、依赖保持、无损连接、冗余减少 | 只写一个维度，缺少综合收束 |

### 2.7 标准答题模板 / 采分点提示

#### 模板1：3NF / BCNF 综合判断题
1. 先写候选键或主码为：……  
2. 再写哪些属性是主属性，哪些是非主属性：……  
3. 再判断是否存在部分依赖、传递依赖，或左部不是超键的依赖：……  
4. 若问 BCNF，再明确写：某条依赖左部是否为超键。  
5. 最后写：因此该关系模式最高达到 …… 范式。

#### 模板2：无损连接 + 保持函数依赖判断题
1. 先列出原关系模式的关键函数依赖。  
2. 写出分解结果：`R → R1, R2`（或 `R1, R2, R3`）。  
3. 先判保持函数依赖：逐条看原依赖是否在子模式中被保留或可推出。  
4. 再判无损连接：先找公共属性，再看公共属性能否决定某个子模式。  
5. 最后分别下两个结论：是否保持依赖、是否无损连接。

#### 模板3：联系属性归位 + 3NF 分解题
1. 先写原关系模式中的实体静态属性和联系属性分别是什么。  
2. 再指出原模式存在的部分依赖 / 传递依赖 / 冗余问题。  
3. 给出分解方案：实体表 + 联系表。  
4. 明确说明联系属性保留在哪张关系表中。  
5. 最后写：该分解减少冗余、避免异常，并使结构更清晰。

原关系模式中存在 `(项目号, 成员号) → 分工角色`、`成员号 → 成员姓名, 部门号`、`部门号 → 部门负责人` 等依赖，其中候选键为 `(项目号, 成员号)`。由于成员侧属性只依赖成员号、部门负责人又依赖部门号，原模式同时存在部分依赖和传递依赖问题。将其分解为 `成员(成员号, 成员姓名, 部门号)`、`部门(部门号, 部门负责人)`、`项目分工(项目号, 成员号, 分工角色)` 后，各条关键依赖分别保留在对应子模式中，因此该分解保持函数依赖。进一步看，项目分工通过成员号连接成员表，成员表再通过部门号连接部门表，分解沿原依赖链逐段展开，可视为合理无损分解。综上，该分解既把联系属性放回了正确位置，也减少了冗余并使结构更清晰。

**采分点：**
- 顺序完整：原依赖 / 候选键 → 问题依赖 → 分解 → 保持依赖 → 无损 → 结论
- 明确点出联系属性位置
- 结尾有价值收束句
